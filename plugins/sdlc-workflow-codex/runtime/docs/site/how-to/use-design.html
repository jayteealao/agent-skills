Use the design pipeline

You are building a change that touches a visual or UX surface and you want design work to be part of it. Design is one /wf sub-command — /wf design — that runs as a compressed workflow. It gives you 20 design commands (as arguments, not separate keys). This page helps you pick the right invocation for your situation. The full design brief and visual contract are authored by the normal lifecycle — /wf shape and /wf plan — not by a command here.

Pre-conditionsIn-workflow shape (slug given): the workflow exists. No-slug shape: nothing required — it creates a new workflow.
GoalChoose the invocation shape and design command that fits the task you are doing right now.
One-time setupRun /wf design setup once per project to create PRODUCT.md and DESIGN.md. Most commands read these files for context.

The two ways to invoke /wf design

The first token after design is an optional slug, resolved by an exact existence check. Its presence decides how far the flow travels.

In-workflow — produce the design, then build it

You have an existing workflow (you have run /wf shape <slug>) and want design to lead the build. Run:

/wf design <slug> <command> [instructions]

For a Transformation command this authors a focused visual contract, then drives slice → plan → implement → verify itself, compressed — the transform owns its downstream flow, there is no hand-back to /wf slice. Example: /wf design my-feature harden builds the accessibility-hardened surface and registers it as an add-on (a registered add-on in 00-index.md). The full design brief + contract are not a command here — /wf shape authors the brief (02b-design.md) and /wf plan authors the visual contract (02c-craft.md).

No slug — a full lifecycle seeded by design

You want to start fresh from a design intent. Run:

/wf design <command> [instructions]

This creates a new slug and runs the complete intake → … → retro lifecycle, with the design intent seeding it. Review, inspection, and context commands (audit, critique, extract, setup, teach) are single-step regardless of slug — they read your project's PRODUCT.md / DESIGN.md for context and write their one artifact.

Picking the right command

The 20 commands fall into four categories. (The design brief and visual contract are authored by the normal lifecycle — /wf shape and /wf plan — not by a command here.)

1. Context (run once, any time)

/wf design setup
Interview-driven setup. Writes PRODUCT.md (audience, voice, register) and DESIGN.md (tokens, components, patterns). Run once per repo. Almost every other command reads these files.
/wf design teach
Re-runs the relevant interview round and updates PRODUCT.md or DESIGN.md in place. Use after the project's brand or design system has evolved.

2. The brief + visual contract (authored by the lifecycle, not a design command)

There is no craft command. When a feature has UI surface, /wf shape <slug> authors the design brief 02b-design.md (register, anti-goals, layout approach, key states, recommended references) as part of shaping. /wf plan <slug> then resolves the visual-direction gates (image probes + confirm) and authors the visual contract 02c-craft.md (north-star mock, mock-fidelity inventory, implementation contract). /wf implement builds against it and runs the critique-and-fix pass. Design rides the normal shape → plan → implement → verify flow — the transforms below are for focused, ad-hoc moves.

3. Review / analysis (single-step)

/wf design <slug> audit
Writes 07-design-audit.md with theming / responsive / anti-pattern judgment and 0–4 scoring. It consumes the a11y / perf / web-vitals already measured by verify rather than re-running them. Also runs as a /wf review dimension.
/wf design <slug> critique
Writes 07-design-critique.md — a register-forked expert critique (visual hierarchy, information architecture, cognitive load). Also runs as a /wf review dimension.

4. Inspection (single-step)

/wf design <slug> extract
Reads existing UI code and pulls reusable tokens and components into a design-notes file plus tokens-extracted.css. Needs no prior context.

5. Transformation (15 commands — applied at the build's implement step)

Each transformation modifies UI code and records itself as an add-on in 00-index.md; handoff and ship then translate each add-on into reviewer mentions and changelog lines automatically. In the compressed flow the transform is the implement step, so it no longer needs pre-existing code — it either creates the surface or modifies the slug's existing implementation.

A worked example

You are building a new settings screen. The workflow slug is settings-v2 and shape has already run.

  1. Design and build the screen in one compressed pass: /wf design settings-v2 craft. It writes the brief 02b-design.md and contract 02c-craft.md, then drives slice → plan → implement → verify itself — no hand-back.
  2. You want a stronger accessibility pass. Run: /wf design settings-v2 harden. The transform builds the hardened surface at its implement step and registers a design-harden add-on.
  3. Review judges it: /wf review settings-v2 runs design-audit as a dimension, interpreting the a11y/perf numbers verify already measured.
  4. At handoff, the plugin reads the add-on and adds "Accessibility improvements applied — N components updated, axe-core scan clean" to 08-handoff.md automatically.
