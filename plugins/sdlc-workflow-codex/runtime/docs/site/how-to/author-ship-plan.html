Author a ship plan

You need to tell the plugin how this repo releases — once, not per workflow. This page walks you through creating .ai/ship-plan.md, a one-time config file that every future release command reads automatically.

Pre-conditionsPlugin v9.42.0+ (for full A–K plan including inbound pipeline; v9.12.0+ for outbound Blocks A–G only). A repo open in Claude Code. No existing .ai/ship-plan.md — if one exists, run /wf-meta amend ship-plan instead.
GoalWrite .ai/ship-plan.md covering the eleven release concerns (Blocks A–K) so that /wf ship has everything it needs.
Time15–30 minutes the first time. Future releases reuse the plan — no re-authoring needed.

Why this file exists

Without a ship plan, the plugin would have to ask you the same questions on every release: which branch is protected? which CI workflow triggers the publish? what command bumps the version? The ship plan captures those answers once so the plugin can act without re-interviewing you.

The ship plan (a Markdown file with a YAML front-matter block) stores eleven "release concerns" — things like branch protection rules, the version-bump command, and rollback steps. Once committed, every /wf ship run reads the file directly.

Step 1 — Run the init command

The init-ship-plan stage (a single automated step in a workflow) reads your repo before asking any questions.

/wf-meta init-ship-plan                              # reads the repo; no seed
/wf-meta init-ship-plan --from-template npm-public   # biases toward npm-public defaults

The command scans CI workflow YAML, infrastructure files (Dockerfile, helm, fly.toml), package manifests (package.json, pyproject.toml, Cargo.toml), release tooling (release-please, goreleaser, changesets), and any runbook directories it finds.

Step 2 — Review the discovery report

Before asking you anything, Claude prints a bullet summary of what it found: inferred release meaning, version source-of-truth candidates, CI workflow candidates, registry hostnames, and secret references. It also flags any optional concerns your repo suggests — for example, a database migrations directory hints that a data-migration block may apply.

If Claude misread something, correct it in plain English before moving on:

the helm dir is dead code, ignore it
actually we publish to a private registry, not GHCR

Step 3 — Confirm each release concern (Blocks A–K)

For each of the eleven blocks, Claude proposes the value it inferred and asks you to confirm, adjust, or replace it. The blocks split into two groups: outbound (A–G, what goes out to users) and inbound (H–K, what gates code coming in).

A — Environment protection
What "ship" means in this repo. Which environments exist (staging, production, canary). How often releases happen. Per-environment GitHub protection rules.
B — Versioning contract
Versioning scheme (semver / calver / sequential). Which files hold the version. The command that bumps it (git-cliff, changesets, release-please, manual). Prerelease suffixes if any.
C — CI/CD pipeline
Which event triggers the release workflow. Which workflow file runs it. The jobs inside it. Required secrets (pre-filled from ${{ secrets.X }} patterns Claude found). The publish command and its dry-run equivalent. Caching, matrix, and concurrency settings.
D — Post-publish checks
How the plugin confirms the publish succeeded: registry API, fresh dependency resolve, GitHub release presence, smoke test, or Kubernetes rollout status. Propagation window and poll interval.
E — Rollout and rollback
Default rollout strategy (immediate / staged / canary / blue-green / feature-flag). The rollback command. Whether database migrations are reversible.
F — Recovery playbooks
Step-by-step remediation for known failure modes. Pre-filled from any runbook files Claude found. Grows over time — add entries later with /wf-meta amend ship-plan.
G — Announcements
Where release notes go (#releases Slack channel, mailing list). An optional message template path.
H — Code-quality gates (inbound)
Format check, lint, type check, and test-coverage tools — each with its command. Commit-message and PR-title conventions, and where each is enforced (local hook vs. CI). These feed the pre-merge check list in Block C.
I — Local developer experience (inbound)
Git-hook framework (husky / lefthook / pre-commit). Editor config. Runtime-version files (.nvmrc, .tool-versions). Task-runner targets and the bootstrap command new contributors run first.
J — Repo governance (inbound)
Branch protection: required checks, approval counts, stale-dismissal, code-owner review, linear history. How to apply rules (GitHub API or manual). CODEOWNERS. PR/issue templates. Dependency automation (Dependabot/Renovate). Merge controls (squash, auto-merge, merge queue).
K — Security and supply-chain gates (inbound)
SAST tool (CodeQL / Semgrep). Dependency audit (npm audit / pip-audit / cargo audit) with a fail-on severity threshold. Secret scanning (gitleaks / trufflehog). SBOM generation. License allow/deny policy. These also feed the pre-merge check list.

Step 4 — Add optional release concerns

Claude asks whether any concerns the standard blocks do not cover apply to your project. Options come from discovery: data-migration, feature-flag-rollout, infrastructure-as-code, mobile-app-store, compliance-gate, data-pipeline, schema-registry, or a custom description. Each one becomes an entry in additional-contracts[]. The core /wf ship command ignores these by default; custom hooks opt in by id.

Step 5 — Confirm and write

Claude shows a summary table of every block. Choose one of three responses:

  • Confirm — writes .ai/ship-plan.md to the repo root.
  • Adjust — returns to any single block to revise it.
  • Cancel — discards all changes.

Step 6 — Commit the plan

The ship plan is a project-level contract. Commit it so every team member — and every CI run — uses the same release definition.

git add .ai/ship-plan.md
git commit -m "chore: author ship plan"

Updating the plan later

When CI/CD changes, secrets rotate, or you want to add a recovery playbook after an incident, run:

/wf-meta amend ship-plan

Pick the block (A–K) or additional-contract id to edit. The amend flow re-runs only that block's confirmation loop — discovery is skipped because the existing plan is the source of truth. plan-version increments by 1. Amending Block H or C also re-derives the pre-merge check lists. After changing an inbound block (H–K), run /wf-meta build-pipeline to bring the repo and remote settings back into compliance.

Technical detail — templates as hypothesis seeds

Templates (kotlin-maven-central, npm-public, pypi, container-image, server-deploy, library-internal) are now hypothesis seeds, not control-flow branches. Passing --from-template <kind> biases the proposed values toward that shape; discovery still runs and can override them. Omit the flag when the repo has enough signals for unbiased discovery.
