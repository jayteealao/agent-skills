Use augmentations

You want to measure performance, add observability, or run an A/B experiment as part of a feature you are already building. Add-ons (precisely: augmentations) let you bolt that work onto a slice — an individual unit of work in a workflow — without changing the main build-and-ship steps.

Pre-conditionsA workflow exists and has passed the shape stage.
When to addMid-workflow, once shape or plan reveals you need observability, an A/B test, or a perf baseline.
EffectAn add-on appends an entry to your slice index and produces one extra file. Your main workflow stages are unchanged.

What add-ons are

Without add-ons, a slice moves straight from plan to implement to review. That is fine for most changes. But sometimes you need more: you want to know a code path is observable before you ship, or you want to gate rollout behind a feature flag, or you want hard numbers before and after a change.

Add-ons give you those extras as named stages that slot in alongside the main flow. They are opt-in — you add them only when you need them. Running an add-on appends a record to the augmentations: list in your workflow's 00-index.md file (the central registry for that workflow). Downstream stages — handoff and ship — read that list and surface the results to reviewers and the changelog automatically.

Instrument — add observability to a code path

You are about to change code that has no logs, metrics, or traces. If something goes wrong in production, you will have nothing to look at. Running the instrument add-on lets you design the signals first, before implementation.

Run it for a specific slice:

/wf instrument auth-refactor login-slice

The command produces 04b-instrument.md. That file names the signals worth recording (log lines, metric counters, or trace spans), where in the code they go, and what a healthy vs. unhealthy reading looks like. When handoff runs later, it translates the entry into a reviewer-visible note: "Added observability — 3 signals for previously unobserved code paths."

Add instrument when: the slice touches a code path that is currently invisible in production.

Experiment — roll out behind a feature flag

You are shipping a behaviour change and want a measured rollout rather than a hard cutover. Running the experiment add-on designs the cohort split, the feature flag, and the rollback signal before you write a line of code.

/wf experiment auth-refactor login-slice

The command produces 04c-experiment.md. That file covers: which users see the new behaviour, what flag controls it, how to read the results, and what triggers a rollback. Ship picks up the entry and adds a corresponding changelog line.

Add experiment when: the slice changes user-visible behaviour and you want the ability to measure and reverse it without a full redeploy.

Benchmark — record a perf baseline before you change anything

Performance regressions are easy to miss without a before-and-after comparison. The benchmark add-on runs in two passes: once before implementation (baseline) and once after (compare). If you skip the baseline, you have nothing to compare against.

/wf benchmark auth-refactor

Run the command before implementation to capture the baseline. Run it again after implementation to compare. Pass baseline or compare as a second argument to force the mode — e.g. /wf benchmark auth-refactor baseline. The command produces 05c-benchmark.md and records the mode (baseline or compare) in the workflow index (00-index.md) alongside the result file.

Add benchmark when: the slice could affect performance — even if you do not expect it to. A baseline is cheap; a regression caught in CI is cheaper than one caught in production.

Technical details — what gets written to 00-index.md

Running any add-on appends an entry to the augmentations: array in 00-index.md. Each entry records type, artifact, status, and created-at. Benchmark also records mode.

augmentations:
  - type: instrument
    artifact: 04b-instrument.md
    status: complete
    created-at: "2026-05-10T14:32:00Z"
  - type: benchmark
    artifact: 05c-benchmark.md
    mode: baseline
    status: complete
    created-at: "2026-05-10T15:10:00Z"

Handoff and ship read this array. Handoff converts each entry into a reviewer-visible mention. Ship converts each into a changelog line. No manual wiring is needed.
