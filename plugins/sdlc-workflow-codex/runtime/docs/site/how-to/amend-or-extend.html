how-to

Correct or extend a workflow

Two things can go wrong after you start building. Either the spec you agreed to turns out to be wrong — the acceptance criteria, scope, or approach needs correcting — or new scope appears mid-work that belongs in the same effort. Both are handled the same way now: you add a new slice. There is no amend — corrections are tracked as fresh work, never as an in-place mutation of a completed artifact, which keeps the provenance clean by construction.

SituationCommand
The spec was wrong/wf intake <slug> <scope> — add a corrective slice describing the fix.
New scope appeared/wf intake <slug> <scope> — add a slice for the new work.
A ship-plan block was wrong/wf ship-plan edit — correct one block of .ai/ship-plan.md.
The spec is right but the code is buggy/wf implement <slug> <slice> reviews — fix the implementation.

Correct or extend — add a new slice

A slug (glossary) is the short identifier for a workflow, like auth-refresh or payment-retry. Each workflow stores its definition in a set of artifacts (glossary) — the markdown files in .ai/ that record the shape, plan, and implementation notes for that slug.

When a review or retro reveals that an acceptance criterion was incorrect, that the approach relied on something that does not exist, or that a piece you agreed on is missing, add a new slice to the existing workflow with /wf intake <slug> <scope>. Passing an existing slug plus a free-text scope auto-routes to an extension: it registers a new slice alongside the ones already there and never touches completed work. This single path replaces the former amend and extend sub-commands.

/wf intake auth-refresh "correct the token-expiry AC — refresh at 90%, not on expiry"
/wf intake auth-refresh "add the missing logout-everywhere endpoint"

After the new slice is registered, run /wf plan auth-refresh <slice> and then /wf implement auth-refresh <slice> as normal. To seed the correction from prior findings, describe it in the scope text — e.g. reference the 07-review blocker or the 10-retro note you are acting on.

Ship-plan corrections — edit the block

The project ship plan (.ai/ship-plan.md) is not a workflow artifact, so it is corrected in place rather than by a new slice. Use /wf ship-plan edit to update a single block — for example, adding a newly discovered failure mode to the recovery-playbooks block.

Implement (reviews mode) — when the code is buggy

If the spec is correct but the implementation has a bug, you do not need a new slice. The spec does not need correcting — only the code does.

/wf implement auth-refresh login-flow reviews

This is the most common path after a review: review finds a blocker, implement-reviews fixes it, verify again, then re-handoff.

Decision rule

What you foundUse
"Acceptance criterion is contradictory"new slice — /wf intake <slug> <scope>
"The approach assumed X but X does not exist"new slice — /wf intake <slug> <scope>
"This feature is missing a piece we agreed on"new slice — /wf intake <slug> <scope>
"A ship-plan block is wrong"/wf ship-plan edit
"The code returns 500 when it should return 400"implement reviews
"The test does not cover the edge case"implement reviews
