Routers reference

All six commands at a glance — what each does and when to reach for it. Pick the right one here, then follow the per-command link for full argument detail.

A command (sometimes called a "router" in older docs) is a slash command the plugin exposes — /wf, /wf-quick, /wf-meta, /wf-design, /wf-docs, and /review. Each command dispatches to a sub-command based on the first argument you pass it.

Commands

CommandOne-line purposeWhen to reach for it
/wf Run a lifecycle stage — intake, shape, slice, plan, implement, verify, review, handoff, ship, retro — plus add-ons. Normal feature or bug work that needs a tracked workflow.
/wf-quick A quick command — a compressed, single-session flow for small changes (fix, hotfix, refactor, investigate, …). Trivial changes or one-off analyses where a full tracked workflow would be overkill.
/wf-meta Navigate and manage existing workflows — check status, resume, amend artifacts, skip stages, close a workflow. Any time you need to inspect or correct a workflow without running a stage.
/wf-design Design pipeline — 22 sub-commands across three invocation modes (workflow stage, workflow + sub-command, freestanding). UI/UX work, design briefs, component specs, visual transformations.
/wf-docs Documentation pipeline — discover gaps, audit, plan, generate, and review docs using the Diátaxis structure. Writing or auditing tutorials, how-tos, reference pages, or explanations.
/review Code review across 31 dimensions and 7 named sweeps via parallel sub-agent dispatch. Pre-merge review, security audit, or targeted quality check on a diff or file.
Looking for full argument detail? This page is the overview. Drill into the per-command pages (linked in the table above) for argument hints, what each command reads, what it writes, and when NOT to use it.

/wf — the lifecycle command

Dispatches across 10 lifecycle stages (the named steps a workflow moves through, from intake to retro) plus 4 add-ons (add-ons are optional add-on stages that layer observability, experiments, or performance benchmarks onto a slice). The stage sub-commands all take a slug (the short identifier for a workflow, e.g. my-feature-abc1); per-slice sub-commands additionally take a slice slug.

Sub-commandArgsStageNotes
/wf intake"description"1Entry point. Creates the workflow directory and slug.
/wf shape<slug>2Deep interview and parallel web research.
/wf slice<slug>3Decompose into vertical slices.
/wf plan<slug> [slice]4Repo-aware plan with reuse scan. Slice is optional — defaults to the selected slice.
/wf implement<slug> <slice> [reviews]5reviews mode applies review-finding fixes.
/wf verify<slug> [slice]6Exercises acceptance criteria and captures evidence. User-gated single-round fix loop.
/wf review<slug> [slice|triage]7Multi-domain parallel review with a per-finding fix loop.
/wf handoff<slug> [slice]8Aggregates complete slices and runs a PR-readiness block.
/wf ship<slug> [environment]9Plan-driven, replayable release.
/wf retro<slug>10Extracts concrete improvement actions.
/wf instrument<slug> [slice]add-onObservability add-on. Writes 04b-instrument.md.
/wf experiment<slug> [slice]add-onFeature flag and cohort design. Writes 04c-experiment.md.
/wf benchmark<slug> [baseline|compare]add-onPerformance baseline and compare. Writes 05c-benchmark.md.
/wf profile<area>add-onProfiling pass (freestanding).

/wf-quick — quick commands

Each sub-command here is a quick command — a self-contained flow that runs in a single session without creating a tracked workflow directory. Use these when the overhead of a full workflow would slow you down.

Sub-commandFor
/wf-quick fixTrivial, one-file change.
/wf-quick hotfixProduction incident (skips review).
/wf-quick rcaRoot-cause analysis — static diagnosis of a reported symptom by reading code and git history.
/wf-quick probe v9.14Runtime-truth verification — drives the running artifact, captures observable output (screenshots, stdout, responses), compares against acceptance criteria, and writes findings. Slug-mode only.
/wf-quick investigateSolution-options sketcher — 2–3 distinct engineering approaches with tradeoffs (effort, blast radius, reversibility, risk). No winner chosen.
/wf-quick discoverHypothesis test — adjudicates a code-level theory with FOR/AGAINST/counter-hypothesis evidence. Verdict: holds / partial / fails / inconclusive.
/wf-quick update-depsTiered dependency updates.
/wf-quick refactorBehaviour-preserving refactor.
/wf-quick ideateRank improvement candidates.
/wf-quick simplify3-agent review and route triage. Classifies findings and recommends downstream commands. Never writes code. Scopes: branch (default) / commit / plan / codebase.

/wf-meta — navigation and management

Sub-commandArgsWhat it does
/wf-meta next[slug]Shows the next recommended command for this workflow.
/wf-meta status[slug]One workflow or all workflows at a glance.
/wf-meta resume[slug]Re-reads the trail and proposes a concrete next invocation.
/wf-meta sync[slug]Repairs 00-index.md against disk reality.
/wf-meta amend<scope> <target>Corrects a prior artifact. scope=ship-plan edits .ai/ship-plan.md.
/wf-meta extend<scope> <target>Adds new slices to an existing workflow.
/wf-meta skip<stage> [slug]Marks a stage skipped with a stub artifact.
/wf-meta close<reason> [slug]Archives a workflow (shipped / abandoned / superseded / archived / stuck).
/wf-meta how<question> · <slug> plan|shape|slice|review|findingsFive-mode Q&A. --research or --quick forces a mode.
/wf-meta announce[slug]Generates a Diátaxis announcement for a completed workflow.
/wf-meta init-ship-plan[--from-template <kind>]Authors .ai/ship-plan.md — the full pipeline contract covering release (Blocks A–G) and developer experience (Blocks H–K).
/wf-meta build-pipeline[--dry-run]Audits the repo against .ai/ship-plan.md and builds the entire CI/CD and DX pipeline. No file is overwritten without confirmation.

/wf-design — design pipeline

22 sub-commands across three invocation modes: as a workflow stage (/wf-design <slug>), combined with a sub-command (/wf-design <slug> <sub>), or freestanding with no workflow context. The main planning and review sub-commands are shape, craft, audit, and critique. Fifteen transformation sub-commands handle targeted visual work: colorize, typeset, animate, harden, quieter, bolder, delight, distill, clarify, adapt, layout, optimize, polish, overdrive, onboard. Three setup and extract sub-commands complete the catalog: setup, teach, extract.

See /wf-design in depth for the full sub-command catalog and argument matrix.

/wf-docs — documentation pipeline

Runs the full pipeline (discover → audit → plan → generate → review) or invokes a single Diátaxis primitive directly (plan, tutorial, how-to, reference, explanation, readme, review).

See /wf-docs in depth for the full sub-command list and pipeline flow.

/review — code review

Per-dimension or named-sweep review across 31 dimensions and 7 sweeps via parallel sub-agent dispatch. Pass the change context and the command selects which dimensions apply. Unlike the other five commands, /review can also auto-invoke when the model detects a change worth reviewing.

See /review in depth for the full dimension list and sweep catalog.
