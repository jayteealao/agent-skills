Routers reference

Every command at a glance — what each does and when to reach for it. Two top-level commands, /wf and /review, plus design, which runs as the /wf design sub-command. Pick the right one here, then follow the per-command link for full argument detail.

A command (sometimes called a "router" in older docs) is a slash command the plugin exposes — /wf and /review. Each command dispatches to a sub-command based on the first argument you pass it. Design, once the standalone /wf-design router, is now the /wf design sub-command. The former /wf-quick, /wf-meta, and /wf-docs routers have all been retired: quick flows are now intake modes (/wf intake <mode>) or top-level keys (/wf probe, /wf simplify), navigation/lifecycle are /wf keys (/wf status, /wf recap, /wf close, /wf ship-plan), and documentation is the /wf docs key.

Commands

CommandOne-line purposeWhen to reach for it
/wf Run a lifecycle stage — intake, shape, slice, plan, implement, verify, review, handoff, ship, retro — plus add-ons. Normal feature or bug work that needs a tracked workflow.
Quick & standalone The former /wf-quick router is retired. Its flows now live under /wf: eight intake modes (/wf intake fix, hotfix, rca, investigate, discover, refactor, update-deps, ideate) plus two top-level keys (/wf probe, /wf simplify). Trivial changes or one-off analyses where a full tracked workflow would be overkill. See Quick & standalone flows.
Navigation & lifecycle The former /wf-meta router is retired. Its members are now /wf keys: /wf status (status, next, sync), /wf recap (resume/explain), /wf close, /wf ship-plan, and /wf intake <slug> <scope> (extend). Any time you need to inspect or correct a workflow without running a stage.
/wf design Design — 21 sub-commands, run as a compressed /wf sub-command. Two shapes: with a slug it builds in-workflow; with no slug it runs the full lifecycle. UI/UX work, design briefs, component specs, visual transformations.
Documentation The former /wf-docs router is retired. It is now the /wf docs key — same pipeline (discover, audit, plan, generate, review) using the Diátaxis structure. Writing or auditing tutorials, how-tos, reference pages, or explanations.
/review Code review across 33 dimensions and 7 named sweeps via parallel sub-agent dispatch. Pre-merge review, security audit, or targeted quality check on a diff or file.
Looking for full argument detail? This page is the overview. Drill into the per-command pages (linked in the table above) for argument hints, what each command reads, what it writes, and when NOT to use it.

/wf — the lifecycle command

Dispatches across 10 lifecycle stages (the named steps a workflow moves through, from intake to retro) plus 4 add-ons (add-ons are optional add-on stages that layer observability, experiments, or performance benchmarks onto a slice). The stage sub-commands all take a slug (the short identifier for a workflow, e.g. my-feature-abc1); per-slice sub-commands additionally take a slice slug.

Sub-commandArgsStageNotes
/wf intake"description"1Entry point. Creates the workflow directory and slug.
/wf shape<slug>2Deep interview and parallel web research.
/wf slice<slug>3Decompose into vertical slices.
/wf plan<slug> [slice]4Repo-aware plan with reuse and learnings scans — it reads the .ai/solutions/ corpus that retro distills and reports what it applies under ## Applied Learnings. Its verification strategy resolves every environment wall up front: each user-observable criterion that needs credentials, a device, or missing infrastructure must carry a constraint-resolution: entry (prerequisite harness, proxy + named clearing event, or recorded risk-acceptance) before the plan completes. Slice is optional — defaults to the selected slice.
/wf implement<slug> [slice|reviews]5reviews mode applies review-finding fixes. Slice defaults to the workflow's selected slice when omitted.
/wf verify<slug> [slice]6Exercises acceptance criteria and captures evidence. User-gated single-round fix loop.
/wf review<slug> [slice|triage]7Multi-domain parallel review with a per-finding fix loop.
/wf handoff<slug> [slice]8Aggregates complete slices and runs a PR-readiness block.
/wf ship<slug> [env|announce|rollback]9Plan-driven, replayable release. /wf ship <slug> announce re-runs the post-publish comms phase only; /wf ship <slug> rollback [<run-id>] reverses a prior completed run via an authored runbook — every step marked reversible or irreversible (irreversible ones surface as mitigations), gated on an explicit Go/No-Go, verified with the ship plan's rollback checks, and recorded as 09-rollback-<run-id>.md.
/wf retro<slug>10Extracts concrete improvement actions, and distills 0–3 durable learnings into the .ai/solutions/ corpus (each must recur, be non-obvious, and be actionable — zero is a legitimate outcome) so future plans start from what this workflow learned.
/wf instrument<slug> [slice]add-onObservability add-on. Writes 04b-instrument.md.
/wf experiment<slug> [slice]add-onFeature flag and cohort design. Writes 04c-experiment.md.
/wf benchmark<slug> [baseline|compare]add-onPerformance baseline and compare. Writes 05c-benchmark.md.
/wf profile<area>add-onProfiling pass (freestanding).
/wf probe<slug> [target]add-onRuntime-truth verification. Slug-mode only. See Quick & standalone flows.
/wf simplify[branch|commit <sha>|plan <slug> <slice>|codebase [<path>]]add-on3-agent review and route triage. Never writes code. See Quick & standalone flows.
/wf auto<slug> [<slice>]driverEnd-to-end lifecycle driver. Drives every slice → final review and stops before handoff; with a slice arg it drives that one slice then routes to the next. Writes no artifact of its own; never opens a PR.

Quick & standalone flows (retired /wf-quick)

The standalone /wf-quick router has been retired. Its ten flows are now distributed across /wf. Eight become intake modes invoked as /wf intake <mode>; two become their own top-level /wf keys. See Quick & standalone flows for the full argument grammar.

Old formNew formFor
/wf-quick fix/wf intake fixTrivial, one-file change.
/wf-quick hotfix/wf intake hotfixProduction incident — scope-locked compressed lifecycle; review still runs (defaults to the security rubric).
/wf-quick rca/wf intake rcaRoot-cause analysis — static diagnosis by reading code and git history.
/wf-quick probe v9.14/wf probeRuntime-truth verification — drives the running artifact. Slug-mode only.
/wf-quick investigate/wf intake investigateSolution-options sketcher — 2–3 distinct engineering approaches with tradeoffs.
/wf-quick discover/wf intake discoverHypothesis test — adjudicates a code-level theory.
/wf-quick update-deps/wf intake update-depsTiered dependency updates.
/wf-quick refactor/wf intake refactorBehaviour-preserving refactor.
/wf-quick ideate/wf intake ideateRank improvement candidates.
/wf-quick simplify/wf simplify3-agent review and route triage. Never writes code.

/wf navigation & lifecycle keys

The former /wf-meta router is retired; its members are /wf keys now.

KeyArgsWhat it does
/wf status[slug] [deep]One workflow or all at a glance. The single-workflow view ends with the next recommended command and reconciles 00-index.md against disk (absorbs the former next and sync). deep adds a reality-drift report.
/wf recap<slug> [focus]Re-reads the trail and proposes a concrete next invocation (the renamed resume). With a focus (plan|shape|slice|review|findings) it explains that artifact.
/wf intake<slug> <scope>On an existing slug with new scope, adds a slice — this replaces the former amend and extend. There is no amend.
/wf close<slug> [<slice> | <reason>]With a slice, terminates just that slice so downstream prerequisites are satisfied (absorbs the former skip, now slice-scoped). With a reason (shipped / abandoned / superseded / archived / stuck), archives the whole workflow.
/wf ship<slug> announceGenerates a Diátaxis announcement for a completed workflow.
/wf ship-planinit | build | editinit authors .ai/ship-plan.md (the pipeline contract — release Blocks A–G and developer-experience Blocks H–K); build audits the repo against it and builds the CI/CD + DX pipeline (nothing overwritten without confirmation); edit corrects a block.

For a code question, codebase exploration, or web research (the former /wf-meta how), use the deep-research skill.

/wf design — design pipeline

20 sub-commands, run as a compressed /wf sub-command. The design brief (02b-design.md) and visual contract (02c-craft.md) are authored by the normal lifecycle — /wf shape and /wf plan — not by a command here. Two invocation shapes: in-workflow (/wf design <slug> <transform>) authors a focused contract and then drives slice → plan → implement → verify itself, compressed; no slug (/wf design <sub>) creates a new workflow and runs the full lifecycle, seeded by the design intent. The review sub-commands are audit and critique. Fifteen transformation sub-commands handle targeted visual work: colorize, typeset, animate, harden, quieter, bolder, delight, distill, clarify, adapt, layout, optimize, polish, overdrive, onboard. Three setup and extract sub-commands complete the catalog: setup, teach, extract.

See /wf design in depth for the full sub-command catalog and argument matrix.

/wf docs — documentation pipeline

The former /wf-docs router is the /wf docs key now. Runs the full pipeline (discover → audit → plan → generate → review) or invokes a single Diátaxis primitive directly (plan, tutorial, how-to, reference, explanation, readme, review).

See /wf docs in depth for the full sub-command list and pipeline flow.

/review — code review

Per-dimension or named-sweep review across 33 dimensions and 7 sweeps via parallel sub-agent dispatch. Pass the change context and the command selects which dimensions apply. Unlike /wf, /review can also auto-invoke when the model detects a change worth reviewing.

See /review in depth for the full dimension list and sweep catalog.
