Artifact types

Every artifact carries a frontmatter type. The renderer dispatches on that value alone — renderers/<type>.mjs — while the storage filename determines the view URL. The tables below map each admitted type to the command that produces it, where it is stored, the renderer that projects it, and whether it may carry a rich .html.fragment.

Source of truthThe admitted types are the oneOf branches in tests/frontmatter.schema.json. The fragment column marks which types carry a fragment-bearing renderer; the enforced rich-tier set — whose .md writes hard-block (exit 2) without a sibling .yaml — is RICH_TIER_TYPES in hooks/post-write-verify.mjs: review, plan, design, ship-run, rca, benchmark, experiment, instrument, profile, simplify-run, review-command, design-audit, design-critique, design-contract (14). Note: design-contract is reminder-gated only; it is NOT in SIBLING_YAML_VALIDATED_TYPES.
DispatchBy frontmatter type → renderers/<type>.mjs. A type with no matching file falls through to the generic fallbackRender.
SchemaAll types use schema: sdlc/v1.

Change-mode lanes & the /wf-meta close record

As of v9.86.0 the /wf intake change-modes (fix, hotfix, refactor, update-deps) are compressed standard lifecycles: they run every stage single-pass and write the standard types (intake, shape, slice-index, plan, implement, verify) under .ai/workflows/<slug>/ — the lead is 01-<mode>.md with type: intake. Retired bespoke types (legacy — back-compat only)

The bespoke pipelines the change-modes used before v9.86.0 — fix-plan (fix), hf-brief/hf-plan/hf-implement/hf-verify (hotfix), rf-brief/rf-baseline/rf-plan/rf-implement/rf-verify (refactor), and the five dep-* types (update-deps) — are retired. Their types stay in the schema so existing on-disk artifacts still validate, and their _paths.mjs view-path rows are retained, but their dedicated renderers were deleted: a legacy artifact now resolves a view path with no renderer and degrades to fallbackRender (a plain readable page). New runs never write these types.

Terminal analysis-mode roots

As of v9.86.0 the terminal analysis modes ideate and simplify root a lightweight type: workflow-index slug workflow with an 01-<mode>.md lead under .ai/workflows/<slug>/ (the lead keeps its analysis type — ideation / simplify-run — and carries a slug). The legacy off-pipeline roots (.ai/ideation/, .ai/simplify/, .ai/profiles/) are still discovered by the walker, so older runs keep rendering at their original URLs.

Two artifact kinds intentionally have no bespoke page. routing (90-next.md) is a regenerable point-in-time snapshot that duplicates the slug overview's next-command, so it renders via the generic fallback by choice. The how-* research-note family (/wf-meta how, in .ai/research/ and 90-how-*.md) carries no schema: sdlc/v1 and is not projected at all. Everything else the plugin writes now has a dedicated renderer.

Schema version

Every artifact uses schema: sdlc/v1. The authoritative list of admitted types and their required fields is tests/frontmatter.schema.json; the machine-wide serve/config schema is schemas/sdlc-config.schema.json. The end-to-end acceptance test (npm run test:e2e) renders one fixture per render-eligible type and fails if any admitted type loses its renderer, so this table and the schema cannot silently drift apart.
