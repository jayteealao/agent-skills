Artifact types

Every artifact carries a frontmatter type. The renderer dispatches on that value alone — renderers/<type>.mjs — while the storage filename determines the view URL. The tables below map each admitted type to the command that produces it, where it is stored, the renderer that projects it, and whether it may carry a rich .html.fragment.

Source of truthThe admitted types are the oneOf branches in tests/frontmatter.schema.json. The fragment column marks which types carry a fragment-bearing renderer; the enforced rich-tier set — whose .md writes hard-block (exit 2) without a sibling .yaml — is RICH_TIER_TYPES in hooks/post-write-verify.mjs: review, plan, design, ship-run, rca, benchmark, experiment, instrument, profile, simplify-run, review-command, design-audit, design-critique (13).
DispatchBy frontmatter type → renderers/<type>.mjs. A type with no matching file falls through to the generic fallbackRender.
SchemaAll types use schema: sdlc/v1.

Two artifact kinds intentionally have no bespoke page. routing (90-next.md) is a regenerable point-in-time snapshot that duplicates the slug overview's next-command, so it renders via the generic fallback by choice. The how-* research-note family (/wf-meta how, in .ai/research/ and 90-how-*.md) carries no schema: sdlc/v1 and is not projected at all. Everything else the plugin writes now has a dedicated renderer.

Schema version

Every artifact uses schema: sdlc/v1. The authoritative list of admitted types and their required fields is tests/frontmatter.schema.json; the machine-wide serve/config schema is schemas/sdlc-config.schema.json. The end-to-end acceptance test (npm run test:e2e) renders one fixture per render-eligible type and fails if any admitted type loses its renderer, so this table and the schema cannot silently drift apart.
