/wf command — every lifecycle stage in depth

You reach for /wf when you have a real feature, fix, or refactor to ship — one where you want clear acceptance criteria, a written plan, proper review, and a documented trail from idea to production. It is the full-lifecycle command: 10 ordered stages (intake through retro) plus 4 optional add-ons (instrument, experiment, benchmark, profile) for observability, flag-gated rollout, and performance work, plus the compressed design workflow (/wf design), runtime-truth verification (/wf probe), a read-and-route triage tool (/wf simplify), and the end-to-end lifecycle driver (/wf auto). All 18 keys are invoked as /wf <key> [args].

Every entry on this page is invoked as /wf <sub-command> [args]. Each sub-command maps to one stage of the pipeline and reads the artifacts the previous stage wrote.

/wf intake "<description>"

You reach for this sub-command when you have a new piece of work and you want the plugin to start tracking it — this is where every full-lifecycle workflow begins.

Stage1 — the entry point
RequiresNothing. This is where workflows start.
Produces00-index.md (control file), 01-intake.md (initial requirements + branch decision)
Default next/wf shape <slug>

Captures the raw ask, generates a stable slug (kebab-case, lowercase), establishes the branch strategy, and writes the control file. The slug is derived from the title and never changes — it's the directory name under .ai/workflows/.

What it reads

  • The free-text description you pass.
  • Recent git activity to suggest a base branch.
  • Any existing .ai/workflows/*/00-index.md to avoid slug collisions.

What it writes

  • .ai/workflows/<slug>/00-index.md — control file with slug, title, status, branch-strategy, current-stage.
  • .ai/workflows/<slug>/01-intake.md — your stated requirements plus structured frontmatter (urgency, scope, constraints).

Interview pattern

3–6 structured questions via AskUserQuestion: branch strategy (dedicated/shared/none), base branch, urgency, anyone-else-touching-this. Plus freeform: any constraints not captured.

When to use

  • You have a real feature or substantive change to make.
  • You want a full lifecycle including review, handoff, and retro.

When NOT to use

  • One-line typo → /wf intake fix.
  • Production incident → /wf intake hotfix.
  • You have a theory about how the code works and want it adjudicated → /wf intake discover.
  • You have a problem and want 2–3 distinct engineering approaches sketched before picking one → /wf intake investigate.

/wf shape <slug>

You reach for this sub-command when intake is done and you want a clear, agreed-on spec before any code is written — it runs the deepest interview of the lifecycle.

Stage2 — deep interview + research
Requires01-intake.md
Produces02-shape.md — full mini-spec with acceptance criteria, edge cases, constraints, and the docs plan
Default next/wf slice <slug>
DesignFor UI features, shape also authors the design brief 02b-design.md; the visual contract follows at /wf plan (Stage 2b)

The deepest interview stage. Shape commits the AI to understanding, not guessing. It dispatches parallel web-research sub-agents (best-practice scan, anti-patterns, gotchas) before asking you to fill in acceptance criteria and the docs plan.

What it reads

  • 01-intake.md
  • Project files for tech-stack context (package.json, pyproject.toml, etc.)
  • Live web research — best practices for the feature kind, recent vendor changes, known gotchas.

What it writes

  • 02-shape.md with frontmatter: docs-needed, docs-types, has-ui, has-migration.
  • Body sections: Goal, Acceptance Criteria, Scope (in/out), Constraints, Edge Cases, Open Questions, Freshness Research, Documentation Plan.

Interview pattern

15–30 structured questions in 4–6 rounds (AskUserQuestion). Topics: goal, acceptance criteria, in/out scope, edge cases, observability needs, documentation needs.

When to use

  • You need a clear spec before any code is written.
  • The change is large enough that misunderstanding it would be expensive.

When NOT to use

  • The acceptance criteria are obvious from the description (rename, typo, version bump). Use /wf intake fix.

/wf slice <slug>

You reach for this sub-command after shape is complete — it breaks the spec into independently deliverable pieces so you can work on each one in parallel or sequence.

Stage3 — decompose into vertical slices
Requires02-shape.md
Produces03-slice.md (master index) + 03-slice-<slice-slug>.md per slice
Default next/wf plan <slug> <slice>

Cuts the shaped work into thin, independently deliverable vertical slices. Each slice can be planned, implemented, verified, and reviewed in parallel. Single-slice features go through one cycle; multi-slice ones can fan out.

What it reads

  • 02-shape.md — goal, acceptance criteria, scope
  • 02b-design.md if present (UI features)

What it writes

  • 03-slice.md — master index with slice array (slug, status, complexity, depends-on)
  • 03-slice-<slice-slug>.md per slice — goal, acceptance criteria, scope (in/out), complexity (xs/s/m/l/xl), dependencies

When to use

  • Whenever shape is complete. Even single-slice work goes through slice (writes one per-slice file).

When NOT to use

  • Use /wf close <slug> <slice> only for quick commands that already decomposed at /wf intake time.

/wf plan <slug> [slice]

You reach for this sub-command before writing any code — it researches your repo and the current best practices, then produces the concrete file-by-file plan that implement will execute.

Stage4 — repo-aware implementation plan
Requires03-slice-<slice>.md
Produces04-plan-<slice>.md
Default next/wf implement <slug> <slice>

Plan is repo-aware. It dispatches parallel Explore sub-agents: reuse scan (does the codebase already have helpers we should use?), convention scan (what file/folder shape do existing features follow?), and web-research (current best practices for the specific pattern).

What it reads

  • 03-slice-<slice>.md + 02-shape.md + 02b-design.md + 02c-craft.md if present
  • Repo files via Explore sub-agents
  • Live web research for current patterns

What it writes

  • 04-plan-<slice>.md — files to touch, order of operations, reuse decisions, test strategy, has-blockers flag, revision count

Auto-review mode

If 04-plan-<slice>.md already exists, re-running plan triggers auto-review: an independent sub-agent reads the plan against the current spec and flags issues. Use this when a review or retro found the plan was incomplete.

When to use

  • Before any implementation. Always.

When NOT to use

  • Hotfix — /wf intake hotfix compresses plan into the single artifact.

/wf implement <slug> [slice] [reviews]

You reach for this sub-command when the plan is written and approved — this is the only stage that actually writes code and commits it.

Stage5 — execute the plan, commit atomically
Requires04-plan-<slice>.md
Produces05-implement-<slice>.md + code + a commit
Default next/wf verify <slug> <slice>

The only stage that writes code. Reads the plan, makes the edits, runs the project's test command, and commits with a message that follows your repo's commit convention.

Modes

  • Default: /wf implement <slug> [slice] — implements per the plan. The slice argument can be omitted when exactly one slice is in-progress (inferred automatically).
  • Reviews: /wf implement <slug> [slice] reviews — applies review-finding fixes. Reads 07-review-<slice>.md and re-implements the targeted fixes.

What it reads

  • 04-plan-<slice>.md, 02-shape.md, 02b-design.md, 02c-craft.md, 04b-instrument.md, 04c-experiment.md when present
  • Existing repo code touched by the slice

What it writes

  • Code edits + tests in your repo
  • 05-implement-<slice>.md with commit-sha, metric-files-changed, metric-lines-added/removed, metric-deviations-from-plan, and a ## Plan deviations section if anything diverged

Plan-deviation discipline

If Claude can't follow the plan exactly (discovered constraint, missing helper, plan was wrong), the deviation is recorded explicitly. The plan stays a plan; the implement file becomes the record of what actually happened.

/wf verify <slug> [slice]

You reach for this sub-command after implement — it exercises each acceptance criterion and captures evidence, so review has facts rather than assumptions to work from.

Stage6 — exercise the acceptance criteria, capture evidence
Requires05-implement-<slice>.md
Produces06-verify-<slice>.md with result: pass | fail | partial | blocked-runtime-evidence-missing v9.14
Default next/wf review <slug> <slice>

Reads the acceptance criteria from 02-shape.md and the slice file, then exercises each criterion. Captures evidence: test output, response bodies, screenshots (for UI), benchmark results (if 05c-benchmark ran).

What it reads

  • Acceptance criteria from 02-shape.md + 03-slice-<slice>.md
  • The just-committed code
  • Project test runner command (npm test, pytest, gradle test, etc.)
  • skills/wf/reference/runtime-adapters.md — adapter registry consumed by interactive sub-agent 3 v9.14

What it writes

  • 06-verify-<slice>.md with frontmatter: result, metric-checks-run, metric-checks-passed, metric-acceptance-met
  • v9.14 additions: metric-acceptance-user-observable, metric-acceptance-code-only, interactive-verification: required | deferred | not-applicable, interactive-verification-defer-reason, adapters-used, bootstrap-failures
  • Body: evidence per acceptance criterion (test output, curl response, screenshot link, etc.), with a new kind: code-only | user-observable column on each entry

User-observable AC gate (Step 7.5, MANDATORY) v9.14

Closes the "verified but actually broken" leak. The rule is simple: runtime evidence is required for every user-observable AC. No evidence, no pass.

Verify now partitions each slice's acceptance criteria into two kinds:

  • code-only — "the new util function handles null inputs". Test suites are sufficient evidence.
  • user-observable — "the Save button persists the value across page reloads". Requires interactive evidence (screenshot, response body, log line) captured by sub-agent 3 driving the running artifact via an adapter from runtime-adapters.md.

Partitioning is hybrid: an explicit observable: true | false annotation on the AC entry wins; otherwise a wording heuristic checks for visible-surface / user-action / observable-post-condition signals.

The gate runs at Step 7.5 (between Step 7 — write the artifact and Step 8 — route to next stage). For every user-observable AC, the gate requires a matching entry in the interactive sub-agent's results. The verdict matrix:

ConditionResult
All AC met (code-only via tests, user-observable via interactive evidence)pass
At least one user-observable AC has no matching interactive evidence AND no deferralblocked-runtime-evidence-missing (new in v9.14)
At least one AC fails or is partial; every user-observable AC has evidence (positive or negative)fail or partial

The blocked-runtime-evidence-missing variant is distinct from fail: it's a procedural failure (evidence not produced) rather than a substantive one (evidence shows the AC is not met). The downstream routing differs — fail recommends /wf implement to fix the code; blocked-runtime-evidence-missing recommends either re-running verify in a capable environment, or applying a deferral.

Escape hatch — interactive-verification: deferred v9.14 · tightened v9.95–v9.99

Some AC are user-observable but genuinely cannot be probed in the current environment. A deferral is a last resort that must be paid for: the defer-reason must enumerate the constraint-ladder rungs actually tried, and a claimed incapability ("no credentials", "no device") is valid only after executing a capability probe and recording its literal command and output — e.g. firebase projects:list for deploy credentials, adb devices for devices. Bare phrases like "no emulator" or "will verify later" are rejected:

interactive-verification: deferred
interactive-verification-defer-reason: "Robolectric covers the state machine (9/9); Roborazzi golden covers
  the visual; AVD boot attempted (failed: HAXM unavailable — output recorded); residual = live multi-touch
  routing on hardware"

Two related tightenings: a suite whose specs all exit via a credential/environment guard (0 executed) is blocked-runtime-evidence-missing with the unmet precondition named — never a silent deferral. And the hatch is unavailable for a criterion whose plan-named environment dependency carries no constraint-resolution: entry — verify refuses to inherit an unengineered wall and routes back to /wf plan to author the resolution (prerequisite harness, proxy + named clearing event, or recorded risk-acceptance).

Verify writes result: partial (not pass) and appends an entry to 00-index.md.runtime-evidence-deferrals[] with {slice, reason, deferred-at, cleared-by: null} — plus repeat-of: <slice> when the same environment wall was already deferred earlier in the slug, which obliges the next plan to either scope the harness that retires the wall or record harness-declined. Verify, review, and handoff surface deferrals as soft warnings and proceed. /wf ship hard-blocks while any deferral has cleared-by: null (see /wf ship Step 6.5 below). Clearing requires either a /wf probe run whose captured evidence satisfies the deferred AC (which sets cleared-by: probe-<descriptor>) or a re-run of verify in a capable environment.

Single-round user-gated fix loop v9.15

After the checks and the AC gate finish, verify owns a single-round, user-gated fix loop — it no longer just bounces back to implement. Each failing check and unmet AC is triaged via AskUserQuestion (Fix / Skip / Escalate per issue); chosen fixes spawn sub-agents that apply the minimal patch, then only the affected checks re-run once. It is one round only: if anything still fails, verify writes convergence: escalated and routes you to re-invoke /wf verify or to /wf implement as a manual escape. The artifact's convergence: frontmatter records not-needed | converged | escalated.

If verify fails

Routes back to /wf implement <slug> <slice> with directed fix. Does NOT loosen acceptance criteria. If the criteria themselves were wrong, corrections are a new slice (/wf intake <slug> <scope>) or a directed re-plan. New v9.14 route: if result: blocked-runtime-evidence-missing, route either to a capable environment for re-run, or annotate with a deferral and proceed (the deferral blocks ship but allows review/handoff).

/wf review <slug> [slice|triage]

You reach for this sub-command once verify passes — it fans out across up to 33 review dimensions in parallel and aggregates the findings into a single ship verdict.

Stage7 — multi-domain parallel review dispatch
Requires06-verify-<slice>.md
Produces07-review-<slice>.md (master verdict) + 07-review-<slice>-<cmd>.md per dimension
Default next/wf handoff <slug> (if ship verdict) or /wf implement <slice> reviews (if blockers)

Not a single reviewer. A dispatch orchestrator that reads shape and slice, decides which of the 33 review dimensions matter for this change, and runs each as a parallel sub-agent. Findings are aggregated, de-duplicated, and triaged with you.

What it reads

  • Every prior artifact: shape, slice, plan, implement, verify, design
  • The current diff (HEAD vs base)

What it writes

  • Per dimension: 07-review-<slice>-<cmd>.md with severity-graded findings
  • Master: 07-review-<slice>.md with verdict: ship | ship-with-caveats | dont-ship, commands-run, all finding counts

Review-owned fix loop v9.15

Review owns its own fix loop rather than only routing back to implement. For each finding you triage Fix, review spawns one sub-agent at a time (explicit model: sonnet) to apply the patch, then records the outcome on the finding (status: fixed / could-not-fix plus a fixed-at stamp) and in the master artifact's accumulating ## Fix Status ledger. Deferred or skipped findings are recorded with a reason, never silently dropped. There is no round counter — re-running review is just another accumulating pass (see below).

Accumulating ledger

The review artifacts are an accumulating ledger, not an overwrite-on-each-run report. Re-running /wf review on the same slug reads the existing 07-review-<slice>.md (and per-dimension files), de-duplicates this run's findings against the prior ones, and merges them in place — each finding carries a surfaced-at timestamp recording when it first appeared. A finding a re-run no longer surfaces (its dimension ran clean) is marked status: resolved and kept for history; nothing is overwritten or deleted. A compact runs: ledger in the frontmatter records each pass. The verdict and blocker counts are computed from the open findings only.

Triage mode

/wf review <slug> triage walks you through the deferred and untriaged findings on the existing ledger without re-running the reviewers. Triage decisions persist across runs unless you change them here.

/wf handoff <slug> [slice]

You reach for this sub-command when review approves the work and you're ready to open a PR — it packages the artifacts, writes the docs, pushes the branch, and runs the full PR-readiness check.

Stage8 — package for PR; run PR-readiness block
RequiresEvery in-scope slice's 05-* + 07-*
Produces08-handoff.md + Diátaxis docs + branch pushed + PR opened
Default next/wf ship <slug> if verdict ready

Aggregates ALL complete slices into one PR by default. Pass a slice slug as the second arg only when each slice ships as its own PR. Handoff refuses when any in-scope slice's review verdict is dont-ship or carries unresolved blocker findings.

The PR-readiness block — the readiness check that ship gates on — runs: commitlint pass, public-surface drift check, doc-mirror regen, T5.0 — watch CI to a terminal state and give bot reviews a bounded window to settle (added v9.41.0 so readiness is judged on real signal, not a stale gh pr view snapshot), the PR comment triage loop (T5.1), rebase onto base (T5.2), and a final live readiness re-watch (T5.3). It outputs a single readiness-verdict that ship gates on.

The review-file layout handoff checks depends on review-scope in 00-index.md: per-slice requires a 07-review-<slice>.md for every slice; slug-wide requires a single 07-review.md.

What it reads

  • Every workflow artifact through review
  • The git branch + base branch state
  • Optional 00-index.md config: public-surface, docs-mirror, review-bots

What it writes

  • Diátaxis docs per 02-shape.md docs plan (loads skills/wf/reference/docs/<type>.md primitive verbatim)
  • 08-handoff.md with the full PR-readiness frontmatter block
  • Pushed branch + new or updated PR

See Handoff schema for every frontmatter field; Triage PR comments for the T5.1 loop; The readiness check for how the verdict is computed.

/wf ship <slug> [environment]

You reach for this sub-command once handoff signals ready — it merges, tags, publishes, watches CI, and polls post-release checks through a replayable 13-step sequence.

Stage9 — plan-driven, replayable release
Requires.ai/ship-plan.md AND 08-handoff.md with readiness-verdict: ready
Produces09-ship-run-<run-id>.md + refreshed 09-ship-runs.md
Default next/wf retro <slug>

Ship first runs a short pre-flight: resolve the slug, read the plan + 00-index.md + the handoff verdict, run the runtime-evidence deferral gate (Step 6.5 below), detect paused runs, and mint a run-id. Then it walks the 13-step run sequence — each step independently re-runnable (detects already-done state before acting), so a resumed run continues from where it stopped.

The 13-step run sequence

  1. Pre-flight — version bump, secrets check, changelog regen (idempotent).
  2. Publish dry-run — mandatory if the plan sets one.
  3. Rollout questions — strategy, window, stakeholders (per-run, never re-asked).
  4. Freshness pass — research only what changed since the last successful run.
  5. Go/No-Go — AskUserQuestion gate.
  6. Merge — gh pr merge (only if ship-meaning includes merging).
  7. Tag + release — for release-trigger: tag-on-main.
  8. Release-workflow watch — gh run watch; recovery playbooks on failure.
  9. Post-publish polling — per plan.post-publish-checks, bounded by the propagation window.
  10. Post-release version bump — for projects with snapshot / dev versions.
  11. Update 09-ship-runs.md index.
  12. Adaptive routing — recommend the next stage.
  13. Write 09-ship-run-<run-id>.md (+ sibling .yaml / .html.fragment).

Step 6.5 — Runtime-evidence deferral gate (HARD BLOCK) v9.14

Runs in ship's pre-flight — orchestration step 6.5, after reading the handoff verdict and before the 13-step run sequence begins. Parses runtime-evidence-deferrals from 00-index.md (treated as empty if absent on older workflows). For every entry whose cleared-by: null, the slug has an open runtime-evidence deferral that must be cleared before ship.

If any entry has cleared-by: null, ship STOPS with a list of the open deferrals and the slices they're attached to. The user clears each by one of:

  • Running /wf probe <slug> <target> in a capable environment — captured evidence that satisfies the deferred AC sets cleared-by: probe-<descriptor> automatically.
  • Re-running /wf verify <slug> <slice> in a capable environment — produces fresh runtime evidence and rewrites the verify artifact without the deferral.
  • Removing the deferral entry from 00-index.md manually (only when the AC has been scoped out via a new slice added with /wf intake <slug> <new scope>).

Cleared deferrals (cleared-by non-null — typically a probe descriptor) do not block ship; they're kept in the index for audit. The block bites only on cleared-by: null entries. This is the hard-block half of the deferral mechanism — earlier stages (verify, review, handoff) surface deferrals as soft warnings; ship is where the block fires.

See Run a release for the walkthrough and Ship-run schema for the artifact contract.

/wf retro <slug>

You reach for this sub-command after ship — it reads the full artifact trail and proposes concrete improvements (CLAUDE.md additions, hooks, tests, CI checks) so the next workflow starts better than this one did.

Stage10 — extract lessons; close out
RequiresThe full artifact trail through ship (or handoff if shipping is external)
Produces10-retro.md

Reads every artifact in the workflow and proposes concrete improvement actions for the project: additions to CLAUDE.md, new hooks, missing test coverage, missing CI checks, missing command configurations.

What it reads

  • Every artifact in .ai/workflows/<slug>/ from intake through ship
  • The workflow's po-answers.md for moments where the user redirected the AI

What it writes

  • 10-retro.md with: workflow-outcome (completed/abandoned/partial), metric-improvement-count, metric-stages-completed
  • Body: lessons learned + concrete suggested additions, ready to copy into CLAUDE.md / hooks / etc.

When to skip

Quick commands (/wf intake <mode>) skip retro by default. The full pipeline should not — small lessons compound. Anti-pattern: skipping retro on "small" work.

/wf instrument <slug> [slice]

You reach for this add-on when the slice changes code paths that are currently invisible in production — no logs, metrics, or traces to tell you what happened.

TypeAdd-on — observability
Requires02-shape.md + a defined slice
Produces04b-instrument.md
Registers asEntry in 00-index.md's augmentations: array

Designs the observability for a slice: which logs, metrics, traces to add. Implement reads the add-on file and adds the signals. Handoff translates to "Added observability — N signals for previously unobserved code paths".

When to use

  • The slice changes code paths that have no logs/metrics/traces.
  • The change is behaviour you want to monitor in production.

/wf experiment <slug> [slice]

You reach for this add-on when you want to gate the behaviour change behind a feature flag and measure a controlled rollout rather than flipping the switch for everyone at once.

TypeAdd-on — feature flag + cohort design
Produces04c-experiment.md

For behaviour changes you want to gate behind a flag with a measured rollout. Writes the cohort split design, success metrics, rollback signal. Implement reads it and adds the gating code.

When to use

  • The slice changes user-visible behaviour and you want a slow rollout.
  • You need an explicit rollback signal beyond "revert the merge".

/wf benchmark <slug> [baseline|compare]

You reach for this add-on when you want regression insurance — run it before implement to capture a baseline, then again after to see exactly what changed.

TypeAdd-on — perf baseline + compare
Produces05c-benchmark.md

Two-phase: run before implement to capture baseline; run after implement (compare mode) to diff against baseline. Compare output flows into 06-verify-<slice>.md evidence and into handoff as "Performance baseline taken; verify-stage comparison: <within tripwires | regression>".

When to use

  • The change could plausibly affect performance, even if you don't expect it to.
  • You want regression insurance.

/wf profile <area>

You reach for this add-on when you suspect a hot path but haven't confirmed where it is — it runs a freestanding profiling pass without needing an active workflow slice.

TypeAdd-on — freestanding profiling pass
Slice-boundNo — doesn't need a workflow slice
Produces.ai/profiles/profile-<timestamp>-<slug>/01-profile.md

The only freestanding add-on. Investigate where time is spent in the named area. Language-detect, static-analysis rubric, dynamic-profiling commands per language, structured-output contract.

When to use

  • You suspect a hot path but don't know where.
  • You need to plan a perf improvement before committing to it.

/wf auto <slug> [slice]

You reach for this when you want to drive an already-started workflow forward without invoking each stage by hand — it sequences stages in-process, respects every quality gate, and stops before handoff.

TypeEnd-to-end lifecycle driver — not a stage
RequiresAn existing workflow slug with at least 01-intake.md written
ProducesNo artifact of its own — each delegated stage writes its own artifact as usual
Stops atAfter the final review; never runs handoff, ship, or retro

/wf auto is a driver, not a stage. It reads 00-index.md, determines which stage runs next, executes that stage's reference in-process (exactly as the /wf dispatcher does), reads the artifact the stage wrote to decide the gate state, then either proceeds to the next stage or pauses and hands back to the user. It writes no artifact of its own — every file in .ai/workflows/<slug>/ is written by the stage it delegates to.

Two modes

  • Slug mode — /wf auto <slug>: drives every slice through to the final review, then stops before handoff. With review-scope: per-slice, each slice runs plan → implement → verify → review; with review-scope: slug-wide, all slices run plan → implement → verify then a single slug-wide review runs once. Endpoint: recommend /wf handoff <slug>.
  • Slice mode — /wf auto <slug> <slice>: drives just that one slice to its end (per review-scope), then routes to the next slice. With review-scope: slug-wide, slice mode stops before the slug-wide review — that runs later via slug mode.

Gate behaviour — intra-stage gates still fire

auto removes inter-stage friction only. Every delegated stage keeps its own user gate: verify owns its Fix/Skip/Escalate loop; review owns its Fix/Defer/Dismiss triage. auto does not suppress or pre-answer any gate. When a stage pauses (e.g., verify returns convergence: escalated, or review leaves open blockers), the chain stops and auto recommends the appropriate re-invocation.

Stop-before-handoff — always

auto always stops after the review. There is no flag to extend past review. It never opens a PR, runs handoff, ship, or retro — those are deliberate, separately-invoked steps. Run /wf handoff <slug> yourself when the review is clean.

Resume is free

Because every stage updates 00-index.md and writes its own artifact, the artifact trail is the run log. If a run is interrupted, re-invoke /wf auto <slug> and it picks up from where the frontmatter shows the last completed stage. auto keeps no separate state file.

What it does not do

  • Does not start a fresh workflow — always requires an existing slug. Use /wf intake <description> first.
  • Does not write its own artifact.
  • Does not open a PR or run CI fixes.
  • Does not suppress any stage's own quality gates.
