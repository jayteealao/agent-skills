Ship-plan schema (.ai/ship-plan.md)

You would look this up when you want to know exactly which field to set in .ai/ship-plan.md — the per-repo release configuration you author once via /wf-meta init-ship-plan and update with /wf-meta amend ship-plan. Every field accepted by /wf ship and /wf-meta build-pipeline is listed here.

What this file is. .ai/ship-plan.md is the project-level release contract — one file per repo, sitting at the repo root (not under .ai/workflows/). It is authored through a guided conversation with /wf-meta init-ship-plan and is not generated per-release. Individual release records live instead in ship-run artifacts.

Two halves. The schema splits into a required core (Blocks A–G — fixed field names that /wf ship reads for outbound release mechanics) and an inbound pipeline (Blocks H–K — developer-experience and governance gates authored by /wf-meta init-ship-plan). A third section, additional-contracts[], is a free-form extension list that /wf ship ignores unless a consumer opts in by id.

Block A — Ship meaning + environments + cadence

You would check this block when configuring what "shipping" means for your repo — a registry publish, a deployment, or a merge — and how often releases run.

ship-meaning: publish | merge-only | deploy-immutable | deploy-rolling | feature-flag-flip
ship-environments:
  - { name: staging, auto-promote: false }
  - { name: production, auto-promote: false }
ship-cadence: on-demand | per-merge | weekly | release-train

Block B — Versioning contract

You would check this block when you need to know which file owns the version number, how version bumps are calculated, and what the post-release development suffix should be.

version-scheme: semver | calver | sequential | none
version-source-of-truth:
  - { path: package.json,   field: version }
  - { path: pyproject.toml, field: project.version }
version-bump-rule: git-cliff | conventional-commits | manual | fixed
version-bump-cmd: "command"
prerelease-suffix: none | -SNAPSHOT | -alpha | -beta | -rc
post-release-version: next-snapshot | next-dev | none
post-release-version-cmd: "command or empty"

Block C — CI/CD contract

You would check this block when you need to know which CI checks are required before merge, which workflow file triggers the release, or which secrets the release pipeline needs.

ci-pipeline:
  pre-merge-checks: [build, test, lint, ...]
  release-trigger: tag-on-main | merge-to-main | manual-dispatch | branch-push
  release-workflow-file: ".github/workflows/release.yml"
  release-jobs: [validate, build, publish]
  publish-dry-run-cmd: "./gradlew publishToMavenLocal"
  publish-cmd: "./gradlew publishAndReleaseToMavenCentral"
  required-secrets:
    - { name: "SECRET_NAME", purpose: "short description" }
  secrets-staleness-threshold-days: 90

Block D — Post-publish verification

You would check this block when diagnosing why a ship run is polling or timing out — it controls how long the workflow waits for the published artifact to appear in the registry.

post-publish-checks:
  - { kind: registry-api,   cmd: "...", expect: "..." }
  - { kind: fresh-resolve,  cmd: "...", expect: "..." }
  - { kind: github-release, cmd: "...", expect: "..." }
  - { kind: smoke-test,     cmd: "...", expect: "..." }
  - { kind: k8s-rollout-status, cmd: "kubectl rollout status ...", expect: "..." }
propagation-window-min-minutes: 5
propagation-window-max-minutes: 30
poll-interval-seconds: 60

Block E — Rollout + rollback

You would check this block when you need to know the rollback mechanism on hand or whether database migrations are reversible.

rollout-strategy: immediate | staged | canary | feature-flag
rollout-stages: ["10%", "50%", "100%"]   # only when staged/canary
rollback-mechanism: git-revert | gh-release-yank | feature-flag-off | blue-green-switch | redeploy-prior
rollback-time-estimate-min: 5
db-migrations-reversible: true | false | n/a

Block F — Recovery playbooks

You would check this block when a ship run hits an error and you want to see the documented recovery steps. The list starts empty and grows each time a run hits a new failure mode and you amend the plan.

recovery-playbooks:
  - id: signing-failure
    triggers: ["gpg signing failed", "InvalidSignatureException"]
    steps:
      - "Re-export key: gpg --export-secret-keys --armor ..."
      - "Re-upload: gh secret set SIGNING_KEY ..."
      - "Re-run failed workflow: gh run rerun $RUN_ID"

Block G — Stakeholder + announcement

You would check this block when configuring where release announcements go and which template they use.

announcement:
  channels: ["#releases", "release-notes@example.com"]
  template-path: ".ai/release-announcement-template.md"

Blocks A–G are the outbound half — read by /wf ship. Blocks H–K below are the inbound half — the developer-experience and governance pipeline also authored by /wf-meta init-ship-plan and enforced by /wf-meta build-pipeline.

Block H — Code-quality gates

You would check this block when you need to know the exact command each quality gate runs, the minimum test-coverage threshold, or which commit-message spec is enforced locally and in CI.

code-quality:
  format-check: { tool: prettier, cmd: "npx prettier --check ." }
  lint:         { tool: eslint,   cmd: "npm run lint" }
  type-check:   { tool: tsc,      cmd: "npx tsc --noEmit" }
  test-coverage: { min-percent: 80, cmd: "npm test -- --coverage" }
  commit-convention:   { spec: conventional, config-path: "commitlint.config.js", enforce: [local, ci] }
  pr-title-convention: { spec: conventional, enforce: [ci] }

The canonical source for each gate's literal command. Enabled gates derive ci-pipeline.pre-merge-checks[]. commit-convention documents the convention that /wf handoff's T3.5 gate detects via config-file presence (.commitlintrc, commitlint.config.js, etc.); CI enforcement is added by /wf-meta build-pipeline (Audit L).

Block I — Local developer experience

You would check this block when you need to know which Git-hook framework the project uses, what the bootstrap command is, or whether an .editorconfig file should exist.

local-dx:
  git-hooks:
    framework: husky        # husky | lefthook | pre-commit | simple-git-hooks | none
    hooks:
      pre-commit: ["npx lint-staged"]
      commit-msg: ['npx --no -- commitlint --edit "$1"']
      pre-push: []
  editorconfig: true
  runtime-version-files: [".nvmrc"]
  task-runner: { kind: npm-scripts, targets: { setup: "npm install" } }
  bootstrap-cmd: "npm install"
  contributing-doc: true

Block J — Repo governance

You would check this block when you need to know the branch-protection rules, the merge method, who owns which paths in CODEOWNERS, or whether /wf-meta build-pipeline can apply protection rules automatically via the GitHub API.

governance:
  branch-protection:
    base-branch: main
    mechanism: branch-protection   # or: ruleset
    required-checks: [build, test, lint, typecheck]
    required-approvals: 1
    dismiss-stale-reviews: true
    require-up-to-date: true
    enforce-admins: false
    require-code-owner-reviews: true   # default true when codeowners[] non-empty — else CODEOWNERS is cosmetic
    require-conversation-resolution: true
    require-linear-history: false
    allow-force-pushes: false
    allow-deletions: false
    apply-via: manual       # gh-api = build-pipeline applies via API (gated); manual = print commands only
  codeowners:
    - { path: "*", owners: ["@maintainer"] }
  pr-template: true
  issue-templates: true
  dependency-automation: { tool: renovate, ecosystems: [npm, github-actions], schedule: "weekly" }
  merge: { method: squash, auto-merge: true, merge-queue: false }

apply-via: gh-api lets /wf-meta build-pipeline mutate remote state (branch protection — Audit O; merge settings — Audit R), each only behind an explicit confirm gate. require-code-owner-reviews is what makes the CODEOWNERS file actually enforced.

Block K — Security & supply-chain gates

You would check this block when you need to know which security scanner runs in CI, the allowed and denied license list, or whether SBOM generation is wired into the release.

security:
  sast:             { tool: codeql, cmd: "", schedule: "weekly" }
  dependency-audit: { tool: npm-audit, cmd: "npm audit --audit-level=high", fail-on: high }
  secret-scanning:  { tool: gitleaks, cmd: "gitleaks detect --no-banner", pre-commit: true }
  sbom:             { tool: syft, format: spdx, publish-with-release: true }
  license-check:    { tool: license-checker, allow: [MIT, Apache-2.0], deny: [GPL-3.0] }

Built by /wf-meta build-pipeline (Audit P): CodeQL workflow, dependency-audit + license-check PR steps, secret-scanning (CI + pre-commit hook), SBOM in the release. Aligns to the supply-chain review dimension. PR-time gates also feed pre-merge-checks.

Block A · environment protection & Block C · ci-ergonomics (extensions)

You would check these when you need per-environment required-reviewer gates or want to understand how caching, matrix builds, and concurrency settings are stored in the plan.

ship-environments:
  - name: production
    auto-promote: false
    protection: { required-reviewers: ["@release-team"], wait-timer-minutes: 0, deployment-branch-policy: protected }

ci-pipeline:
  ci-ergonomics: { dep-cache: true, matrix: { os: [ubuntu-latest], versions: ["20","22"] }, release-concurrency: true, path-filters: false }

Environment protection is the third gated remote mutation (Audit Q, via gh api). CI-ergonomics (Audit S) folds caching, build matrix, and concurrency settings into the generated workflows.

Extensions — additional-contracts[]

You would look here when you need to add a project-specific contract that /wf ship does not need to read — such as a database-migration review policy or a feature-flag rollout rule. Each entry is free-form; consumers opt in by id.

additional-contracts:
  - id: data-migration
    purpose: "Liquibase migrations must be reviewed by DBA before any release"
    fields:
      tool: liquibase
      reversibility-policy: forward-only
      dba-review-required: true
    enforced-by: "Pull request label + /wf ship pre-merge check"
  - id: feature-flag-rollout
    purpose: "All user-facing changes go behind a LaunchDarkly flag"
    fields:
      provider: launchdarkly
      flag-naming-convention: "release.<slug>"
      cleanup-policy: "remove after 30 days at 100%"
    enforced-by: "Code review checklist"

Each entry takes the shape { id, purpose, fields: { ... }, enforced-by: "..." }. /wf ship does not read this list. Entries are authored during /wf-meta init-ship-plan Step 3 and are amendable via /wf-meta amend ship-plan by passing the id at the "Other" prompt.

Technical details — template-hint and plan-version fields

template-hint records the --from-template <kind> seed (if any) that biased the authoring conversation: kotlin-maven-central, npm-public, pypi, container-image, server-deploy, library-internal, or none. Informational only — no downstream code reads it. Useful for retro analysis ("which template best matched this project's eventual plan?") and for telling later amend conversations which seed values to compare against.

plan-version starts at 1 and increments by 1 every time /wf-meta amend ship-plan writes a change. Ship-run artifacts record plan-version-at-run for retro analysis.
