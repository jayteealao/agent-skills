Skills reference

This page is for when you need to understand how the plugin commands are implemented — for example to debug unexpected behavior or trace why a command ran differently than expected.

The plugin ships eleven skills — self-contained instruction files the model reads and executes. Six of them are commands (also called routers): each accepts a sub-command and dispatches to the matching reference file. The other five are support skills the model draws on for focused tasks like debugging, testing, or refactoring.

Every command can be run explicitly with a slash — /wf, /wf-quick, /wf-meta, /wf-design, /wf-docs, and /review. Skills are also how the model auto-invokes behavior without you typing a slash command; which skills can auto-invoke is covered in Auto-invocation below.

The six commands

CommandWhat it coversTypical use
/wf10 lifecycle stages plus 4 add-on typesRun explicitly: /wf <stage> <slug>.
/wf-quick10 quick commands — compressed, single-shot flowsRun explicitly: /wf-quick <flow> <slug>.
/wf-meta12 navigation and management sub-commandsRun explicitly. init-ship-plan and status are the most common.
/wf-design22 design sub-commandsThree invocation modes: as a lifecycle stage, paired with a sub-command, or freestanding.
/wf-docsDoc pipeline plus 7 Diátaxis primitivesRun explicitly, or auto-triggered when the model detects a documentation gap.
/review31 review dimensions across 7 named sweepsCan auto-invoke; the model selects dimensions based on change context.

How a command works internally

When you are seeing unexpected output from a command, the dispatch chain is the first place to look. Every command follows the same two-step structure:

  1. A SKILL.md file at skills/<name>/SKILL.md — the dispatcher. It reads your arguments and decides which sub-command was requested.
  2. A reference/ directory with one file per sub-command. The dispatcher loads the matching file and the model executes its instructions.

Example: /wf ship my-feature resolves like this — skills/wf/SKILL.md reads $ARGUMENTS, identifies ship, loads skills/wf/reference/ship.md, and the model follows its 13 steps.

If a command runs the wrong sub-command, check SKILL.md for argument-matching logic. If a step behaves unexpectedly, check the matching reference file under reference/.

Auto-invocation

Most commands never auto-invoke — you must run them explicitly. The exception is /review, which the model can trigger on its own when it detects a change worth reviewing. The five support skills below also auto-invoke on their own triggering conditions.

Support skills

Five skills under skills/ are not commands. They are focused capabilities the model invokes directly — either on its own, or when a command loads them internally:

SkillWhat it does
error-analysisRoot-cause analysis of errors, stack traces, and logs. Auto-invokes on debugging, incident-response, or "why is this failing" requests.
test-patternsPatterns for generating and organizing tests — unit, integration, data factories, coverage — across languages and frameworks.
refactoring-patternsSafe behavior-preserving refactoring — extract, rename, move, simplify — with test-baseline and rollback guidance.
wide-event-observabilityDesigning and implementing wide-event logging with tail sampling for context-rich, queryable observability.
imagegenGenerates an image from a text prompt with automatic fallback. Used internally by /wf-design; not directly invocable.
Advanced toggles

Five of the six commands set disable-model-invocation: true in their skill frontmatter: wf, wf-quick, wf-meta, wf-docs, and wf-design. This flag prevents the model from calling these commands without an explicit slash. The review command omits this flag, which is why it can auto-invoke.

If you are troubleshooting a command that ran when you did not expect it to, check its SKILL.md frontmatter for this key. If you are troubleshooting a command that did not run when you expected auto-invocation, the same key — or the triggering description field — is where to look.
