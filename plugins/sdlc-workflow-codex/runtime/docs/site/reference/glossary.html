Glossary

Plain-language definitions for every term the plugin uses. Each entry gives you the plain meaning first, then the precise internal name in parentheses where they differ.

Add-on (augmentation)
An optional extra pass you bolt onto a slice — performance measurement, observability instrumentation, or A/B experiment setup — without interrupting the main workflow. The three main types are instrument, experiment, and benchmark, all sharing type: augmentation with an augmentation-type discriminator. profile is a freestanding variant with type: profile, not tied to a workflow slice. All are recorded on 00-index.md's augmentations: array.
Artifact (workflow note)
A small markdown file the plugin writes next to your code that records what it decided and why. Each stage of the lifecycle writes one. They live under .ai/workflows/<slug>/ (or .ai/ship-plan.md at the repo root). Committed to git alongside your code — the decisions are in file history, not just conversation logs.
Base branch
The branch the feature branch will eventually merge back into (e.g. main). Set at intake, captured in 00-index.md.
Branch strategy
How this workflow relates to git branches. Three options: dedicated (its own feature branch and PR), shared (commits to an existing collaborative branch), none (no branch operations — just commits).
Command (router)
One of the six things you type: /wf, /wf-quick, /wf-meta, /wf-design, /wf-docs, /review. Internally called a router — it dispatches to sub-commands — but "command" is the user-facing term. See Commands reference →
Conditional input
An artifact that a stage will consume if it exists but does not require. Each stage's preamble lists its conditional inputs.
Dashboard (Sunflower view)
A local web page that shows all your workflows at a glance — status, branch, slug, and a card per workflow. Start with npm run serve; open http://localhost:4173. See Serve daemon & hub →
External Output Boundary
The rule that internal workflow details — artifact file paths, stage names, slash-command invocations — must never appear in outputs a user or external system sees (commit messages, PR bodies, release notes, docs).
Fragment
A companion .yaml file that sits next to a rich-tier artifact .md and carries the structured data the dashboard needs. 13 artifact types require a sibling fragment. Opt out per-file with fragment: none frontmatter.
Freshness pass
A web-research sub-agent step that gathers current external information — vendor advisories, platform status, recent CVEs — before a stage finalises its output.
Frontmatter
The YAML block between --- markers at the top of every artifact file. Contains all machine-readable state. The markdown body below is for human-readable narrative.
Plan-version
An integer counter on .ai/ship-plan.md that increments on every amend. Each ship run stamps the counter value it ran against as plan-version-at-run.
Quick command (compressed flow)
A shortcut under /wf-quick for small, well-defined changes that don't need the full 10-stage lifecycle. Ten in all: fix, rca, probe, investigate, discover, hotfix, update-deps, refactor, ideate, simplify. Produces 2–4 artifact files instead of up to 10.
Readiness check (readiness verdict)
The set of gates that must all say "ready" before the plugin will open or progress a PR. Runs inside /wf handoff. The verdict — ready, awaiting-input, or blocked — is written to 08-handoff.md and gates /wf ship. See The readiness gate →
Run-id
A compact UTC timestamp (YYYYMMDDTHHMMZ) that identifies a single ship run. Used as a suffix on ship-run artifact filenames: 09-ship-run-<run-id>.md.
Slice
A thin, independently shippable piece of the overall shaped work. Each slice gets its own plan, implement, verify, and review artifacts.
Slug
The short, stable name for one piece of work. Example: add-health-endpoint. Set at intake, never changes. Matches the directory name under .ai/workflows/. Used in every subsequent command: /wf shape add-health-endpoint.
Stage
One step of the full /wf lifecycle. The 10 stages in order: intake, shape, slice, plan, implement, verify, review, handoff, ship, retro. Each reads what previous stages wrote and writes its own artifact.
Sub-command
An option under a command. Example: /wf-design colorize — where colorize is the sub-command. /wf-quick has 10, /wf-design has 22, /review has 31 dimensions plus a sweep mode.
Triage loop (T5.1)
The bounded step inside /wf handoff that processes PR review comments. Fetches unresolved threads, classifies them (blocker / non-blocker / question), fixes blockers via /wf implement reviews, and resolves threads after fixes commit. Runs up to 5 iterations.
