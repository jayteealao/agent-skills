Pipeline reference (10 stages)

Read this if you want to understand the exact contract between each stage — what every stage reads, what it writes, and what it gates. You do not need this page to use the plugin; start with Full feature walkthrough instead.

A stage (glossary) is one named step in the workflow lifecycle. Each stage reads a set of artifacts (glossary — the Markdown workflow notes the plugin writes under .ai/workflows/<slug>/) and produces new ones that unlock the next stage. The tables below specify that contract exactly.

The 10 stages

Stage 1 — Intake: capture the problem

This is where you start. Intake records what you want to build, in plain language, before any code is written. It produces the artifact trail that every later stage reads.

Command/wf intake "<description>"
RequiresNothing. Entry point.
Produces00-index.md, 01-intake.md
Default next/wf shape <slug>
Skip-to/wf plan for trivial changes (rare)

Stage 2 — Shape: build the spec

Shape turns a rough idea into a deep specification — scope, constraints, docs plan, and open questions. Web-research sub-agents run in parallel to surface best practices before you write a line of code.

Command/wf shape <slug>
Requires01-intake.md
Produces02-shape.md (deep spec + docs plan)
Conditional inputsWeb-research sub-agents (best-practice, anti-patterns, gotchas) fire by default; they are skipped automatically when no new external dependencies, no security-sensitive area, and no new dependency API surface are introduced.
Default next/wf slice <slug>
DesignFor UI features, /wf shape also writes the design brief 02b-design.md; the visual contract 02c-craft.md follows at /wf plan

Stage 2b — Design (woven into shape + plan): the UX brief and visual contract

When you are building a UI surface, the design brief and visual contract are authored by the normal lifecycle — there is no separate design command. Skip nothing; non-visual work simply produces neither file.

CommandAuthored by /wf shape (brief) and /wf plan (contract) — no craft command
Requires02-shape.md
Produces02b-design.md (UX brief, register, anti-goals — at shape) + 02c-craft.md (visual contract — north-star mock, mock-fidelity inventory — at plan)
When to useUI features, dashboards, public-facing surfaces.

Stage 3 — Slice: break the work into deliverable chunks

Slice decomposes the spec into vertical slices — independently shippable pieces of work. Each slice gets its own artifact and its own plan, implement, verify, and review cycle.

Command/wf slice <slug>
Requires02-shape.md
Produces03-slice.md (master index) + 03-slice-<slice>.md per slice.
FrontmatterPer slice: complexity, depends-on.

Stage 4 — Plan: write a repo-aware task list

Plan produces a concrete, step-by-step task list for one slice. It runs a reuse scan and convention scan against your repo so the plan fits the codebase you already have.

Command/wf plan <slug> <slice>
Requires03-slice-<slice>.md
Produces04-plan-<slice>.md
Sub-agentsReuse scan, convention scan, web-research — all parallel.

Stage 5 — Implement: write and commit the code

Implement follows the plan and produces a commit. The artifact records exactly what changed, so later stages can compare reality against the plan.

Command/wf implement <slug> <slice> or … reviews for review-driven fixes
Requires04-plan-<slice>.md
Produces05-implement-<slice>.md + code + commit
Frontmattercommit-sha, metric-files-changed, metric-deviations-from-plan

Stage 6 — Verify: confirm the acceptance criteria pass

Verify exercises each acceptance criterion and records observable evidence. A single user-gated fix loop is available; the outcome is written to convergence: in the artifact.

Command/wf verify <slug> <slice>
Requires05-implement-<slice>.md
Produces06-verify-<slice>.md with evidence

Stage 7 — Review: multi-domain quality verdict

Review runs parallel sub-agents across multiple quality dimensions and produces a single ship/hold verdict. Each dimension gets its own artifact; the aggregate verdict is in 07-review-<slice>.md.

Command/wf review <slug> <slice>
Requires06-verify-<slice>.md
Produces07-review-<slice>.md (verdict) + 07-review-<slice>-<cmd>.md (per dimension)
Verdictship / ship-with-caveats / dont-ship

Stage 8 — Handoff: open the pull request

Handoff aggregates all completed slices, runs the PR-readiness block, and opens a pull request. The readiness check (glossary) must reach ready before stage 9 can proceed.

Command/wf handoff <slug> [slice-slug]
RequiresEvery in-scope slice's 05-* + 07-*
Produces08-handoff.md + PR + Diátaxis docs
PR-readiness blockT3.5 commitlint, T3.6 surface drift, T3.7 doc-mirror, T5.1 PR triage, T5.2 rebase, T5.3 live check
Gates ship viareadiness-verdict: ready

Stage 9 — Ship: execute the release

Ship runs the plan from .ai/ship-plan.md in 13 idempotent steps. If it is interrupted, re-running it picks up from the last status: awaiting-input checkpoint.

Command/wf ship <slug> [environment]
Requires.ai/ship-plan.md AND 08-handoff.md with verdict ready
Produces09-ship-run-<run-id>.md + refreshed 09-ship-runs.md
Replayable13 idempotent steps; resumes from status: awaiting-input

Stage 10 — Retro: close the loop

Retro reads the full artifact trail and extracts concrete improvements — additions to CLAUDE.md, new hooks, tests, and CI rules — so the next workflow starts with lessons from this one already baked in.

Command/wf retro <slug>
RequiresThe full artifact trail through ship (or handoff if shipping is external)
Produces10-retro.md — concrete additions to CLAUDE.md, hooks, tests, CI
Technical details — frontmatter keys, exit codes, and artifact naming

Each stage artifact is a Markdown file with YAML frontmatter. The status key drives the next-stage gate check. Key frontmatter fields by stage:

  • 05-implement: commit-sha, metric-files-changed, metric-deviations-from-plan
  • 06-verify: result (pass / fail / partial / blocked-runtime-evidence-missing), convergence (not-needed / converged / escalated)
  • 07-review: verdict (ship / ship-with-caveats / dont-ship)
  • 08-handoff: readiness-verdict (ready / not-ready)
  • 09-ship-run: status (complete / awaiting-input / failed)

Artifact files live under .ai/workflows/<slug>/ in the repo root. The 00-index.md in the same directory is the authoritative registry of all slices and their stage progress.
