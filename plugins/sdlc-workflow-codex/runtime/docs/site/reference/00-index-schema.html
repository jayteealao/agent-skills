00-index.md schema

Every workflow stores its state in a file called 00-index.md — a plain YAML header with no body text. Stages read it to decide what to do next, and update it when they finish. You would look this page up when debugging a workflow that is stuck, or when writing a script that reads workflow state.

The file lives at .ai/workflows/<slug>/00-index.md where <slug> is the stable identifier for your workflow. "Slug" means the short, filesystem-safe name assigned at intake — for example, add-dark-mode. The file contains only YAML frontmatter: a block delimited by --- lines at the top. There is no markdown body below it.

Optional fields (set as stages run)

These fields are absent from a fresh index file. Each is written by the stage that first needs it.

slicesArray of per-slice summary objects (slug, status, complexity, depends-on). Written once slices are defined.
augmentationsArray of { type, artifact, status, created-at } objects (benchmark adds mode; design transforms add files-modified). Appended when plan/implement apply a shape-decided augmentation (instrument, experiment, benchmark, profile) or a /wf design operator runs. These are augmentations, not /wf keys — shape decides them (augmentations-needed).
workflow-typeSchema enum: feature, fix (legacy: quick), rca, investigate, rf, refactor, hotfix, dep-update, update-deps, docs, discover, standard, adopt.
provenance v9.118authored (default; do not stamp explicitly) or adopted. adopted means the artifact was reverse-engineered from an already-existing diff by /wf intake adopt; downstream stages and any pattern-mining over 04-plan/05-implement (retro learnings, reuse analysis) must treat it as reconstruction, not forethought.
runtime-evidence-deferrals v9.14Array of deferral objects (see the expanded shape below). Appended by /wf verify when a slice's user-observable acceptance criterion is annotated with interactive-verification: deferred. /wf ship Step 6.5 hard-blocks while any entry has cleared-by: null. Cleared by a /wf probe run whose captured evidence satisfies the deferred criterion (sets cleared-by: probe-<descriptor>) or by a verify re-run in a capable environment.
intent-risks v9.126The intent-risk (RIM) ledger. Array of { id, risk, severity, status, adjudicated-by, decision, po-ratified }. Intake distils the "Risks if Misunderstood" prose into tracked entries; shape MUST adjudicate every one (set status to adjudicated or carried, plus a decision, and PO-ratify when it touches a PO directive) before it can complete. /wf handoff and /wf ship hard-block on any status: open; carried entries are legal but surfaced in the PR body and Reviewer Focus Areas.
charter v9.130The intake's 3–7 falsifiable positive commitments as first-class spec. Array of { id, commitment, source, status } where status is honored / at-risk / broken. Deliberately few — a charter that restates the whole intake is boilerplate. Set by verify runs, the intent-fidelity review, and yolo checkpoints; a broken commitment is a yolo stop. Absent on compressed lifecycles (fix/hotfix/refactor/update-deps/adopt).
evidence-quality v9.128Slug-level evidence-rung rollup — a map of rung → count, aggregated from the 06-verify AC status tables. The dashboard renders it as a chip; a slug with zero live-rung evidence on charter-cited ACs cannot ship without recorded PO acceptance.
unproven-integrations v9.128External integrations whose real behaviour has not been observed live in this workflow. Array of { name, introduced-by, first-light }. While first-light is null, ACs depending on the integration cap at partial (mock/emulator rungs are proxies, never a pass). Any live observation stamps first-light.
steering-honored v9.120Base field shared with the handoff and ship-run artifacts — records that the active steer.md standing instructions were applied to this stage's work.

runtime-evidence-deferrals shape v9.14 expanded v9.126

runtime-evidence-deferrals:
  - slice: <slice-slug>             # the slice whose verify deferred a user-observable AC
    reason: <one-line explanation>  # mirrors interactive-verification-defer-reason from the verify artifact
    deferred-at: <ISO 8601>          # when verify wrote the deferral
    cleared-by: null                 # null until truly evidenced; probe-<descriptor> when a probe/re-verify produces matching runtime evidence
    cleared-acs: [<ac-id>, ...]      # for a multi-AC deferral, the AC ids evidenced so far while cleared-by stays null
    absorbed-by: [<slice-slug>, ...] # slices that inherited this deferral; a THIRD absorption is a stop, not an absorption
    needed-by: <slice-slug>          # slice by which the clearing evidence is due (set at plan time); status escalates past it
    probe: <capability-probe cmd + output tail>  # the literal attempt-before-declare probe, carried into the ship-block hand-back
    ship-override-authorization:     # explicit PO risk-acceptance that lets ship proceed past an un-evidenced deferral
      by: <who>
      at: <ISO 8601>
      reason: <one line>

Only slice is required; the rest are filled in as the deferral progresses. An entry never moves out of the array — clearing is recorded by mutating cleared-by from null to the clearing artifact's identifier. This preserves the audit trail: a future reader can see which slices once had open deferrals and how each was eventually closed. The /wf status Runtime column derives its display from this field (count of cleared-by: null entries → deferrals: N).

cleared-by must be an evidence descriptor — a probe or re-verify run that produced matching runtime evidence — never a prose risk-acceptance string. PO risk-acceptance rides ship-override-authorization instead: an explicit override surfaced in the ship summary, distinct from evidence. needed-by (set at plan time) escalates a deferral's status once the named slice completes with cleared-by still null, and a charter-critical prerequisite past its needed-by boundary is a yolo stop.
