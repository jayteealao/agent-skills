00-index.md schema

Every workflow stores its state in a file called 00-index.md — a plain YAML header with no body text. Stages read it to decide what to do next, and update it when they finish. You would look this page up when debugging a workflow that is stuck, or when writing a script that reads workflow state.

The file lives at .ai/workflows/<slug>/00-index.md where <slug> is the stable identifier for your workflow. "Slug" means the short, filesystem-safe name assigned at intake — for example, add-dark-mode. The file contains only YAML frontmatter: a block delimited by --- lines at the top. There is no markdown body below it.

Optional fields (set as stages run)

These fields are absent from a fresh index file. Each is written by the stage that first needs it.

slicesArray of per-slice summary objects (slug, status, complexity, depends-on). Written once slices are defined.
augmentationsArray of { type, artifact, status, created-at } objects (benchmark adds mode; design transforms add files-modified). Appended by /wf instrument, experiment, benchmark, and /wf design operators. "Add-ons" (glossary) are the actions that append to this array.
workflow-typeSchema enum: feature, fix (legacy: quick), rca, investigate, rf, refactor, hotfix, dep-update, update-deps, docs, discover, standard.
runtime-evidence-deferrals v9.14Array of { slice, reason, deferred-at, cleared-by } objects. Appended by /wf verify when a slice's user-observable acceptance criterion is annotated with interactive-verification: deferred. /wf ship Step 6.5 hard-blocks while any entry has cleared-by: null. Cleared by a /wf probe run whose captured evidence satisfies the deferred criterion (sets cleared-by: probe-<descriptor>) or by a verify re-run in a capable environment.

v9.14 runtime-evidence-deferrals shape

runtime-evidence-deferrals:
  - slice: <slice-slug>             # the slice whose verify deferred a user-observable AC
    reason: <one-line explanation>  # mirrors interactive-verification-defer-reason from the verify artifact
    deferred-at: <ISO 8601>          # when verify wrote the deferral
    cleared-by: null                 # null until cleared; probe-<descriptor> when probe satisfies it

An entry never moves out of the array — clearing is recorded by mutating cleared-by from null to the clearing artifact's identifier. This preserves the audit trail: a future reader can see which slices once had open deferrals and how each was eventually closed. The /wf status Runtime column derives its display from this field (count of cleared-by: null entries → deferrals: N).
