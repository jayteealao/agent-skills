Ship-run schema (09-ship-run-<run-id>.md)

Documents every field in 09-ship-run-<run-id>.md — the artifact (workflow note) the plugin writes once per release run to record what happened, what was decided, and what evidence was gathered.

One file is created per run. Files accumulate — they are never overwritten. The run ID is a UTC compact ISO-8601 timestamp (YYYYMMDDTHHMMZ) so runs sort chronologically by filename.

Frontmatter

Every field is YAML frontmatter between --- markers. Fields absent from the file mean that step has not run yet.

schema: sdlc/v1
type: ship-run
slug: <slug>
run-id: "<YYYYMMDDTHHMMZ>"       # UTC compact ISO-8601
status: complete | awaiting-input | failed | rolled-back
plan-ref: ../../ship-plan.md
plan-version-at-run: 1
created-at: <ISO>
updated-at: <ISO>

# Per-run inputs
environment: <env name>
version: "<chosen version>"
prior-version: "<last release tag>"
go-nogo: go | conditional-go | no-go
merge-strategy: rebase | squash | merge | none

# Per-run evidence (set as steps complete; absent = not yet run)
head-sha-at-start: "<sha>"
pre-flight-status: pass | warn | fail
publish-dry-run-passed: true | false | skipped
merge-sha: "<sha or empty>"
release-tag: "<vX.Y.Z or empty>"
release-workflow-run-id: "<gh run id or empty>"
release-workflow-conclusion: success | failure | cancelled | empty
post-publish-checks:
  - { kind: <kind>, status: pass | fail | pending, observed-at: <iso>, evidence: "..." }
post-release-bump-sha: "<sha or empty>"

# Per-run outcomes
recovery-actions-taken: [<playbook-id>, ...]
rolled-back: true | false
rollback-sha: "<sha or empty>"
rollback-reason: ""
announcements-sent: [<channel>, ...]

# Agent bookkeeping (required; populated automatically)
tags: []
refs:
  index: 00-index.md
next-command: wf-retro
next-invocation: "/wf retro <slug>"

Field groups

Identity and bookkeeping

FieldMeaning
schemaAlways sdlc/v1. Required by the write-time validator.
typeAlways ship-run.
slugThe slug (short identifier) of the workflow this run belongs to. Must match the parent directory name.
run-idUTC compact ISO-8601 timestamp. Doubles as a sortable filename component.
statusCurrent state of this run. See the state machine below.
plan-refRelative path to the ship-plan.md this run executed against.
plan-version-at-runThe version: value in ship-plan.md at the moment the run started. Lets you detect if the plan changed after the run.
created-at / updated-atISO-8601 timestamps. updated-at advances on every step completion.

Per-run inputs

These are set at the start of a run, before any steps execute.

FieldMeaning
environmentTarget deploy environment (e.g. production, staging).
versionThe version string chosen for this release (e.g. v2.4.0).
prior-versionThe last published release tag before this run. Used to scope changelogs and rollback comparisons.
go-nogoThe readiness check verdict — go, conditional-go, or no-go. A no-go stops the run after step 5 without proceeding to merge or publish.
merge-strategyHow the feature branch is merged: rebase, squash, merge, or none (no merge step).

Per-run evidence

Each field is absent until the corresponding step completes. Absent means "not yet run" — it is not the same as a failure.

FieldSet when
head-sha-at-startCaptured immediately after the readiness check passes.
pre-flight-statusResult of the pre-flight checks: pass, warn, or fail.
publish-dry-run-passedWhether the dry-run publish step succeeded. skipped if the plan has no dry-run.
merge-shaThe commit SHA after the merge lands. Empty if merge was skipped.
release-tagThe git tag pushed for this release (e.g. v2.4.0). Empty if not yet tagged.
release-workflow-run-idThe GitHub Actions run ID for the release workflow. Empty if not triggered.
release-workflow-conclusionsuccess, failure, cancelled, or empty while pending.
post-publish-checksArray of check results. Each entry has kind, status (pass / fail / pending), observed-at timestamp, and evidence text.
post-release-bump-shaCommit SHA of the version-bump commit pushed after the release. Empty if not yet done.

Per-run outcomes

FieldMeaning
recovery-actions-takenList of playbook IDs executed during incident recovery for this run.
rolled-backtrue if the release was reversed after publication.
rollback-shaThe commit or tag the repo was reset to during rollback. Empty if not rolled back.
rollback-reasonFree text explaining why the rollback occurred.
announcements-sentList of channels or audiences notified after the release (or rollback).

Agent bookkeeping

These four fields are required by the write-time validator and are populated automatically by the agent — you do not need to fill them in manually.

FieldMeaning
tagsClassification tags for the run (array of strings). Written as [] if none apply.
refsCross-references to related artifacts (e.g. index: 00-index.md).
next-commandCommand name the agent recommends running next (e.g. wf-retro).
next-invocationFull slash command, ready to copy (e.g. /wf retro <slug>).

Status state machine

StatusMeaning
completeEvery step finished. go-nogo can still be no-go — the run completed by stopping at step 5 on user choice.
awaiting-inputA step paused and is waiting for a human decision. Re-running /wf ship offers to resume from where it stopped.
failedA recovery playbook ran but did not restore success, and the user marked the run failed.
rolled-backThe release was published but later reversed. rollback-sha and rollback-reason are populated.

Run index (09-ship-runs.md)

Alongside the individual run files, the plugin maintains a companion index file at 09-ship-runs.md. Its frontmatter holds a summary array; its markdown body is a human-readable table regenerated from that array on every ship run.

schema: sdlc/v1
type: ship-runs-index
slug: <slug>
updated-at: <iso>
runs:
  - { run-id, version, environment, status, go-nogo, notes }
