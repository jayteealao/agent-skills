/wf review — 33 review dimensions and 7 sweeps

You reach for /wf review when you want a structured code review. It runs one named dimension (a focused rubric — like "security" or "correctness") inline, or fans out every dimension in a named sweep in parallel and gives you a unified verdict.

I want to…What to type
Check one specific angle (e.g. security) on my current diff/wf review security
Run a focused multi-angle sweep before merging/wf review sweep pre-merge
Review a specific file, not the whole diff/wf review correctness ./src/auth.ts
Run all 33 dimensions before a major release/wf review sweep all

Single dimension vs sweep — which to use

A dimension is one focused review rubric. When you already know which angle matters — you just rewrote the auth layer and want to check for security issues — run a single dimension. It costs one sub-agent call and returns results inline in seconds.

A sweep is a named bundle of dimensions run in parallel. Each dimension gets its own sub-agent simultaneously, so a five-dimension sweep costs five agents but takes about as long as one. Use a sweep when you want defensive breadth: before merging a large feature, before a release, or when you are not sure which angles to worry about.

ModeFormCostUse when
Single dimension/wf review security
/wf review correctness ./src/auth.ts		One sub-agentYou know which angle to investigate.
Sweep/wf review sweep pre-merge
/wf review sweep all worktree		N sub-agents in parallelYou want defensive breadth across multiple angles at once.

Three names overlap — architecture, infra, and security are both sweep names and individual dimensions. A bare invocation runs the dimension. Add the word sweep to run the bundle: /wf review sweep security.

The 7 sweeps

Each sweep is a preset bundle of dimensions. Pick the one that matches your moment in the development cycle.

Sweep nameDimensions includedWhen to use

pre-mergecorrectness, testing, security, refactor-safety, maintainabilityStandard gate before merging a feature branch.

infrainfra, ci, release, migrations, logging, observabilityInfrastructure or devops change review.
uxaccessibility, frontend-accessibility, frontend-performance, interface-craft, motion, ux-copyUI feature review.

The 33 dimensions — by category

Each dimension below is a named rubric you can pass to /wf review. For example: /wf review data-integrity or /wf review ux-copy ./src/components/FormError.tsx.

Correctness & data

correctnessLogic bugs, off-by-one, missing edge cases, error handling, return values.
data-integritySchema constraints, foreign keys, transactions, unique violations, race conditions on writes.
migrationsForward + reverse SQL migrations, data backfills, deployment ordering, idempotency.
api-contractsBreaking changes in public APIs, missing version bumps, deprecation handling.

Security & privacy

securityAuth/authz, input validation, secrets handling, SQL injection, XSS, CSRF.
infra-securityIAM policies, network policies, secret storage, container security.
supply-chainDependency risk, lock-file integrity, registry sources, build provenance.
privacyPII handling, GDPR/CCPA, data retention, telemetry boundaries.

Severity levels

Every finding is tagged with one of four severity levels. The level tells you how to act on it — and where it routes inside a workflow.

LevelMeaningWhat to do
BLOCKERCorrectness, security, data loss, or crash risk.Fix before merging. Inside a workflow, the review stage runs its own fix loop — /wf review triages each BLOCKER via AskUserQuestion (Fix/Defer/Dismiss) and dispatches sub-agent patches inline. /wf implement <slug> <slice> reviews exists as a manual escape only when the fix loop cannot resolve a finding.
HIGHShould fix; might ship with an explicit caveat.Triage via Fix/Defer/Dismiss. Fixed findings go through the review-owned fix loop inline. Deferred HIGH findings produce a Ship-with-caveats verdict and do not block shipping. /wf implement reviews is only a manual escape.
MEDWorth addressing but not blocking.Address in a follow-up or inline before merge.
LOW / NITStyle, naming, minor clarity.Ship. Address if convenient.

Targeting — files, PRs, worktree

The optional target argument controls what the reviewer reads. If you omit it, the reviewer scopes to the current diff (HEAD vs base branch).

Target formWhat it reviews
(omitted)Current diff — HEAD vs base. Default.
./src/auth.ts or any pathThat specific file or directory.
worktreeAll uncommitted changes in the working tree.
#123A GitHub PR by number.
repoThe entire repository. Use sparingly — best paired with a focused dimension.

Integration with the /wf lifecycle

Both forms are the same /wf review key, distinguished by the first argument. Run it ad-hoc — a dimension or sweep with no slug (/wf review <dimension>) — to spot-check the current diff at any time. Pass a workflow slug instead (/wf review <slug> [<slice>|triage]) and the same key runs as the lifecycle's review stage.

When called via the workflow, the reviewer does more than check the diff. It reads the feature's shape, slice definition, plan, and implementation notes to understand what the code is supposed to do before picking which dimensions to apply. The steps:

  1. Reads the shape, slice definition, plan, and implement artifact to understand what the feature does.
  2. Reasons about which dimensions are relevant — not keyword-driven, but feature-driven.
  3. Dispatches one sub-agent per chosen dimension in parallel.
  4. Each sub-agent writes findings to 07-review-<slice>-<dimension>.md.
  5. The orchestrator aggregates findings, deduplicates by file and root cause, and triages with you.
  6. Writes a master 07-review-<slice>.md with the final verdict.

Artifact names depend on review-scope (set at intake): per-slice mode (default) produces 07-review-<slice>.md + 07-review-<slice>-<dimension>.md; slug-wide mode produces 07-review.md + 07-review-<dimension>.md and reviews the entire branch diff.

Re-running /wf review <slug> does not overwrite the previous review — findings merge into an accumulating ledger. Each finding carries a surfaced-at marker and a status; a re-run that no longer reproduces an earlier finding marks it resolved rather than dropping it, so the history of what was raised and cleared survives. A sibling .yaml fragment holds the open-only findings the dashboard renders.

Each finding also carries pre-existing: true|false. A BLOCKER that predates this workflow (pre-existing: true) is routed to a ## Pre-existing Debt bucket that points you at /wf intake fix or /wf intake refactor, and it does not count toward the Ship verdict — only findings introduced by this change gate the release.

Re-run with /wf review <slug> triage to re-visit deferred findings without re-running the full review — useful after initially deferring HIGH findings and then deciding to address them.

Use the ad-hoc form (/wf review <dimension>) when you want to spot-check a file outside a workflow, or run a sweep on a long-lived branch that isn't tracked as a slug.
