# CE vs sdlc-workflow: Adoption Report

## TL;DR

The single most important thing compound-engineering (CE) has that sdlc lacks is a **knowledge-compounding loop**: the ability to write solved problems into a searchable, YAML-fronted corpus (`docs/solutions/`), grow a shared vocabulary (`CONCEPTS.md`), surface that corpus back into planning and review, and maintain it over time. sdlc's `/wf retro` is genuinely good at extracting per-workflow lessons, but those lessons die inside `.ai/workflows/<slug>/10-retro.md` and are never read by future workflows. The maintainer already recognized this in `plugins/sdlc-workflow/docs/internal/ideas/IDEAS-3.md` — it has simply not been implemented. Closing this gap should be the first move.

Beyond compounding, CE is ahead in two concentrated areas: the **review synthesis pipeline** (confidence-anchor gating, pre-existing flag, per-finding independent validator wave, triage groups) and **product/feedback intelligence** (a strategy document upstream of intake, a production-signal pulse skill, a user-session feedback ingestion path). sdlc's 31 deep rubric files are a structural advantage CE does not have and should not be replaced; CE's synthesis layer should be grafted on top of them.

The single highest-leverage move is implementing `/wf-meta compound` — the learnings-capture subcommand that writes retro lessons to `.ai/solutions/` and wires them back into `/wf plan`'s Phase 1 research. Everything else in this report compounds on top of that.

## Where CE Is Genuinely Ahead

### Knowledge Compounding (the core gap)

Every sdlc workflow starts from scratch with respect to prior failure patterns. CE closes this with three interlocking skills:

**`ce-compound`** (`.scratch/compound-engineering-plugin/plugins/compound-engineering/skills/ce-compound/SKILL.md`) runs three parallel subagents (Context Analyzer, Solution Extractor, Related Docs Finder), classifies lessons into two tracks — Bug-track (`Symptoms / What Didn't Work / Solution / Why This Works / Prevention`) and Knowledge-track (`Context / Guidance / Why This Matters / When to Apply / Examples`) — does five-dimension overlap detection before writing to avoid duplicates, and writes a YAML-fronted `.md` to `docs/solutions/<category>/`. Every run also does a Discoverability Check: it patches the repo's `AGENTS.md`/`CLAUDE.md` to surface `docs/solutions/` to future agents, ensuring the corpus compounds value even without the plugin loaded.

**`ce-compound-refresh`** (`.scratch/compound-engineering-plugin/plugins/compound-engineering/skills/ce-compound-refresh/SKILL.md`) maintains corpus quality with five outcomes (Keep/Update/Consolidate/Replace/Delete) and a Phase 1.75 document-set analysis that catches redundancy before it silently accumulates. Without this, any corpus rots.

**`ce-sessions`** (`.scratch/compound-engineering-plugin/plugins/compound-engineering/skills/ce-sessions/SKILL.md`) mines session JSONL files via bundled shell+Python scripts to surface investigation dead-ends as optional input to `ce-compound` — the "what didn't work" that never makes it into commit messages.

The live corpus at `.scratch/compound-engineering-plugin/docs/solutions/` has 30+ active entries, proving the system functions in production. `CONCEPTS.md` at the repo root holds a live 8-cluster vocabulary grown incrementally by every compound run. Downstream skills `ce-plan`, `ce-ideate`, and `ce-debug` all read `docs/solutions/` as institutional memory via a `ce-learnings-researcher` subagent — closing the loop so documented solutions reduce future investigation time.

sdlc's gap is confirmed: `plugins/sdlc-workflow/skills/wf/reference/retro.md` produces only `10-retro.md` under `.ai/workflows/<slug>/`. The `## Suggested Repo Instruction Updates` section (retro.md lines 193-196) is advisory copy-paste text for the user, not a programmatic write to any persistent store. `/wf plan` has no step that reads prior retro learnings.

### Review Synthesis Pipeline

sdlc's 31 deep rubric files are better than CE's persona-as-instructions approach for producing raw findings. But CE's synthesis layer is significantly more rigorous about filtering those findings before they reach the user:

- **Confidence anchor gate** (`ce-code-review/references/findings-schema.json`): discrete 0/25/50/75/100 anchors with behavioral criteria; findings below 75 are suppressed from the primary set (P0 escapes at 50). sdlc uses free-text `High/Med/Low` with no numeric gate — low-confidence findings flow unconditionally into triage.
- **Pre-existing flag** (`references/findings-schema.json` line 83): `pre_existing: boolean` — findings in unchanged baseline code are separated from diff-introduced issues and do not count toward the Ship/Don't Ship verdict. sdlc has no such flag; a large-debt codebase produces false "Don't Ship" verdicts on innocent commits.
- **Per-finding independent validator wave** (Stage 5b): up to 15 fresh-context subagents, each confirming a finding is real, diff-introduced, and not guarded elsewhere. sdlc uses AskUserQuestion human triage instead — correct but expensive.
- **Thematic triage groups** (Stage 5 step 9b): named groups of related findings with shared root cause and preferred resolution. sdlc presents findings as independent severity-bucketed rows.
- **Model tiering gap in `/wf review`**: `wf/reference/review.md` Step 3 dispatches all selected review commands with `model: sonnet` uniformly; CE reserves session-model (potentially Opus) for correctness, security, and adversarial reviewers, and uses mid-tier for everything else. sdlc's standalone `/review` skill already does partial tiering (haiku default, sonnet for architecture/security), but the wf-integrated path pays sonnet cost for ux-copy and style-consistency dimensions that do not need it.
- **Previous-comments reviewer**: CE's `ce-previous-comments-reviewer` (gated on `hasPriorComments: true`) checks that prior review threads were addressed before a re-review. sdlc has no equivalent; second review passes treat the PR as fresh.

### Planning and Requirements Pipeline

- **No fresh-context claim verifier** before shape or plan artifacts land. CE's `ce-brainstorm` Phase 2.6 dispatches a generation-tier verifier — a subagent that never saw the discovery dialogue — to refute absence claims and stale file references before the requirements doc is written. sdlc's `shape.md` and `plan.md` run codebase sub-agents but have no dedicated adversarial verifier step.
- **No scope-confirmation gate** before plan research dispatch. CE Phase 0.7 composes a three-bucket scope draft (Stated/Inferred/Out of scope), surfaces material forks, and waits for user confirmation before spending research budget. sdlc dispatches 3-4 parallel sub-agents without a prior scope summary, meaning scope forks that would redirect research are only surfaced after the budget is spent.
- **Ideate axis-decomposition and basis-tagging absent** from `intake/ideate.md`. CE Phase 1.5 decomposes the topic into 3-5 orthogonal axes before dispatching frame agents, enforces axis coverage (recovery dispatch for empty axes), and requires every idea to carry `basis: direct/external/reasoned`. sdlc's six fixed lenses can all converge on the same interpretation, and `IDEA-NNN` candidates carry no basis field.
- **Product-pressure-test gap lenses** (Evidence, Specificity, Counterfactual, Attachment, Durability) in `ce-brainstorm` Phase 1.2 have no equivalent scan in sdlc's `shape.md`, which runs 20 discovery questions but applies no named rigor checks to the intake brief.

### Autonomous Execution

- **No single-command end-to-end pipeline** (`lfg`). CE's `lfg/SKILL.md` chains plan→work→simplify→review→fix→browser-test→commit/PR→CI-watch+autofix in one invocation with a DONE sentinel. sdlc's per-stage orchestrators are richer but each must be invoked individually.
- **Residual review findings have no durable sink**: CE's `lfg` Steps 5-6 file unresolved findings to a tracker and appends a Residual section to the PR body before DONE. sdlc's findings land in `07-review-<slug>.md` but are not surfaced in the PR body if they remain unresolved at review close.
- **No standalone PR feedback resolution**: CE's `ce-resolve-pr-feedback` fetches unresolved GraphQL threads, dispatches parallel fix subagents with file-collision avoidance, posts GraphQL replies, and resolves threads. sdlc's equivalent (handoff T5.1) is embedded inside `/wf handoff` and cannot be re-invoked when new review comments arrive after handoff completes.
- **Per-task idempotency check is stage-level only**: CE's `ce-work` Phase 2 Step 1 checks whether a task's Verification criteria are already satisfied before implementing. sdlc's `implement.md` Step 0.6 only checks whether the artifact file exists, not whether the code changes are already present — a real risk on resume-after-compaction or when picking up someone else's branch.

### Debugging

The gap here is severe. `ce-debug/SKILL.md` is a four-phase investigation-first skill with a hard causal-chain gate ("Do not proceed to Phase 3 until you can explain the full causal chain with no gaps"), a mandatory assumption audit, predictions required for uncertain causal links, smart escalation when hypotheses are exhausted, and conditional defense-in-depth when the root-cause pattern appears in 3+ files.

sdlc's `plugins/sdlc-workflow/skills/error-analysis/SKILL.md` is a static reference document — gather → categorize → RCA template → fix template — with no execution phases, no causal-chain gate, no sub-agent dispatch, and no sub-agent parallelism for multi-subsystem bugs. This is the largest quality gap in a single skill.

### Optimization

`ce-optimize/SKILL.md` is a metric-driven iterative loop with a YAML spec, three-tier evaluation (degenerate gates → LLM-as-judge → diagnostics), parallel worktree experiment dispatch, six disk checkpoints for crash recovery, and a strategy digest that drives hypothesis batches. sdlc has `benchmark.md` (before/after delta for a known change), `profile.md` (hotspot analysis), and `experiment.md` (A/B rollout design with no measurement loop). Nothing combines define-metric-goal → generate-hypotheses → run-parallel-experiments → keep-winners → iterate.

### Product and Feedback Intelligence

sdlc has no product-facing layer at all. CE has:
- **`ce-strategy`**: a Rumelt-structured interview producing `STRATEGY.md` at repo root; downstream `ce-ideate`, `ce-brainstorm`, and `ce-plan` read it as grounding. sdlc's `intake/default.md` records a technical stack fingerprint but no product direction.
- **`ce-product-pulse`**: time-windowed query of analytics/tracing/payments, PII-free report to `docs/pulse-reports/`, strategy-seeded config, scheduling offer. sdlc has no post-ship observation skill.
- **`ce-slack-research`**: synthesizes org context (decisions, constraints, discussion arcs) from Slack into a digest with cross-cutting analysis.
- **`ce-riffrec-feedback-analysis`**: converts screen+voice recordings into graded findings and structured intake-brief requirements.

### Design

- **Visual evidence for PRs entirely absent**: `ce-demo-reel/SKILL.md` is an 8-step pipeline (project detection, motion/states classification, tool preflight, 4 capture tiers, upload, secret-scrubbing). sdlc's `handoff.md` builds PR bodies from workflow artifacts but never captures visual evidence.
- **No live dev-server polish loop**: `ce-polish/SKILL.md` auto-detects dev server (8 frameworks), starts it in background, and supports a free conversational iteration loop. sdlc's `design/polish.md` is an offline reference checklist.
- **Design system context not classified**: CE's Layer 0 (ce-frontend-design SKILL.md lines 29-58) counts design signals and classifies the project as Existing/Partial/Greenfield/Ambiguous (4+ signals = Existing; 1-3 = Partial), then adjusts how opinionated guidance is. sdlc's `_design-context.md` runs 4 parallel codebase inspection sub-agents that gather equivalent signals but stops there — no mode classification, no threshold, no rule that aesthetic guidance yields to detected tokens.

### Verification

- **Flow-mapping phase absent**: CE's `ce-dogfood-beta/SKILL.md` Phase 2 mandates drawing Mermaid `flowchart TD` diagrams per user journey before deriving the test matrix — entry, action, branch points, side effects, true end state including email click-through destinations. sdlc's `verify.md` sub-agent 3 drives each user-observable AC directly without a prior flow-mapping step, missing journey-level failures.
- **No test matrix taxonomy**: `ce-dogfood-beta/references/test-matrix-taxonomy.md` is a 5-section structured taxonomy (Journeys / Functional / Experiential / Edge-error-empty / Cross-cutting) with a file-to-route mapping table. sdlc's adversarial micro-tests and failure-mode probes cover important ground but are individual mandated checks, not a derivation-from-diff taxonomy.
- **No branch-scoped holistic QA pass**: `ce-dogfood-beta` is a branch-scoped orchestrator (refuses to run on main, resumes from durable report, auto-fixes small bugs with one regression test per fix, escalates large ones). sdlc's `/wf verify` is per-slice/forward; `/wf probe` is backward but read-only and no durable report.
- **Verify fix loop actively discourages regression tests**: sdlc `verify.md` Fix dispatch says "Do NOT touch tests unless the issue is a test failure that requires a test edit." CE's dogfood-beta mandates a regression test per autonomous fix. The sdlc wording is counter-productive and should be changed.

### Plugin Self-Maintenance

- **No CI enforcement of skill-content conventions**: CE's `tests/skill-conventions.test.ts` and `tests/skill-shell-safety.test.ts` mechanically gate reference integrity, frontmatter limits, and shell safety across all skills. sdlc's 432-test suite is entirely infrastructure-focused; skill-prose correctness is unverified by CI.
- **No version-verification CI gate**: sdlc has a 5-spot version bump surface (plugin.json, package.json, marketplace.json entry, marketplace.json top-level, `_shell.mjs` PLUGIN_VERSION) with no automated gate. CE's `bun run release:validate` catches drift.
- **No plugins/sdlc-workflow/AGENTS.md**: authoring conventions for the skills layer (router architecture, dist build rule, 6-spot version bump, Codex mirror procedure) live only in session MEMORY.md. CE's `AGENTS.md` encodes a 30+ item Skill Compliance Checklist that is auto-loaded in any contributor agent session.

## Where sdlc Is Already Ahead

**31 deep rubric files vs. persona-as-instructions.** CE's review personas receive their criteria inline in a subagent prompt built from `references/subagent-template.md`. sdlc's `review/reference/correctness.md` is 1000+ lines with eight numbered checklist categories, invariant enumeration, concrete failure scenarios, diff-format patch examples, and test scaffolding. The rubrics produce better raw findings; CE's synthesis layer produces better filtered output. The correct adoption path is grafting CE's synthesis innovations onto sdlc's existing rubric architecture — not replacing rubrics with personas.

**Sunflower HTML view.** sdlc's rendering pipeline (`renderers/design.mjs`, `design-contract.mjs`, 24-cell swatch matrix, token table with copy controls, narrative fragment system) has no CE equivalent. CE produces code in the project's repo; sdlc produces a rendered, navigable workflow-artifact view in the hub. Every CE design capability adopted for sdlc should consider how it integrates with the sunflower view — writing a sibling `.yaml` or `.html.fragment` so the rendered result compounds over the workflow's lifetime.

**Full lifecycle artifact trail.** sdlc's 10-stage idempotent artifact trail (00-index through 10-retro) gives every subsequent stage full context from all prior stages without re-explanation. CE's `lfg` pipeline is impressive but does not persist the same artifact granularity. sdlc's ship-plan contract, rollback playbooks, and recovery procedures have no CE equivalent.

**Codex parity.** sdlc has a hand-mirrored Codex tree with runtime-parity tests and a sync:codex script. CE's automated converter is more scalable but CE's Codex tree is purely CE's own concern — sdlc has already solved this for its architecture.

**Deliberate CI-fix policy.** sdlc's diagnose-first-then-ask pattern (T5.0/T5.3 in handoff.md) is a validated jayte decision documented in MEMORY.md. CE's silent auto-fix in `lfg` Step 9 is a different trust model. sdlc should not adopt CE's autonomous CI fix behavior.

**Runtime adapter registry.** sdlc's `runtime-adapters.md` covers web, Android, iOS, CLI, desktop, service, and notebook platforms with per-platform bootstrap/drive/observe/teardown recipes. CE's testing skills cover web and iOS only.

**Design subcommand depth.** sdlc's 22 `/wf design` subcommands (craft, 15 transforms, audit, critique, extract, setup, teach) plus the brand/product register that forks all design decisions is richer than CE's `ce-frontend-design` + `ce-polish` + `ce-demo-reel` combination for lifecycle-integrated design work.

**"Refuted" findings worth noting.** Several initially-reported gaps turned out to be illusory on verification:
- sdlc's `simplify.md` Agent 3 already has the recurring-no-op-updates rule verbatim — CE parity exists.
- sdlc's design guidance already covers CE's Module A/B/C landing vs app vs component distinction through the brand/product register and craft.md.
- sdlc's beta-skill gap is moot: `disable-model-invocation:true` means auto-trigger noise (the primary risk a beta flag guards against) does not apply to the router.

## Learnings Worth Recording

These are durable principles surfaced by the comparison, independent of any specific adoption decision. They could themselves seed sdlc's first `docs/solutions/skill-design/` entries.

**1. Bug-track vs Knowledge-track: two shapes, not one.** The Bug-track schema (`Symptoms / What Didn't Work / Solution / Why This Works / Prevention`) is optimized for future agents investigating the same failure. The Knowledge-track schema (`Context / Guidance / Why This Matters / When to Apply / Examples`) is optimized for future agents making design decisions. Forcing bug-level incidents into a convention-doc structure (or vice versa) produces structurally wrong docs. CE's `ce-compound/SKILL.md` lines 147-160 and `references/schema.yaml` encode this split explicitly.

**2. Discoverability is the unlock for any knowledge corpus.** A corpus that isn't surfaced in `AGENTS.md`/`CLAUDE.md` only benefits agents who already know it exists. CE's Discoverability Check (built into every `ce-compound` run, including headless mode) edits the closest `AGENTS.md` with the smallest viable addition matching the file's existing tone. Without this step, a corpus built by `/wf-meta compound` will compound value only for the session that built it.

**3. Rubric-per-dimension beats persona-as-instructions for review raw quality; CE's synthesis layer beats sdlc's for filtered output quality.** The correct model is: keep sdlc's 31 rubric files as the finding-generation layer, adopt CE's confidence-anchor gate + pre-existing flag + validator wave + triage groups as the synthesis layer. CE's `docs/solutions/skill-design/confidence-anchored-scoring.md` explains why: continuous scales cluster on round values and create coin-flip boundary bands; discrete anchors with behavioral criteria are honest.

**4. Research subagents return text data; orchestrators write files.** CE's `docs/solutions/skill-design/research-agent-pipeline-separation.md` codifies this. sdlc's retro sub-agents (retro.md line 106: "Merge all sub-agent findings and write into sections") lack the explicit prohibition on sub-agents writing files directly. Without this constraint, parallel sub-agents can race to write the same artifact and clobber each other.

**5. Post-menu routing belongs inline, not in a reference file.** CE's `docs/solutions/skill-design/post-menu-routing-belongs-inline.md` explains: routing logic that fires after a blocking `AskUserQuestion` must live in the main SKILL.md body, not in a reference file the orchestrator loads, because the reference may not be loaded in context by the time the user answers. sdlc's router-dispatch-to-reference pattern is mostly correct for pre-menu routing; this applies to any step where a reference is loaded mid-skill after a user gate.

**6. Strategy-seeded metrics prevent measuring the wrong things.** If sdlc adds a product-pulse skill, it must read `STRATEGY.md` first and seed its metric questions from the strategy's key metrics. Without this wiring, the pulse measures whatever happens to be easy to instrument rather than what determines product success. CE's `docs/skills/ce-product-pulse.md` explains this as the "strategy-seeded interview" design principle.

**7. CE's autonomous CI fix vs sdlc's diagnose-then-ask policy: different trust models, both correct for their context.** CE's `lfg` auto-fixes CI without asking (up to 3 rounds). sdlc's handoff T5.0/T5.3 diagnoses only and asks before any fix. The jayte-validated rule is the correct call for a system with ship-plan contracts and recovery playbooks. Do not adopt CE's silent auto-fix here.

**8. PROTOCOL/JUDGMENT classification for skill modernization.** CE's `docs/solutions/skill-design/frontier-model-skill-modernization-methodology.md` provides an 8-step sequence: classify blocks as PROTOCOL (deterministic steps the model executes) or JUDGMENT (decisions that need reasoning); define extraction/generation/ceiling model tiers once by name; prune JUDGMENT blocks to principle + one contrast pair; use information-asymmetric load stubs for late-sequence content. This is actionable for any sdlc skill that has grown too large.

**9. The `disable-model-invocation: true` design principle should be documented, not assumed.** Lifecycle stages and meta-navigation subcommands (all `/wf` and `/wf-meta` entries) correctly use `disable-model-invocation: true`. Standalone utility skills (`/review`, `/error-analysis`, etc.) allow auto-invoke and should remain discoverable. This design decision is correct but is not written down anywhere discoverable — it lives only in MEMORY.md.

**10. Idempotency should be checked at the task level, not the artifact level.** CE's `ce-work` Phase 2 Step 1 checks whether a task's Verification criteria are already satisfied before implementing. sdlc's `implement.md` Step 0.6 checks only whether the artifact file exists. On resume-after-compaction or when picking up another session's branch, an artifact can exist while the code is not yet present. The code-state check is the load-bearing gate.

### Code review

*CE:* CE's `ce-code-review` (`.scratch/compound-engineering-plugin/plugins/compound-engineering/skills/ce-code-review/SKILL.md`) is a six-stage pipeline: scope detection → intent discovery → diff-aware persona selection → bounded parallel dispatch → confidence-gated merge/dedup synthesis → report and optional fix apply. ### Overall architecture, philosophy & packaging

*CE:* CE is a flat collection of 37 independently-triggerable `ce-*` skills, most of which have auto-trigger descriptions so the model can invoke them without the user typing a slash command (only `ce-setup` and beta skills use `disable-model-invocation: true`). Its philosophy is explicit: every unit of work should compound into the next, so the loop closes with `ce-compound` writing a structured learning to `docs/solutions/` with YAML frontmatter (module, tags, problem_type), and `CONCEPTS.md` accumulating a live domain vocabulary as a side-effect of every compound run. `ce-compound-refresh` maintains quality over time and `ce-compound` includes a Discoverability Check that actively edits `AGENTS.md`/`CLAUDE.md` to ensure future agents find the knowledge store. Upstream, `ce-strategy` anchors the whole pipeline in a durable `STRATEGY.md` (problem, approach, persona, metrics, tracks) that `ce-ideate`, `ce-brainstorm`, and `ce-plan` read as grounding. Skills are self-contained directories with explicit references/, the `AGENTS.md` plugin instructions file contains a structured skill-design authoring guide (PROTOCOL/JUDGMENT classification, model-tier vocabulary, cross-platform blocking-question patterns, information-asymmetric load stubs, `CLAUDE_SKILL_DIR` guards), and `docs/solutions/skill-design/frontier-model-skill-modernization-methodology.md` codifies a repeatable 8-step modernization sequence. CE also ships a TypeScript converter CLI that translates the plugin to Codex, OpenCode, Gemini, Cursor, and others — eliminating the need for a hand-maintained parallel tree. `ce-worktree` is a composable isolation primitive that checks for existing isolation before creating a worktree, prefers the native harness tool, and falls back to git with `.worktrees/` gitignoring. Files reside at `.scratch/compound-engineering-plugin/README.md`, `.scratch/compound-engineering-plugin/AGENTS.md`, `.scratch/compound-engineering-plugin/CONCEPTS.md`, `.scratch/compound-engineering-plugin/plugins/compound-engineering/skills/ce-compound/SKILL.md`, `.scratch/compound-engineering-plugin/plugins/compound-engineering/skills/ce-strategy/SKILL.md`, `.scratch/compound-engineering-plugin/plugins/compound-engineering/skills/ce-worktree/SKILL.md`, and `.scratch/compound-engineering-plugin/docs/solutions/skill-design/frontier-model-skill-modernization-methodology.md`.

*sdlc:* sdlc compresses its entire lifecycle into one `/wf` router (`plugins/sdlc-workflow/skills/wf/SKILL.md`) with `disable-model-invocation: true` and 17 subcommand keys dispatched by reading reference files. This is architecturally deliberate: one entry point, strict stage separation, artifacts persisted under `.ai/workflows/<slug>/`.
