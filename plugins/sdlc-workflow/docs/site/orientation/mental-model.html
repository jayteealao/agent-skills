How it works

Before you type your first command, this page gives you the map. One read and you'll know what every command does, why the plugin writes files, and how all the pieces fit together.

The six things you type

The entire plugin is six commands. Everything else is a sub-command under one of these six.

CommandWhat you reach for it when…What it does
/wf You have a real feature, fix, or refactor to ship Runs the full 10-stage lifecycle. One stage at a time, each building on the last. Sub-commands: intake, shape, slice, plan, implement, verify, review, handoff, ship, retro
/wf-quick You want a faster path for small, well-defined changes Quick commands for specific patterns — a typo fix, a hotfix, a refactor, a dependency update. Fewer stages, fewer files. Sub-commands: fix, hotfix, refactor, rca, investigate, and more
/wf-meta You want to look up, navigate, or manage your workflows Status, resume, next, sync, amend, close — everything for working with workflows that already exist. Sub-commands: status, resume, next, sync, amend, close, init-ship-plan, build-pipeline
/wf-design You want design work on a screen, component, or visual artifact 22 design sub-commands — brief, contract, colorize, polish, audit, critique, and more. Can run as stage 2b inside a full /wf workflow or stand-alone
/wf-docs You want to generate or update documentation Generates site pages, API docs, or reference content from your code and artifacts. Integrates with the doc site generator
/review You want a structured code review on a PR or a diff 31-dimension code review. Sub-commands: a single dimension inline (/review correctness), or a full sweep (/review sweep) that fans out one agent per dimension

Not sure which one to reach for? The command chooser maps your situation to the right command and sub-command.

Why the plugin writes files

Claude has no memory between sessions. When you close a conversation, everything Claude learned about your codebase — the scope you agreed on, the edge cases you ruled out, the reason you chose one approach over another — is gone. The next session starts blank.

The plugin solves this by writing artifacts — small markdown files next to your code, one per stage. Each stage reads what previous stages wrote. Instead of re-explaining "we're adding a health endpoint to the payments service and we ruled out the gRPC approach because…", you just point Claude at the artifacts from last session.

What an artifact looks like A file like .ai/workflows/add-health-endpoint/02-shape.md contains YAML frontmatter (stage, status, slug, timestamp) followed by the structured reasoning from the shape stage — the scope, the acceptance criteria, the ruled-out approaches. It's a markdown file you can read, edit, and commit to git.

How a full change flows

Here's the 10-stage sequence a change goes through with /wf. You don't run all 10 in one go — you run one stage, review what Claude produced, and move to the next when you're happy. You can also skip stages, loop back, or branch into parallel slices.

  intake[1·intake] --> shape[2·shape]
  shape -.opt.-> design[2b·design]
  shape --> slice[3·slice]
  design --> slice
  slice --> plan[4·plan]
  plan --> impl[5·implement]
  impl --> verify[6·verify]
  verify --> review[7·review]
  review --> handoff[8·handoff]
  handoff --> ship[9·ship]
  ship --> retro[10·retro]

Each stage writes one artifact file. By the time you reach handoff, there are up to 10 files capturing every decision you made along the way.

The dashboard

The plugin includes a local web viewer — the dashboard — that shows all your workflows at a glance. Run npm run serve from the plugin root to start it, then open http://localhost:4173. The dashboard reads the artifact files from .ai/workflows/ and renders them in a readable format. Multiple repos can share one hub.

Putting it all together

flowchart TD
  you["You type a command"] --> cmd["/wf, /wf-quick, /wf-meta, etc."]
  cmd --> stages["Claude runs one or more stages"]
  stages --> artifacts["Writes artifacts to .ai/workflows/<slug>/"]
  artifacts --> next["Next stage reads previous artifacts"]
  artifacts --> dash["Dashboard renders them at localhost:4173"]
  artifacts --> git["You commit them — decisions are in your git history"]
