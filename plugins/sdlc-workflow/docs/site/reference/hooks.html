Hooks reference

Hooks are what make the plugin active during normal Claude Code use — they fire automatically when you write files, start sessions, and run commands. Two of the five hooks act as guards: they catch malformed artifact files before they are saved, so broken frontmatter never silently corrupts your workflow history.

What the hooks do for you

A hook is a small script that Claude Code calls at a specific moment — before a file write, after a file write, or when a session starts. You do not call hooks yourself; they run in the background automatically once the plugin is installed.

The two guard hooks are the most important:

  • pre-write-validate — runs before any artifact file is saved. If the filename or frontmatter (the YAML block at the top of every artifact) would be invalid, it blocks the write and tells Claude what to fix. You never end up with a broken file on disk.
  • post-write-verify — runs after a write. It performs a deeper schema check and, for the 14 rich-tier artifact types, also checks that the required companion .yaml fragment file exists. If either check fails it exits with an error and prompts Claude to repair the file.

The other three hooks handle automation:

  • post-write-auto-stage — automatically runs git add on changed code files during the implement stage (one step of the full workflow lifecycle), so you do not have to stage files manually.
  • post-write-render — refreshes the local dashboard (the Sunflower HTML view) after any artifact changes. It never blocks a write; it is best-effort.
  • session-start-orient — runs once when a session opens (and again after context compaction). It starts background rendering of the dashboard view and self-heals the tray. It surfaces no session context of its own — the /wf commands re-read each workflow's 00-index.md when they run.

Two optional leak guards ship off by default (enable them via the semantic config block below):

  • leak-guard-bash — before a publishing command runs (git commit/git tag messages, gh pr create titles and bodies, gh release notes), it scans the outward-facing text for internal workflow vocabulary — artifact paths, command strings, stage-file names — and advises you to translate to product language before publishing.
  • leak-guard-write — the same scan for writes to public documentation paths (README, CHANGELOG, CONTRIBUTING, docs/ outside the internal roots). Internal paths are never scanned — workflow vocabulary belongs there.

All seven are wired via hooks/hooks.json and run from committed esbuild bundles under dist/, not the hooks/*.mjs source files. This means they run with no npm install on a fresh install — all dependencies are inlined.

What each hook validates

pre-write-validate PreToolUse: Write

Fires before any write to a workflow artifact path (*/.ai/workflows/<slug>/*.md). Blocks writes that violate:

  • Filename convention — files must match NN-stagename.md or NNa-stagename.md (for sub-stages like 02b-design.md), plus the allowlisted names risk-register.md, estimate.md, announce.md, skip-<name>.md (close/skip records), po-answers.md, and anything under design-notes/.
  • YAML frontmatter — must be present between --- markers and contain schema, type, and slug fields. (The po-answers.md prose log is exempt.)
  • Schema version — schema: sdlc/v1.
  • Slug stability — the frontmatter slug must match the directory name.

Note: .ai/workflows/INDEX.md (the registry) and .ai/ship-plan.md are handled separately — the per-slug path pattern does not match them.

post-write-verify PostToolUse: Write / Edit / MultiEdit / NotebookEdit

Fires after the write and reads the file from disk. Runs deep Ajv validation of the frontmatter type and required fields against tests/frontmatter.schema.json for artifacts under .ai/workflows/, .ai/simplify/, .ai/profiles/, .ai/docs/, and the solutions corpus (.ai/solutions/<category>/*.md, frontmatter type: solution — the corpus INDEX.md is a frontmatter-less registry and is exempt, as is the po-answers.md prose log). Exits 2 with diagnostics on failure so Claude can repair the frontmatter in place.

Sibling-fragment hard block v9.47

Rich-tier artifact types need a companion .yaml fragment file — a small structured file the dashboard reads to render workflow cards. If you write one of the 14 gated types without its companion, post-write-verify exits 2 and blocks the write.

The 14 gated types are: review, plan, design, ship-run, rca, benchmark, experiment, instrument, profile, simplify-run, review-command, design-audit, design-critique, and design-contract.

Two escape hatches let you opt out:

  • Add fragment: none (also accepts skip or n/a) to the artifact's frontmatter to exempt a single file.
  • Set hooks.remindMissingFragments: false in .ai/sdlc-config.json to disable the check project-wide.

The types sync-report and docs-index are intentionally excluded from gating — a hard block would wedge their automated regenerators.

post-write-auto-stage PostToolUse

Auto-stages changed code files with git add when an active workflow is in the implement stage and the branch strategy is dedicated or shared. To disable auto-staging for a repo without touching config, create the file .ai/.no-auto-stage.

post-write-render PostToolUse

Re-renders the affected part of the dashboard HTML view under .ai/_view/ after an artifact changes. Never blocks the write; runs best-effort in the background.

session-start-orient SessionStart

Fires once per session and re-fires after a context compaction (source: compact). It enqueues a whole-repo render of the dashboard view (drained by the hub) and self-heals the tray — re-pointing an enabled autostart launcher at the current version and reconciling a stale running tray. It runs entirely in the background and surfaces no session context of its own. Resume after a compaction is driven by the /wf commands instead: each re-reads its workflow's 00-index.md on invocation (and the auto/yolo drivers run their own branch-posture check), so orientation is pulled just-in-time rather than pushed into context here.

leak-guard-bash PreToolUse: Bash · off by default

Fires before a Bash command, but only acts on publishing commands — git commit, git tag, gh pr create, gh release create|edit — and scans only their message-bearing arguments (never the whole command, so reading an internal file with cat is never flagged). The scan uses a lexicon derived from the plugin's single-source output-boundary rule: internal paths (.ai/…, .claude/…), workflow command strings, stage-artifact filenames, and the artifact schema tag. Plain English words that happen to be stage names ("plan", "review", "ship") are deliberately not matched. In advisory mode it emits a note naming the matches; in enforce mode it blocks the command with the same message.

leak-guard-write PreToolUse: Write / Edit / MultiEdit · off by default

The same lexicon scan for writes whose target is a public documentation path: README*, CHANGELOG*, CONTRIBUTING*, or anything under a docs/ directory outside the internal roots. Writes under .ai/ or .claude/ are never scanned — workflow vocabulary belongs there. Both guards sit out entirely during external-model dispatch and fail open on internal errors: advisory infrastructure never breaks a tool call.

Configuration

The five hooks-block toggles live in .ai/sdlc-config.json and default to true; the two leak guards are controlled by the separate semantic block, which defaults to off.

KeyDefaultControls
autoStagetrueWhether post-write-auto-stage runs git add.
validateOnWritetrueWhether pre-write-validate runs filename and frontmatter checks.
verifyOnWritetrueWhether post-write-verify runs deep Ajv validation.
remindMissingFragmentstrueWhether post-write-verify enforces the sibling-fragment hard block. Set false to disable both the block and the reminder.
validateSiblingYamltrueWhether post-write-verify validates a present sibling .yaml against its schema and blocks on violations. Only applies to types with a reconciled schema (plan, review, design, simplify-run, ship-run).
semantic.enabledfalseMaster switch for both leak guards. Off by default — the guards exit silently.
semantic.modeadvisoryadvisory emits a non-blocking note naming the matched vocabulary; enforce blocks the tool call with the same message. Promote to enforce only after the advisory false-positive rate proves ~0 on real work.
Technical details — the build model and file layout

Hooks run from dist/*.mjs: committed esbuild bundles with all third-party dependencies inlined. That is what lets a fresh install run with no runtime npm install. The source lives in hooks/. After editing any hook (or anything under lib/, renderers/, or components/) you must run npm run build to refresh dist/, or the CI freshness gate fails and production runs stale code. See Build & dist model →

Note: hooks/render-on-artifact-write.mjs exists in source as a helper module but is not a wired hook — the rendering hook is post-write-render.mjs.

plugins/sdlc-workflow/
├── hooks/
│   ├── hooks.json                  # wiring — points every entry at dist/
│   ├── pre-write-validate.mjs      # source
│   ├── post-write-verify.mjs       # source
│   ├── post-write-auto-stage.mjs   # source
│   ├── post-write-render.mjs       # source
│   └── session-start-orient.mjs    # source
└── dist/                           # committed bundles — what actually runs
    ├── pre-write-validate.mjs
    ├── post-write-verify.mjs
    ├── post-write-auto-stage.mjs
    ├── post-write-render.mjs
    └── session-start-orient.mjs

Earlier releases shipped shell and Python equivalents under hooks/scripts/; those were retired in v9.34.3–v9.34.5. The no-op PreCompact hook was removed in v9.41.0. Validation, verification, staging, rendering, and orientation are now entirely native Node.
