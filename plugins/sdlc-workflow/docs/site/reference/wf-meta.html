/wf-meta — navigation and management

You reach for /wf-meta when you want to look up, navigate, or manage workflows that already exist. None of its sub-commands create new work — they help you understand where you are, correct mistakes, and close things out cleanly.

/wf-meta has 12 sub-commands (named actions you pass after the command). None of them produce stage artifacts — the output files a workflow stage writes. They manage existing workflows or answer questions about them. disable-model-invocation: true means these sub-commands only run when you type them explicitly.

/wf-meta next [slug]

Reach for this sub-command when you know the workflow well and just need a one-line reminder of what to type next.

Reads00-index.md for the named slug (the short identifier for a workflow) or the currently active workflow if no slug is given
WritesNothing — reports only

Lightweight "what's next" lookup. Reads the index's recommended-next-command and recommended-next-invocation fields and prints them. It does not re-read artifact history or explain context.

When to use

  • You know which workflow you're in; you just forgot the next slash command.

When NOT to use

  • You've been away for days and need full re-orientation — use resume instead.

/wf-meta status [slug]

Reach for this sub-command at the start of a work session to see what is in flight, or before shipping to check for open blockers.

ReadsEvery .ai/workflows/*/00-index.md (or one if slug given). For the dashboard mode, also reads runtime-evidence-deferrals[] and compressed-slices[] (filtered for slice-type: probe) to compute the Runtime column. v9.14
WritesNothing

Daily entry point. With no slug, lists every active workflow with current stage, slice (a unit of work within a workflow) status, blockers, and recent activity. With a slug, shows the full progress map across all 10 stages × 5 states, plus slice statuses and recent artifact writes.

Runtime column on every dashboard table v9.14

All three dashboard tables (Active / Blocked / Completed) carry a Runtime column showing the slug's runtime-evidence-status. This column is orthogonal to lifecycle status — a workflow marked Completed can still carry open deferrals if it shipped via a legacy path; a Blocked workflow can carry probe findings because those findings may be exactly what's blocking it.

ConditionRuntime value
runtime-evidence-deferrals absent OR every entry has cleared-by non-null AND no probe slice with findings-count > 0clean (rendered as —)
Any runtime-evidence-deferrals entry has cleared-by: nulldeferrals: <N> (count of null entries)
Any probe compressed-slice exists with findings-count > 0 AND the findings haven't been routed through plan/quickprobe-findings: <N>
Both applyBoth, separated by + (e.g., deferrals: 2 + probe-findings: 3)

This makes the "verified but actually broken" failure mode visible at a glance. A slug that ships without clearing deferrals is blocked at /wf ship Step 6.5 after v9.14, but legacy completed workflows or in-progress slugs may carry open deferrals — the dashboard makes that obvious.

When to use

  • Daily startup — "what's in flight?"
  • Context-recovery — "what was I working on?"
  • Pre-ship audit — "are there open runtime-evidence deferrals on this slug that will block ship?" v9.14

/wf-meta resume [slug]

Reach for this sub-command after being away from a workflow for hours, days, or weeks — when you need the full picture before you can act safely.

ReadsEvery artifact in the workflow
Writes90-resume.md — orientation brief

Heavyweight re-orientation. Reads the full artifact trail, identifies the first unfinished step, and proposes a concrete next invocation. Use this when next would not give you enough context to act confidently.

What it computes

  • Last completed stage
  • Stages skipped (with reasons)
  • Slices in-progress vs. complete
  • Open questions in po-answers.md
  • Whether a paused ship run exists

/wf-meta sync [slug]

Reach for this sub-command when the status output looks wrong, or after pulling a branch that added new artifacts you didn't write yourself.

ReadsEvery file on disk under .ai/workflows/<slug>/
WritesUpdated 00-index.md + 00-sync.md (report)

Reconciles the control file with disk reality. Re-reads every artifact, detects drift (for example, an artifact that exists on disk but is missing from workflow-files), and rebuilds the index. Writes a sync report.

When to use

  • After pulling a branch that landed new artifacts.
  • After manual edits to 00-index.md.
  • When status or resume produce unexpected output.

Anti-pattern

Don't hand-edit 00-index.md to "fix" status fields. Use sync instead — it reads ground truth from the artifacts on disk.

/wf-meta amend <scope> [target]

Reach for this sub-command when the original spec or criteria were wrong and need correcting — not when the implementation has a bug.

Scopes<slug> from-review · <slug> from-retro · <slug> <slice> · ship-plan
WritesVersioned amendment artifacts: 02-shape-amend-N.md, 03-slice-<slug>-amend-N.md, OR a block edit of .ai/ship-plan.md

Corrects a prior artifact's definition — spec, criteria, scope, or approach — without overwriting completed work. Writes versioned amendment files. Never touches files with status: complete.

Workflow modes

FormEffect
amend <slug> from-reviewSeeds amendment from 07-review-<slice>.md findings flagged as spec errors.
amend <slug> from-retroSeeds from 10-retro.md items tagged as "scope was misunderstood" etc.
amend <slug> <slice>Direct slice amendment — you describe the correction.
amend <slug>General amendment — interview-driven discovery.
amend ship-planEdits one block (A–K) of .ai/ship-plan.md; bumps plan-version.

When to use vs. extend vs. implement reviews

  • amend — the spec was wrong.
  • extend — new scope appeared that doesn't fit existing slices.
  • implement reviews — the spec is fine, the implementation has a bug.

/wf-meta extend <scope> [target]

Reach for this sub-command when review or retro reveals genuine new scope — work that belongs in this workflow but doesn't have a slice yet.

Scopes<slug> from-review · <slug> from-retro · <slug>
WritesNew per-slice files with extension-round in frontmatter

Adds new slices to an existing workflow. It does not overwrite or change the original shape — if the shape itself needs fixing, that is amend.

What it preserves

  • Existing slices' status (no change).
  • The original shape — extends don't overwrite it.

/wf-meta skip <slug> <stage> [reason]

Reach for this sub-command when you have a deliberate reason to bypass a stage and want that decision recorded, not hidden.

Writesskip-<stage>.md (skip record) + stub artifact for the skipped stage (e.g., 06-verify-<slice>.md with status: skipped, for a verify skip)

Marks a stage as skipped without running it. The stub artifact satisfies downstream prerequisite checks but records the skip reason. Use sparingly — every skip is accountability for what got bypassed.

When to use

  • Verify keeps finding flakes you've decided not to chase right now — skip verify with a reason.
  • Review deliberately skipped on a hotfix.

Anti-pattern

Don't skip review on regular workflows. Review exists to catch what you missed.

/wf-meta close <slug> [reason]

Reach for this sub-command when a workflow is done — or definitively not going forward — and you want to remove it from the active list while keeping the audit trail on disk.

Reasonscancelled · superseded · deferred · completed-externally · merged-into
Writes99-close.md with reason + rationale

Archives a workflow at any stage. The workflow stops appearing as "active" in status output but stays on disk as an audit trail.

Reason semantics

ReasonUse when
cancelledWork is no longer wanted. Decision to not build / not fix.
supersededAnother approach, PR, or workflow replaced this one.
deferredWork is valid but not now. May be reopened later.
completed-externallyThe change was made outside this workflow (hotfix, manual edit, another PR).
merged-intoThis workflow's scope was absorbed into a larger workflow or PR.

/wf-meta how <question> | <slug> plan|shape|slice|review|findings

Reach for this sub-command when you need an answer — about the codebase, about an artifact, or about an open architecture question — before you start work or while you're stuck.

Modes5: quick · codebase · research · artifact-explain · findings-explain
Forcing flags--research <question> (deep research) · --quick <question> (short answer)
WritesOptional 90-how-<topic>.md or .ai/research/<topic>-<ts>.md

Question-answering command. A free-form <question> auto-routes by heuristics. Passing <slug> plan|shape|slice explains an artifact in plain language. Passing <slug> review|findings explains review output. --research and --quick force a mode explicitly.

ModeFor
quick (default)"What does X return when Y?" — short code answer.
codebase"How does the auth middleware work?" — parallel Explore agents + synthesis.
--research"State of the art in zero-downtime migrations" — 6–8 web agents, 200+ sources.
artifact-explain"Explain <slug> plan in plain English" — translates an artifact.
findings-explain"Explain review findings for <slug>" — same but for review/verify output.

When to use

  • Before planning work in an unfamiliar subsystem — codebase mode.
  • Stuck understanding what an artifact says — artifact-explain.
  • Open architecture question — research mode.

/wf-meta announce [slug]

Reach for this sub-command after a workflow is complete and you need a polished, copy-pasteable announcement for a release channel or release notes.

RequiresA completed workflow (typically through retro or close)
Writesannounce.md in the workflow directory

Produces a Diátaxis-aligned external announcement. Reads handoff + ship-run + retro to extract the user-visible change, translates it per External Output Boundary rules, and writes a copy-pasteable announcement for #releases / release-notes / blog.

/wf-meta init-ship-plan [--from-template <kind>]

Reach for this sub-command once per project, the first time you want to codify how this repo ships — its versioning contract, CI/CD pipeline, quality gates, and developer experience.

Templateskotlin-maven-central, npm-public, pypi, container-image, server-deploy, library-internal
Writes.ai/ship-plan.md at repo root (project-level, NOT under .ai/workflows/)
Run frequencyOnce per project. Use amend ship-plan to edit.

One-time setup for the whole pipeline. Works by discovery → hypothesis → confirm: reads what's already in the repo (CI workflows, infra-as-code, package manifests, runbooks, linters, hook frameworks, governance files), proposes a pipeline shape, then lets you confirm or correct each contract block. Templates are hypothesis seeds, not control-flow branches.

The 11 blocks

The plan has two halves — the outbound release (blocks A–G) and the inbound developer experience (blocks H–K).

BlockWhat
AShip meaning + environments + cadence
BVersioning contract (scheme, source-of-truth files, bump rule)
CCI/CD contract (release trigger, workflow file, required secrets)
DPost-publish verification checks
ERollout + rollback strategy
FRecovery playbooks (grow over time)
GStakeholder + announcement channels
HCode-quality gates (format/lint/type-check/coverage, commit + PR-title convention)
ILocal developer experience (git hooks, editorconfig, version files, task runner, bootstrap)
JRepo governance (branch protection + fidelity, merge controls, CODEOWNERS, PR/issue templates, dependency automation)
KSecurity & supply-chain gates (SAST/CodeQL, dependency-audit, secret-scanning, SBOM, license policy)

Three extensions thread through existing blocks: per-environment GitHub protection (Block A), CI ergonomics — caching/matrix/concurrency (Block C), and merge controls (Block J).

See Ship-plan schema for every field; Author a ship plan for the walkthrough. Once authored, run build-pipeline to build it.

/wf-meta build-pipeline [--dry-run]

Reach for this sub-command after writing or amending your ship plan — it turns the plan into real CI workflows, hooks, and governance files.

Reads.ai/ship-plan.md (the spec) + the repo's existing config and workflows
WritesGitHub Actions workflows + inbound-DX config files; .ai/pipeline-compliance.md; optionally mutates three remote settings — branch protection, environment protection, repo merge settings (each gated)
Run frequencyAfter init-ship-plan, and again after any amend ship-plan.

Treats the ship plan as the specification and the repo as the current implementation, then closes the gap. Audits each requirement as Compliant / Missing / Non-compliant, shows a gap report, and on confirmation creates missing files and patches existing ones. No file is ever overwritten without confirmation. With --dry-run it reports the full gap report and writes nothing.

What it builds (Audits A–S)

HalfAuditsOutput
OutboundA–JPre-merge + release workflows, version-bump, post-publish checks, rollback workflow, runbook stubs
InboundK–SCode-quality CI gates, commitlint + PR-title CI + config, git hooks, dev-experience files, CODEOWNERS / templates / dependency automation, security gates (CodeQL, dependency-audit, secret-scan, SBOM, license), CI ergonomics, plus the three gated remote settings

Remote mutation is limited to three settings — branch protection, environment protection, and repo merge settings — each only when the plan opts in (apply-via: gh-api). Even then it shows the current-vs-desired diff and the exact gh api payload, then requires an explicit confirm — with a "print commands only" escape hatch. It never pushes code, opens PRs, sets secrets, or runs a release.
