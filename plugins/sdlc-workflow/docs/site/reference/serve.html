Renderer-hosted serve

The plugin can serve the rendered sunflower view (.ai/_view/) over HTTP so you can browse it in a real browser with live reload, instead of opening file:// pages. There are two shapes:

  • Hub — one machine-wide process that serves every registered repo at /r/<id>/, with a multi-repo inbox at /. Default-on since v9.34.0.
  • Per-repo daemon — a standalone fallback that serves a single repo's view. Runs only when this repo opted out of the hub (no hub started) and the machine-wide kill switch allows it.
Config locationMachine-wide — ~/.sdlc/hub-config.json. Per-repo serve config does not exist.
Per-repo controlOne field only: view.hub.enabled in .ai/sdlc-config.json (does this repo participate?).
Default bind127.0.0.1:4173 — loopback, no network exposure.

Why config is machine-wide

The hub is a single process for the whole machine, so every setting that governs how the one server binds — host, port, resource caps, Tailscale exposure — is a per-developer-machine choice, not a per-repo one. It lives in ~/.sdlc/hub-config.json and is never committed. This mirrors git's model: repo-local choices in .git/config, machine identity and global behaviour in ~/.gitconfig.

The per-repo .ai/sdlc-config.json carries only view.hub.enabled. The two files share zero fields, so there are no precedence rules to reason about. Putting a serve setting (host, port, tailscale…) in the per-repo file is rejected — the per-repo schema is additionalProperties: false.

Branch-aware multi-repo hub v9.49

The hub serves every registered repo from one process and groups the landing inbox repo → branch → slug. Registry identity is repo-scoped (REGISTRY_VERSION = 2): a repo's id is hash(repoRoot) alone — branch is no longer part of the digest, so switching branches no longer forks a repo into duplicate registry entries. Per-slug rows carry branch metadata (branch, branchStrategy, baseBranch, prNumber, prUrl) and a soft liveness badge — live / merged / gone / unknown — so stale branches are visually de-emphasised. Liveness is advisory and never blocks anything (see Branch strategy).

Prefer a GUI to manage the hub? The tray app (npm run tray) puts hub status and controls in your system tray.

Configuration reference

Every key in ~/.sdlc/hub-config.json (defaults shown — the file is created with these on first run):

KeyTypeDefaultMeaning
versioninteger1Config schema version (for forward migration).
hoststring127.0.0.1Bind address. 0.0.0.0 is refused unless Tailscale is enabled and acknowledged (see Security model).
portinteger4173Listen port — the canonical SDLC URL in both hub and per-repo modes.
perRepoServebooleantrueMachine-wide kill switch. When false, no per-repo daemon ever spawns (any running one is reaped); the hub is the sole server.
liveReloadbooleantrueInject the live-reload script into the standalone per-repo fallback daemon. (The hub always live-reloads.)
maxSseClientsinteger200Aggregate Server-Sent-Events client cap across all repos (live-reload channel).
maxWatchedReposinteger50Beyond this many registered repos the hub polls instead of using fs.watch.
tailscale.enabledbooleanfalseExpose the server over Tailscale.
tailscale.modestringserveserve = tailnet-only; funnel = public internet.
tailscale.pathstring/Path prefix to expose.
tailscale.httpsbooleantrueTerminate TLS via Tailscale.
tailscale.acknowledgedPublicbooleanfalseOne-time, per-machine acknowledgement that exposing the server publishes every registered repo at once. Required for any non-loopback bind.

Per-repo render & bootstrap keys

Separate from the machine-wide hub config, each repo's .ai/sdlc-config.json may tune how its own view is built under view.render and view.bootstrap (all optional):

KeyDefaultMeaning
view.render.concurrency4Max concurrent render jobs during a bootstrap pass (1–32).
view.render.debounceMs2000PostToolUse render debounce window (0–60000 ms).
view.bootstrap.enabledtrueRun the SessionStart bootstrap render pass at all.
view.bootstrap.renderMissingtrueRender views for active workflows that have none yet.
view.bootstrap.renderStaletrueRe-render views whose storage mtime is newer than the view.

view.serve is not a per-repo key — it was removed in v9.38.0 (a breaking change). Any view.serve block in a repo config is rejected by the schema; serve/host/port/Tailscale settings live only in ~/.sdlc/hub-config.json. Upgraders with an old view.serve block should move those values there.

Security model

  • Loopback by default. The server binds 127.0.0.1 and changing that requires explicit machine-wide config.
  • No write endpoints. The server is read-only — it projects rendered HTML, it never mutates artifacts.
  • Path validation. Requests are confined to the view root; directory-traversal attempts are rejected.
  • Live-reload only into HTML. The reload script is injected only into text/html responses.
  • Non-loopback is gated twice. A 0.0.0.0 bind is refused unless tailscale.enabled: true and tailscale.acknowledgedPublic: true.
  • Exposure is all-or-nothing. Because the hub serves every registered repo, enabling Tailscale publishes them all. That is why acknowledgedPublic is a deliberate, machine-level gate rather than a committable per-repo flag.
  • Tailnet vs. public. tailscale.mode: serve exposes only to your own tailnet; funnel exposes to the public internet. On a shared tailnet, ACL the served port to yourself in your Tailscale ACLs.

Tailscale one-liners

Tailnet-only (your devices, not the public internet) — in ~/.sdlc/hub-config.json:

{
  "tailscale": { "enabled": true, "mode": "serve" }
}

Public funnel — requires the explicit acknowledgement:

{
  "host": "0.0.0.0",
  "tailscale": { "enabled": true, "mode": "funnel", "acknowledgedPublic": true }
}

Lifecycle

The session-start-orient hook kicks off detached bootstrap rendering, which calls ensureServeLifecycle. That function:

  1. Reads ~/.sdlc/hub-config.json (host, port, tailscale, liveReload, perRepoServe).
  2. If perRepoServe: false — reaps any running per-repo daemon and returns; the hub serves this repo at /r/<id>/.
  3. If a hub is alive (PID-file-gated probe of ~/.sdlc/hub.pid) — defers to it, reaping any redundant per-repo daemon.
  4. Otherwise spawns the standalone fallback daemon — dist/render-sunflower-serve.mjs in installs, falling back to scripts/render-sunflower-serve.mjs for maintainers before their first build (resolveEntrypoint prefers the committed bundle) — writing a PID file at .ai/_view/.serve.pid.

A running daemon that reports a different plugin version on its health endpoint is treated as stale and reaped, so a new install deterministically replaces the old daemon's code.

Health endpoint

GET /__sdlc/health → 200 with a JSON body. The supervisor uses the reported version for stale-daemon reaping:

{ "status": "ok", "pid": 12345, "version": "<plugin-version>", ... }

Troubleshooting

SymptomCheck
Server not starting.ai/_view/.bootstrap.log for the bootstrap render's output.
Wrong repo shown at /A pre-hub per-repo daemon may be squatting port 4173. Set perRepoServe: false and restart the session, or reap the stale PID in .ai/_view/.serve.pid.
Port conflictChange port in ~/.sdlc/hub-config.json.
