Artifacts reference

This page is for when you need to understand the exact format of the files the plugin writes. An artifact — a small markdown file the plugin writes next to your code that records what it decided and why — is the plugin's unit of persistent memory. Every command that makes a decision writes one.

Artifacts live under .ai/ in your repository root. Each workflow gets its own subdirectory named after its slug (the short identifier you chose at intake, e.g. add-login). Files inside are numbered by stage (the plugin's name for a step in the workflow, e.g. intake → shape → plan → implement).

Workflow directory tree

.ai/
├── ship-plan.md                         # Project-level. Authored once via /wf ship-plan init.
├── workflows/<slug>/
│   ├── 00-index.md                      # Control file (pure YAML frontmatter, no body)
│   ├── 00-sync.md                       # Written by /wf status if run
│   ├── 01-intake.md                     # Initial requirements
│   ├── 02-shape.md                      # Deep spec + docs plan
│   ├── 02b-design.md                    # Design brief (UX register, anti-goals) — /wf shape
│   ├── 02c-craft.md                     # Visual contract (NOT code) — /wf plan
│   ├── 02-shape-amend-N.md              # Legacy amendment record (retired type)
│   ├── 03-slice.md                      # Master slice index
│   ├── 03-slice-<slug>.md               # Per-slice definition
│   ├── 03-slice-<slug>-amend-N.md       # Per-slice amendment
│   ├── 04-plan.md                       # Plan master index (multi-slice workflows)
│   ├── 04-plan-<slug>.md                # Per-slice plan
│   ├── 04b-instrument.md                # Optional — observability add-on (workflow-scoped)
│   ├── 04c-experiment.md                # Optional — experiment add-on (workflow-scoped)
│   ├── 05-implement.md                  # Master index

│   ├── 05c-benchmark.md                 # Optional — perf add-on (workflow-scoped)

│   ├── 07-review-<slug>.md              # Per-slice master review

│   ├── 07-design-audit.md               # /wf design <slug> audit — accessibility/perf/theming/responsive

Add-on artifacts

Some files above are marked "Optional — … add-on." An add-on (the plugin's term for an optional capability you layer onto a stage) writes its own artifact alongside the stage it extends. The three add-ons that appear in the tree are instrument (adds observability hooks), experiment (adds A/B scaffolding), and benchmark (adds performance measurement). They are activated by running the corresponding sub-command at plan or implement time.

Shared frontmatter contract

Every artifact file's first lines are YAML frontmatter between --- markers. This frontmatter is what the plugin reads to understand the file's role and status. Every artifact contains at minimum:

schemaAlways sdlc/v1. Identifies the file as belonging to this plugin.
typeActive types include: index, workflow-index, intake, shape, slice, plan, implement, verify, review, review-command, handoff, ship-run, ship-runs-index, ship-rollback, ship-plan, retro, solution, design, design-contract, design-critique, design-audit, design-augmentation, augmentation, profile, rca, ideation, simplify-run, sync-report, resume, skip-record, shape-amendment, slice-amendment, slice-index, plan-index, implement-index, verify-index, docs-index, project-context, announce, risk-register, estimate. Legacy (back-compat only): fix-plan. See Artifact types for the complete list.
slugMust match the directory name. The plugin enforces this (called the slug stability invariant) so artifacts can never drift from the workflow they belong to.

Stage artifacts additionally carry stage-number, status, created-at, and updated-at. Each artifact type's full field set is described on the per-schema reference pages.

Below the frontmatter, every artifact now opens with a narrative ## The <Stage> story section (added in v9.92) — a short prose framing of what this stage decided and why — before the structured detail that follows.
