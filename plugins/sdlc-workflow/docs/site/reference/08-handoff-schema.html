Handoff schema (08-handoff.md)

Documents every field in 08-handoff.md — the readiness check result that /wf ship gates on before merging a pull request.

08-handoff.md is an artifact (a structured workflow note written to your slug directory) of type handoff. It is created at stage 8 of the /wf command flow, after all implementation slices are complete. /wf ship reads readiness-verdict from this file; a verdict of anything other than ready stops the ship sequence and tells you why.

Frontmatter — base fields

schema: sdlc/v1
type: handoff
slug: <slug>
slice-slugs: [<slug-1>, ...]
handoff-mode: aggregate | single-slice
status: complete | awaiting-input
stage-number: 8
created-at: <ISO 8601>
updated-at: <ISO 8601>
pr-title: "<PR title>"
pr-url: "<url or empty>"
pr-number: <N or null>
branch: "<branch>"
base-branch: "<target>"
has-migration: true | false
has-config-change: true | false
has-docs-changes: true | false
docs-generated: [<paths>]
tags: []
refs:
  index: 00-index.md
  slice-index: 03-slice.md
  implements: [05-implement-<slug>.md]
  reviews: [07-review-<slug>.md]
next-command: wf-ship
next-invocation: "/wf ship <slug>"

handoff-mode: aggregate means multiple slices (independent units of work under the same slug) were combined into one PR. single-slice means there is exactly one slice.

status: awaiting-input is written when the stage cannot finish — for example, CI timed out, a rebase conflict could not be resolved automatically, or the triage bound was exceeded. Re-running /wf handoff offers to resume from where it stopped.

PR-readiness block fields

These fields are written by the triage and rebase steps (T3.5–T5.3) that run after the PR is opened. Together they form the input to the verdict computation below.

FieldValuesFrom step
commitlint-statuspass | warn | fail | skippedT3.5
public-surface-driftnone | regenerated | drift-without-regen | skippedT3.6
docs-mirror-statusup-to-date | regenerated | skippedT3.7
triage-iterationsinteger 0–5T5.1
triage-fixes-appliedintegerT5.1
triage-fixes-skippedintegerT5.1
triage-deferred-thread-idsarray of thread ID stringsT5.1
has-deferred-commentstrue | falseT5.1
rebase-statusfast-forward | rebased-clean | conflicts | lease-failure | skippedT5.2
rebase-onto-shaSHA of base branch at rebase timeT5.2
live-review-decisionAPPROVED | CHANGES_REQUESTED | REVIEW_REQUIRED | nullT5.3
live-checks-failingarray of check namesT5.3
live-checks-pendingarray of check namesT5.3
readiness-verdictready | awaiting-input | blockedcomputed

Verdict computation

readiness-verdict: ready requires ALL of:

  • live-review-decision ∈ {APPROVED, null}
  • live-checks-failing is empty
  • commitlint-status ≠ fail
  • public-surface-drift ≠ drift-without-regen
  • rebase-status ∈ {fast-forward, rebased-clean, skipped}
  • has-deferred-comments = false

blocked is set when any hard-fail condition above is true. awaiting-input is set for soft fails: pending checks, deferred review suggestions, or required reviewers who have not yet responded.

CI-watch and review-settle block v9.41

Written by step T5.0 and refreshed by the final T5.3 re-watch. If these fields are absent, the step was skipped — for example when branch-strategy: none or no PR exists yet.

ci-watch-conclusion: green | red | timed-out | skipped   # terminal state of the final CI watch
ci-watch-rounds: <N>                # total poll iterations across all watches this run
ci-watch-fix-rounds: <N>            # apply-fix → push → re-watch loops run on CI red
bot-reviews-landed: [<login>, ...]  # review-bots that posted within the settle window
review-settle-elapsed-seconds: <N>  # seconds spent in the bot-review settle window

T5.0 drives CI to a terminal state and gives bot reviewers a bounded window to post before triage runs. This ensures the verdict is computed on real signal, not a stale snapshot. T5.0 does not decide the final verdict — that is T5.3, after fixes and rebase.
