/wf-docs — documentation pipeline

You reach for /wf-docs when you want to generate or update documentation. It offers two surfaces: a five-stage pipeline for a project-wide doc audit, and seven single-document generators called primitives — one for each Diátaxis quadrant plus plan, readme, and review.

On this page

  • The 5-stage pipeline — discover, audit, plan, generate, review
  • The 7 Diátaxis primitives — plan, tutorial, how-to, reference, explanation, readme, review

The 5-stage doc pipeline

When your project has grown a pile of docs that may be stale, misfiled, or missing entirely, run the pipeline. It walks every stage in order and writes progress notes to .ai/docs/<run-id>/ so you can stop and resume at any stage.

/wf-docs discover

Stage1 — inventory
Writes.ai/docs/<run-id>/discover.md

Walks the repo and inventories every existing doc file — READMEs, docs/ folders, in-code docstrings, comments in config files. Records frontmatter, classifies each file by Diátaxis quadrant, and captures last-modified date.

/wf-docs audit

Stage2 — accuracy, freshness, and quadrant fit
Writes.ai/docs/<run-id>/audit.md

For each inventoried doc: checks accuracy against current code, flags freshness (last-modified vs. relevant code change), and judges quadrant fit — for example, spotting a page labelled "tutorial" that is really a reference page in disguise.

/wf-docs plan

Stage3 — prioritized action plan
Writes.ai/docs/<run-id>/plan.md

Translates the audit into a list of actions — create, update, rewrite, or delete — one per doc, prioritized P0–P4.

Priority levels
  • P0 — broken or inaccurate
  • P1 — missing
  • P2 — wrong quadrant
  • P3 — stale
  • P4 — enhancement

/wf-docs generate

Stage4 — write the docs
Writes.ai/docs/<run-id>/generate.md + actual doc files in the repo

Executes the plan. For each doc to write, loads the matching Diátaxis primitive (described below) and follows its rules verbatim. The primitive enforces quadrant-specific rules — for example, tutorial steps must be imperative with stated outcomes.

/wf-docs review

Stage5 — independent review of the generated docs
WritesUpdates to generate.md; the docs-index artifact 08b-docs-index.md (+ sibling .yaml); then commits.

An independent sub-agent reviews each generated doc against its primitive's self-check rules — catching mixed-quadrant content, missing pre-conditions, prose-pseudocode, and similar problems.

After review it writes a compact 08b-docs-index.md — scoped to .ai/workflows/<slug>/ when run inside a workflow, or to .ai/docs/<run-id>/ in project or path mode — with a sibling 08b-docs-index.yaml containing a docs: array of {path, type, action, status} entries so the dashboard can render the docs table. It then commits all documentation changes.

The 7 Diátaxis primitives

A primitive is a reference body that defines the rules for one document type. Each lives at skills/wf-docs/reference/<type>.md. You can invoke any primitive directly for a one-off document, or let the pipeline's generate stage call them automatically.

Primitives are loaded by three callers:

  • Handoff (Stage 8) when generating per-feature docs from a workflow's docs plan.
  • /wf-docs <primitive> for one-off generation.
  • /wf-docs generate (pipeline stage 4) when executing audit-driven actions.

plan

The docs-planning primitive. Takes a feature context and an audience description, then outputs a Diátaxis plan: which quadrants are needed, what each should cover, and where each doc should live. Loaded when a shape file's docs plan needs more detail.

tutorial

Learning-oriented. Imperative voice. Pre-conditions stated up front. Every step has an expected observable outcome ("you should see…"). One worked example end-to-end.

Anti-patterns: optional digressions, mixing in reference detail.

how-to

Task-oriented. Numbered steps. Pre-conditions and goal stated up front. Variations as separate steps, not branches inside a step.

Anti-patterns: explaining concepts ("how does X work"), mixing in tutorial scaffolding.

reference

Information-oriented. Alphabetised within sections. Exhaustive. No opinions. Frontmatter-style summary tables.

Anti-patterns: narrative ordering, "first do X then Y" prose.

explanation

Understanding-oriented. Discursive prose. Trade-offs, history, comparison with alternatives. May reference how-to and reference docs but does not substitute for them.

Anti-patterns: imperative steps, exhaustive lookup tables.

readme

Composite — a single README mixes all four quadrants intentionally, but each section belongs to one quadrant. Structure: brief intro (explanation), quick start (tutorial), common tasks (how-to), API summary (reference), and explicit pointers to deeper docs.

review

Quality-check primitive. Reads a doc, identifies its declared quadrant, then checks the body against that quadrant's rules. Outputs P0–P4 findings.
