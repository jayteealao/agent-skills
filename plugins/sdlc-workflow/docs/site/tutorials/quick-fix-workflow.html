Quick-fix walkthrough

You have a one-line fix — a typo, a wrong constant, a missing null check. You want it done in minutes, not hours, but you still want a record of what changed and why. /wf-quick is the fast path: it skips the deep planning stages of the full command /wf and leaves three artifacts (workflow notes) instead of twelve.

Pre-conditionsThe plugin is installed (install tutorial). You have identified a small, bounded change — one file, one problem.
You will produce.ai/workflows/typo-invalid-payload/00-index.md, 01-fix.md, and 05-implement.md.
You will not produceA shape, slice, plan, verify, or review artifact. The fast path trades that depth for speed.
TimeRoughly 5–10 minutes for a real one-line fix.

The scenario

You are reviewing a pull request and notice the wrong spelling in a user-facing error message in src/api/parser.js:

// before
throw new Error("Recieved invalid payload");

// after
throw new Error("Received invalid payload");

This is textbook: low risk, one file, one line, easy to verify. It belongs on /wf-quick fix, not the full pipeline.

Step 1 — Start the quick-fix workflow

Type this in your Claude Code session:

/wf-quick fix "typo in invalid-payload error message"

Claude asks two or three focused questions — which file, what the exact change is, whether a test needs updating. Answer them. Claude then creates two files under .ai/workflows/typo-invalid-payload/:

.ai/workflows/typo-invalid-payload/
  00-index.md      ← registry entry for this workflow
  01-fix.md        ← the brief, scope, and acceptance criteria in one file

What 00-index.md looks like:

---
slug: typo-invalid-payload
workflow-type: fix
status: in-progress
created: 2026-06-09
---
# typo-invalid-payload

One-line spelling fix in the invalid-payload error message in src/api/parser.js.

What 01-fix.md looks like:

---
type: fix-plan
slug: typo-invalid-payload
---
# Fix plan — typo-invalid-payload

## Problem
`src/api/parser.js` line 42 spells "Recieved" instead of "Received" in the
thrown Error message. This surfaces directly to API consumers.

## Change
- File: src/api/parser.js
- Line: 42
- Before: `throw new Error("Recieved invalid payload");`
- After:  `throw new Error("Received invalid payload");`

## Acceptance criteria
1. The string "Recieved" no longer appears anywhere in src/api/parser.js.
2. Existing tests continue to pass.
3. No behaviour change.

The fix-plan skips the deep requirements interview that /wf shape runs. For a spelling fix the acceptance criteria are obvious, so Claude inlines them rather than walking through the full shape interview.

Step 2 — Implement

/wf implement typo-invalid-payload

The implement sub-command (a stage under the full /wf command) reads 01-fix.md directly, makes the edit, runs your test command, and commits. It knows this is a quick-fix workflow because of the workflow-type: fix field in 00-index.md, so it writes a single implement artifact rather than one per slice:

.ai/workflows/typo-invalid-payload/
  00-index.md
  01-fix.md
  05-implement.md  ← added now

What 05-implement.md looks like:

---
type: implement
slug: typo-invalid-payload
---
# Implementation — typo-invalid-payload

## Change made
src/api/parser.js line 42: "Recieved" → "Received"

## Test result
npm test — 47 passed, 0 failed

## Commit
fix(api): correct spelling in invalid-payload error message (a3f19c2)

Step 3 — Handoff or commit directly

For a one-liner on a shared branch you may not need a separate PR. Set branch-strategy: none during step 1 and the workflow stops here — the commit is the deliverable.

For a fix on its own branch:

/wf handoff typo-invalid-payload

This opens a PR. The PR readiness check (commitlint, surface drift, doc-mirror, triage, rebase, live check) still runs. The fast path reduces the artifact count up front; it does not bypass quality gates on the way out.

When NOT to use /wf-quick fix

  • The change touches behaviour. A fix that changes what a function returns is not a one-liner. Use /wf intake to get a real shape.
  • More than one file or one area of the codebase. The fast path cannot carry cross-cutting complexity without breaking the artifact discipline.
  • You want a retrospective. Quick-fix workflows do not get a retro by default. Use /wf if the lesson-learning matters.

Other quick commands

/wf-quick has nine sub-commands beyond fix. Each one is a focused fast path — smaller artifact footprint than the full pipeline, but still leaving a record:

CommandForWhat it produces
/wf-quick hotfixProduction incident — must ship in minutes6-stage artifact trail, scope-locked, skips review
/wf-quick rcaRoot-cause analysis from code and git history (static — no running app needed)Single artifact; recommends /wf plan, /wf-quick fix, or hotfix as next step
/wf-quick probeRuntime-truth check — drives the running artifact and compares output against acceptance criteriaEvidence directory + findings artifact (slug required)
/wf-quick investigateSketch 2–3 distinct engineering approaches with tradeoffs before committing to oneSingle artifact with options A/B/C; no winner picked — you choose, then route to /wf-quick fix or /wf intake
/wf-quick discoverTest a theory about how the codebase works against real evidenceSingle artifact with verdict: holds / partial / fails / inconclusive
/wf-quick update-depsAudit and tier dependency updates (P0 security → P1 major → P2 safe → hold)4-stage trail under .ai/dep-updates/<run-id>/
/wf-quick refactorBehaviour-preserving refactor with test baseline capture and re-verificationStage-locked: baseline → refactor → re-verify
/wf-quick ideateBrainstorm and rank improvement candidatesSingle artifact under ideation/
/wf-quick simplifyThree-agent triage of a branch, commit, plan, or codebase — routes findings, never writes codeRouting report under simplify/
