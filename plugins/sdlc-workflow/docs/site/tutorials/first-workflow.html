Full feature walkthrough

You want to see the whole picture — one real feature, from first command to open PR, with every decision recorded along the way. This walkthrough adds a /health endpoint end-to-end: intake → shape → slice → plan → implement → verify → review → handoff. About 30–45 minutes of real work; the page itself reads in 10.

Why go through 8 stages for one endpoint? You could ask Claude to "add a health endpoint" in one message. Sometimes that works. But three sessions later, when you or a reviewer asks "why did you skip downstream dependency checks?", there is no answer. The stages exist to capture decisions as you make them — so you are never re-explaining scope, and the reasoning travels with the code into git history.
Pre-conditionsYou installed the plugin (previous tutorial). You have a repo to work in. gh CLI authenticated. A ship plan is not required for this tutorial; we'll defer ship.
You will produceAbout 12 files under .ai/workflows/add-health-endpoint/. A branch feat/add-health-endpoint. A pull request.
You will not produceA merged release. We stop at handoff for this tutorial — see Run a release for the ship step.

The route we'll take

sequenceDiagram
  participant U as You
  participant W as /wf router
  participant FS as .ai/workflows/add-health-endpoint/
  U->>W: /wf intake "add /health endpoint"
  W->>FS: 01-intake.md
  U->>W: /wf shape add-health-endpoint
  W->>FS: 02-shape.md
  U->>W: /wf slice add-health-endpoint
  W->>FS: 03-slice.md + 03-slice-route.md
  U->>W: /wf plan add-health-endpoint route
  W->>FS: 04-plan-route.md
  U->>W: /wf implement add-health-endpoint route
  W->>FS: 05-implement-route.md (+ code)
  U->>W: /wf verify add-health-endpoint route
  W->>FS: 06-verify-route.md
  U->>W: /wf review add-health-endpoint route
  W->>FS: 07-review-route.md
  U->>W: /wf handoff add-health-endpoint
  W->>FS: 08-handoff.md (+ branch pushed, PR opened)

Each stage reads its predecessors. You drive — the plugin orchestrates. Notice no stage writes the same artifact as another.

Stage 1 — Intake

Tell the plugin what you want, in your own words:

/wf intake "add a /health endpoint to the server that returns 200 + JSON {status: ok}"

Claude asks intake-flavored questions: what does success look like; how urgent; any constraints; what branch strategy. Pick:

  • Branch strategy: dedicated (we want a real PR).
  • Base branch: main (or your default).
  • Slug: Claude will propose add-health-endpoint; accept it. The slug is stable for the life of the workflow.

Outcome: a directory appears at .ai/workflows/add-health-endpoint/ with 00-index.md (the control file) and 01-intake.md (your stated requirements).

Confirm:

/wf status add-health-endpoint

You should see current-stage: intake, status: complete, and one of the recommended-next options is /wf shape add-health-endpoint.

Stage 2 — Shape

Run:

/wf shape add-health-endpoint

Shape is the deepest interview stage. Claude reads your intake, runs parallel web-research sub-agents (best-practice, anti-patterns, gotchas), then asks you to fill in acceptance criteria, edge cases, constraints, and — importantly — the documentation plan. For our endpoint:

  • Acceptance: GET /health → 200 + application/json + {"status":"ok"}.
  • Edge case: should the endpoint check downstream dependencies? No, this is a liveness probe — keep it independent.
  • Docs needed: a reference entry under docs/api/health.md.

Outcome: 02-shape.md with full spec and docs-needed: true, docs-types: [reference] in its frontmatter.

Why so many questions? Shape is where the AI commits to understanding rather than guessing. The questions you answer here become the artifact every later stage reads. Two hours of vague chat collapses into a structured interview — a 20-question baseline (five rounds of four) that extends only while blocking ambiguity remains.

Stage 3 — Slice

Run:

/wf slice add-health-endpoint

Slice decomposes the shaped work into thin, independently deliverable vertical slices. For a single endpoint we'll probably get one slice:

  • route — the route handler, response shape, content-type header, and unit test.

Bigger features get more slices. The slice file records complexity (xs/s/m/l/xl) and depends-on arrays so later stages can fan out in parallel.

Outcome: 03-slice.md (master index) plus 03-slice-route.md (per-slice file).

Stage 4 — Plan

Run:

/wf plan add-health-endpoint route

Plan is repo-aware. Claude dispatches Explore sub-agents in parallel:

  • Reuse scan — does the codebase already have a route registration utility, response helper, or test factory we should use?
  • Convention scan — what file/folder shape do existing routes follow?
  • Web-research — current best practices for liveness probes in our framework.

The plan file lists the files to touch, the order, and explicit reuse decisions ("use app.addRoute from src/server/routes.ts; don't write a new helper"). It records has-blockers: false if everything is straightforward.

Outcome: 04-plan-route.md.

Stage 5 — Implement

Run:

/wf implement add-health-endpoint route

Implement executes the plan — it's the only stage that writes code. Claude:

  1. Reads 04-plan-route.md and the convention/reuse decisions baked into it.
  2. Makes the actual edits.
  3. Runs the project's test command to verify atomicity.
  4. Commits with a message that follows your repo's convention (often Conventional Commits).

If Claude deviates from the plan — say it discovers a constraint the plan missed — the deviation is recorded explicitly in 05-implement-route.md's ## Plan deviations section. The plan stays a plan; the implement file becomes the record of what actually happened.

Outcome: 05-implement-route.md with commit-sha, metric-files-changed, metric-lines-added in frontmatter. Code in your repo. One commit on the workflow branch.

Stage 6 — Verify

Run:

/wf verify add-health-endpoint route

Verify exercises the acceptance criteria from 02-shape.md. For our endpoint Claude will:

  • Run the unit test that was added in implement.
  • Run an integration test if one fits (start the server, curl /health, check status + content-type + body).
  • Capture evidence (test output, response bodies) in 06-verify-route.md.

If verify finds issues, it runs a single-round, user-gated fix loop — you triage each failing check (Fix/Skip/Escalate) and verify dispatches sub-agents for the Fix decisions, then re-runs affected checks once. Only if the loop cannot resolve everything (convergence: escalated) does it suggest returning to /wf implement.

Outcome: 06-verify-route.md with result: pass and evidence sections.

Stage 7 — Review

Run:

/wf review add-health-endpoint route

Review is a dispatch orchestrator: Claude reads shape and slice, decides which of the 33 review dimensions apply (probably correctness, api-contracts, reliability, testing), and runs each as a parallel sub-agent. The sub-agents write findings to 07-review-route-<dimension>.md files. The orchestrator then aggregates, de-duplicates, and triages with you.

Findings carry one of three severities:

SeverityWhat it meansRouting
BLOCKERCorrectness, security, data loss — must fix./wf implement <slug> <slice> reviews
HIGHShould fix — but might ship with caveats.Same path, or carry as a caveat in handoff.
MED/LOW/NITStyle, naming, nit-with-merit.Ship.

Suppose review finds one HIGH ("the endpoint returns text/plain when Accept doesn't include json — should be json regardless"). Review triages findings with you inline — you pick Fix/Defer/Dismiss for each BLOCKER/HIGH finding. Fix decisions spawn sub-agents within the review stage itself (Step 4c). /wf implement add-health-endpoint route reviews remains available as a manual escape if the review-owned loop cannot resolve a finding (convergence: escalated).

Outcome: 07-review-route.md (master verdict) plus per-dimension files. verdict: ship | ship-with-caveats | dont-ship.

Stage 8 — Handoff

Run:

/wf handoff add-health-endpoint

Handoff is the most complex stage in the pipeline. It aggregates all complete slices into one PR package and runs the PR-readiness block:

  • T1 — read prior artifacts
  • T2 — write a reviewer-friendly summary
  • T3 — generate Diátaxis docs from 02-shape.md's docs plan (here: a reference entry at docs/api/health.md)
  • T3.5 — commitlint pass (if you have .commitlintrc; otherwise skipped silently)
  • T3.6 — public-surface drift (skipped if 00-index.md has no public-surface: block)
  • T3.7 — doc-mirror regen (skipped likewise)
  • T4 — push the branch
  • T5 — create or update the pull request
  • T5.0 — watch CI to a terminal state and let bot reviews settle v9.41 (so readiness is judged on real signal, not a stale snapshot)
  • T5.1 — PR comment triage loop (only matters once reviewers leave comments)
  • T5.2 — rebase onto base + --force-with-lease
  • T5.3 — live PR readiness check via gh pr view --json
  • T6 — write 08-handoff.md

The artifact's frontmatter ends with readiness-verdict: ready | awaiting-input | blocked. Ship will not start unless that field is ready.

Outcome: A live pull request on GitHub with a reviewer-friendly description; the readiness-verdict stamped into 08-handoff.md; docs/api/health.md committed and referenced.

What "ready" actually means readiness-verdict: ready is the contract between handoff and ship. It packs the answers to: did the commits pass commitlint? did the public API surface change without regenerating the mirror? are there pending CI checks? did a reviewer request changes? is the branch behind base? are there unresolved blocker comments? If any of those say "no", the verdict downgrades to awaiting-input or blocked, and ship refuses to proceed. See The readiness check.

Stopping here (Ship + Retro)

For this tutorial we stop at handoff. Why? Because ship is plan-driven — it requires .ai/ship-plan.md at the repo root, which you author once per project with /wf ship-plan init. The two pages that take you the rest of the way:

how-to Author a ship plan One-time per project. Discovery-led: Claude reads your repo, proposes a pipeline, you confirm each block. how-to Run a release The 13 idempotent ship steps. Per release, not per workflow.

After ship completes, /wf retro add-health-endpoint reads the full trail (intake through ship-run) and proposes concrete additions to your CLAUDE.md, hooks, test coverage, and CI checks. Retro is short — usually 5 minutes — but it's where institutional learning compounds.

Takeaways

  • Each stage produces a permanent artifact under .ai/workflows/<slug>/. Commit them. Three weeks later, future-you can still read why a decision was made.
  • Stages don't bleed. Plan describes intent; implement describes what was done; review finds gaps; handoff packages. Each artifact has one shape.
  • You drive. Every stage offers Options A/B/C/D for the next step — the default is sequential, but you can skip-to, loop back, or branch.
  • You can stop and resume. Walk away after stage 4. Come back next week. /wf recap add-health-endpoint picks up where you left off — the artifacts hold the state.
