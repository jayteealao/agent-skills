tutorial

Install the plugin

By the end of this page, you'll have the plugin loaded in Claude Code, all skills discovered, and a working /wf-meta status command. About five minutes.

Pre-conditionsClaude Code installed; gh CLI authenticated; a repo open as the working directory.
You will produceA loaded plugin and one verified slash command.
You will not produceAny workflow artifacts — that's the first-workflow tutorial.

Step 1 — Add the plugin to Claude Code

From inside any Claude Code session, install the plugin from the marketplace it lives in. Treat the marketplace URL as whatever your team uses; here we'll use the canonical source:

/plugin marketplace add jayteealao/agent-skills
/plugin install sdlc-workflow@jayteealao/agent-skills

Claude Code echoes back the loaded plugin's name, version, and skill count. Install the latest available release — the features documented across this site span v9.5.0 (the PR-readiness block) through the v9.4x branch-aware hub and tray app, so an older build will be missing chunks of what you read here.

Already on an older version? Run /plugin update sdlc-workflow. Features across this site only exist on recent builds — the handoff PR-readiness block and replayable ship runs (v9.5.0), committed dist/ bundles (v9.45), the tray app (v9.46), and the branch-aware hub (v9.49).
No npm install needed. Since v9.45.0 the hooks, renderer, and serve daemon run from committed dist/ bundles with dependencies inlined — installing the plugin is all an end user does. (npm install + npm run build are only for maintainers editing the plugin source.) An optional system-tray app (npm run tray) puts hub controls in your notification area.

Step 2 — Confirm the skills loaded

The plugin's behaviour is bundled into six commands. Confirm Claude can see them:

/skills

You should see at least: wf, wf-meta, wf-design, wf-docs, review. Each dispatches across many sub-commands — see Skills reference.

Step 3 — Verify a meta command works

Open a session at the root of any git repo and run:

/wf-meta status

You should see one of two outcomes:

  • "No active workflows." — Expected on a fresh repo. The plugin can read the file system and there are no .ai/workflows/<slug>/ directories.
  • A table of one or more workflows. — You already have workflow state in this repo; the plugin discovered it.

Either output proves the plugin is wired up.

Step 4 — Verify the pre-write hook is active (optional)

The plugin ships a PreToolUse hook (run from the committed dist/ bundle) that gates writes to .ai/workflows/*.md against a schema. To confirm it's loaded, try writing an invalid artifact:

# Tell Claude:
"Write `.ai/workflows/dummy/00-index.md` with the literal contents: hello"

The hook should block the write with a diagnostic about the missing schema / frontmatter. The block proves the hook is running. (You don't need to fix the dummy file — it never got written.)

Step 5 — You're done

If steps 1–3 succeeded, you have a working installation. Continue to:

Troubleshooting

/wf-meta status says "command not found"
The command didn't load. Run /plugin list to confirm sdlc-workflow is installed and enabled. If it's missing, re-run step 1.
The hook isn't blocking invalid writes
Hooks load lazily. Restart the session: /exit then re-open. If still nothing, see Hooks reference for the path the hook should be at.
gh pr create later in /wf handoff fails with auth
Run gh auth login outside Claude Code, then retry. The plugin doesn't manage gh auth.
Mermaid diagrams don't render in this site
Open the page via a local server (python3 -m http.server) rather than file://. The diagram source stays readable either way — it's the text inside <pre class="mermaid">.
