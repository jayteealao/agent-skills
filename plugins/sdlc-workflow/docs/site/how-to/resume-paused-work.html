Resume paused work

You stepped away from a piece of work — maybe days ago — and now need to pick it back up. Run /wf-meta resume <slug> and the plugin reads your notes, finds the first unfinished step, and tells you exactly what to do next.

Pre-conditionsA workflow exists. You have walked away and lost your place.
GoalResume at the right point with full context restored.

Step 1 — find your paused workflow

First, list every open workflow to remind yourself which one you want:

/wf-meta status

This prints a table of open workflows. Each row shows a slug — the short identifier for a workflow, for example auth-token-refresh — along with its current stage and how long it has been open.

Example output:

slug                   stage         open since
auth-token-refresh     03-design     4 days
perf-cache-layer       05-implement  2 hours

Step 2 — resume the workflow

Pick the slug you want and run:

/wf-meta resume auth-token-refresh

The command reads every artifact (a workflow note — a markdown file the plugin writes as a stage completes) in that workflow's directory, works out which stage is unfinished, and proposes a concrete next command. You can accept the suggestion or type something different.

Step 3 — if a ship run stalled

A ship run is the automated sequence that bumps the version, merges the PR, tags the release, and polls post-publish checks. Sometimes it pauses mid-sequence — for example, a registry API returns 404 while a package is still propagating.

When that happens the run records status: awaiting-input and waits. To continue, run ship again against the same slug:

/wf ship auth-token-refresh

The plugin detects the paused run and offers three choices:

ChoiceWhat it does
Resume the paused runContinues from the failed step. The same run ID is kept; only the pending checks are re-polled.
Start freshOpens a new run. The paused run stays on record at awaiting-input.
Mark prior failed, start freshCloses the paused run as failed and opens a new one. Useful when a CI config bug was the root cause.

Each step inside a ship run is safe to re-run. If a PR is already merged or a tag already exists, that step is detected as done and skipped automatically.

When to start fresh instead of resuming

  • The plan changed significantly after the run started. Compare the plan-version-at-run field in the run artifact against the current plan version. If they differ, a fresh run records the updated plan.
  • The failure came from a CI bug you have since fixed. A fresh run produces a new workflow run ID, which makes the audit trail cleaner.
