Pick the right entry point

You have something to change in your codebase — which command do you type? This page maps your situation to the right entry point, no prior knowledge required.

Pre-conditionsPlugin installed.
GoalPick the right /wf command that fits your situation, then go type it.

Pick an entry point

By scenario — find yours first

Scan the list, match your situation, and use the command shown. Each entry links to a walkthrough if you want to see what happens next.

"I need to add a new feature."
/wf intake "<feature description>"
Runs the full 10-stage pipeline — design, implementation, review, handoff. Slowest option; most thorough. See the walkthrough →
"There's a typo, a one-line patch, or a docs edit."
/wf intake fix "<description>"
Five artifacts (01-fix.md, 02-shape.md, 03-slice.md, 04-plan.md, 00-index.md), about 10 minutes. See the walkthrough →
"Production is on fire."
/wf intake hotfix "<symptom>"
Scope-locked and fast. Runs the full standard lifecycle single-pass — review still runs (defaults to the security rubric) — and leaves an artifact trail so you can audit after the incident.
"The dependencies are out of date. Some have CVEs."
/wf intake update-deps
Audits, tiers by priority (P0 security → P1 major → P2 safe → hold), and applies in order.
"I want to refactor this without changing behaviour."
/wf intake refactor "<area>"
Captures a test baseline first. Refactors in incremental green steps. Re-verifies parity at the end.
"There's a bug. I don't know where it is yet."
/wf intake rca "<symptom + repro>"
Root-cause analysis from code and git history — static diagnosis, no running code. If the symptom is only visible at runtime (a button does nothing, a screen renders wrong), use probe instead.
"The slice says 'verified' but the running app looks broken." v9.14
/wf probe <slug> [target]
A slug (the short id that names a workflow in progress) is required. Drives the running artifact, captures observable output — screenshots, stdout, response bodies — and compares against acceptance criteria. Does not write a fix.
"I want 2–3 distinct engineering approaches sketched before I pick one."
/wf intake investigate "<problem>"
Produces an A/B/C option set with effort, blast radius, reversibility, and risk for each. No winner is picked — that decision stays with you. After you decide, route to /wf intake fix (small) or /wf intake (medium+).
"I have a theory about how this code works and want it tested against the codebase."
/wf intake discover "<hypothesis>"
FOR / AGAINST / counter-hypothesis sub-commands adjudicate the claim. Verdict: holds, partial, fails, or inconclusive, with cited evidence. Different from /wf recap <slug> <focus> — that explains a workflow artifact; discover tests whether your theory is correct.
"I want to read about an unfamiliar subsystem."
/wf recap <slug> <focus>
Explains a workflow artifact in depth. Doesn't write a stage artifact unless you opt in.
"I want to clean up or triage a branch before going further."
/wf simplify [branch|commit|plan|codebase]
Three parallel sub-agents (Code Reuse, Code Quality, Efficiency) classify findings and hand back a queue of copy-pasteable downstream commands. Routes; never writes code.

Still unsure? Use the decision tree

flowchart TD
  start{"What kind of change?"}
  start -- "Production incident, must ship in minutes" --> hotfix["/wf intake hotfix"]
  start -- "Single-file typo, one-line patch" --> fix["/wf intake fix"]
  start -- "Dependency updates, security or major version" --> deps["/wf intake update-deps"]
  start -- "Behaviour-preserving refactor" --> refactor["/wf intake refactor"]
  start -- "Bug report, root cause unknown (static diagnosis from code/git)" --> rca["/wf intake rca"]
  start -- "Runtime-truth check on a running artifact — verified but actually broken?" --> probe["/wf probe <slug>"]
  start -- "Need 2–3 distinct engineering approaches sketched before committing" --> investigate["/wf intake investigate"]
  start -- "Have a theory about how this code works and want it adjudicated" --> discover["/wf intake discover"]
  start -- "A real feature or substantive change" --> wf["/wf intake"]

Yellow = compressed entry point (same 10-stage pipeline, but planning ceremony is condensed for focused or fast-turnaround work; /wf probe is a runtime-truth check, not an intake mode). Blue = full /wf pipeline (10 stages, every artifact, adaptive routing).

Choosing between /wf full lifecycle and quick modes

If the scenario list above left you with two candidates, use this checklist to break the tie.

QuestionIf yesIf no
Could someone reasonably ask for a code review on this?/wf/wf intake <mode>
Does it change observable behaviour that a user might notice?/wf/wf intake fix
Are the acceptance criteria obvious from the description alone?/wf intake <mode>/wf (the intake stage will shape it)
Will the change touch more than two files?/wfEither
Do you want a retro artifact at the end?/wf/wf intake <mode>

Two or more "/wf" answers → run /wf intake. Otherwise a quick command is fine.
