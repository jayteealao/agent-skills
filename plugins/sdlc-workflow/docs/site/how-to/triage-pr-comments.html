Triage PR comments

A reviewer or bot left comments on your PR. Now what? You do not have to manually read through every thread and decide what to fix — the handoff triage loop classifies each comment, fixes the blockers automatically, and asks you about the rest.

Pre-conditionsYou ran /wf handoff <slug>; a PR exists; at least one reviewer has left a comment.
GoalGet readiness-verdict: ready stamped on your handoff note so /wf ship can proceed.
Bound5 iterations. After that the loop records status: awaiting-input and stops — it never spins forever.

What the triage loop does

When you run /wf handoff <slug>, the final block of the handoff stage does three things before it closes out:

  1. Waits for real signal. It watches CI to a terminal state (pass or fail) and gives bot reviewers a bounded window to post their comments. This means the loop works on comments that have actually arrived, not a half-empty snapshot.
  2. Classifies every open thread. Each unresolved review thread gets a severity label: blocking, suggestion, or informational.
  3. Acts on what it finds. Blocking comments are routed to a sub-agent for fixes. Suggestions are surfaced to you with a multi-choice prompt — apply, defer, or decline. Informational comments are acknowledged and resolved.

After each round of fixes the loop re-fetches open threads and repeats. Once no blocking comments remain and every suggestion is accounted for, it stamps a readiness check — a verdict recorded in 08-handoff.md that tells /wf ship whether this PR is safe to push. A readiness check (glossary) is simply a machine-readable record of whether all review blockers have been resolved.

If CI comes back red, the wait step delegates fixes to a sub-agent and re-watches. It never blocks on humans — if something requires your decision, it routes the verdict to awaiting-input and stops, leaving a clear record of what needs attention.

Re-run handoff to re-triage

Re-running handoff is safe at any time. Each step is idempotent — your branch, PR, and existing notes are preserved. Only the triage loop re-runs against the latest comment set.

/wf handoff <slug>

Severity classification

The loop assigns one of three labels to each open review thread:

BlockingSuggestionInformational
CHANGES_REQUESTED; correctness bug; crash; security issue; data loss; missing migration; breaking API change; explicit "must fix" language. Style; naming; documentation gap; test gap; refactor recommendation; nit with merit. Walkthrough summary; praise; declined-nit acknowledgment; "FYI" comment.

When a comment is ambiguous, the loop prefers the more severe class. Bot walkthrough summaries are not treated as blocking unless they contain explicit blocker language.

Deferring and declining suggestions

When the loop surfaces a suggestion, the multi-choice prompt gives you three options:

  • Apply — a sub-agent implements the suggestion and resolves the thread.
  • Defer — the thread ID is recorded in triage-deferred-thread-ids inside 08-handoff.md and has-deferred-comments: true is set. The readiness check downgrades to awaiting-input, so /wf ship will warn you before proceeding.
  • Decline — the loop posts a brief rationale as a new comment via gh pr comment, but does NOT resolve the thread (it stays open). The declined item is counted in triage-fixes-skipped in the handoff frontmatter.

Override the default bot list

The loop watches these reviewer bots by default:

coderabbitai
greptile-dev
gemini-code-assist
chatgpt-codex-connector[bot]

To change the list for a project, add review-bots: to 00-index.md:

review-bots:
  - my-custom-bot
  - another-bot[bot]
Technical details — T5.0 / T5.1 / T5.2 / T5.3 step breakdown

T5.0 — wait for CI and bots. Runs first. Watches the CI run to a terminal state (success, failure, or cancelled) using gh run watch, then gives bot reviewers a bounded window to post. If CI is red, a sub-agent is delegated to diagnose and fix; handoff re-watches after. If the sub-agent cannot resolve CI, the stage sets status: awaiting-input and halts rather than blocking on humans.

T5.1 — PR comment triage. Queries reviewThreads(first: 100) via GitHub GraphQL, filtering for isResolved=false. Each thread yields { threadId, author, file, line, body }. The loop builds a triage table: Source | Line | Severity | Summary | Action. Blocking threads are sent to /wf implement <slug> <slice> reviews one by one; each returns a commit SHA. Suggestion threads are presented via AskUserQuestion multi-select. Resolved threads are closed via resolveReviewThread mutation.

T5.2 — Rebase onto base. Fetches the base branch, checks fast-forward eligibility, runs git rebase origin/<base-branch>, and force-pushes with lease.

T5.3 — re-watch and finalize. After fixes land, CI is re-watched to a terminal state. If no blocking threads and no un-triaged suggestions remain, the loop exits and stamps the readiness check. Otherwise it repeats from T5.1, up to 5 total iterations.
