Amend or extend a workflow

Two things can go wrong after you start building. Either the spec you agreed to turns out to be wrong — the acceptance criteria, scope, or approach needs correcting — or new scope appears mid-work that belongs in the same effort. These are different problems and they use different commands.

SituationCommand
The spec was wrong/wf-meta amend — correct the definition without touching completed work.
New scope appeared/wf-meta extend — add new slices to the existing workflow.
The spec is right but the code is buggy/wf implement <slug> <slice> reviews — fix the implementation. Do not reach for amend.

Amend — when the spec was wrong

A slug (glossary) is the short identifier for a workflow, like auth-refresh or payment-retry. Each workflow stores its definition in a set of artifacts (glossary) — the markdown files in .ai/ that record the shape, plan, and implementation notes for that slug.

When a review or retro reveals that an acceptance criterion was incorrect, or that the approach relied on something that does not exist, use /wf-meta amend. It writes versioned amendment files alongside the originals — for example, 02-shape-amend-1.md and 03-slice-<slice-slug>-amend-1.md (where <slice-slug> is the individual slice identifier, not the workflow slug) — and never overwrites artifacts that were already completed.

/wf-meta amend auth-refresh from-review    # seeds the correction from 07-review findings
/wf-meta amend auth-refresh from-retro     # seeds from 10-retro.md
/wf-meta amend auth-refresh                # describe the correction manually
/wf-meta amend ship-plan                   # edit one block of .ai/ship-plan.md

After amend writes the corrected artifacts, re-run /wf plan auth-refresh <slice> to update the plan, then /wf implement auth-refresh <slice> to implement the correction.

Extend — when new scope appeared

Sometimes a review or retro surfaces scope that genuinely belongs in this workflow but does not exist as a slice yet. Use /wf-meta extend. It adds new slices to the existing workflow without touching the slices that are already there.

/wf-meta extend auth-refresh from-review
/wf-meta extend auth-refresh from-retro
/wf-meta extend auth-refresh

After extend registers the new slices, run /wf plan and then /wf implement for each new slice as normal.

Implement (reviews mode) — when the code is buggy

If the spec is correct but the implementation has a bug, do not amend. The spec does not need correcting — only the code does.

/wf implement auth-refresh login-flow reviews

This is the most common path after a review: review finds a blocker, implement-reviews fixes it, verify again, then re-handoff.

Decision rule

What you foundUse
"Acceptance criterion is contradictory"amend
"The approach assumed X but X does not exist"amend
"This slice should have been two slices"amend (split) + extend (new slice)
"This feature is missing a piece we agreed on"extend
"The code returns 500 when it should return 400"implement reviews
"The test does not cover the edge case"implement reviews
