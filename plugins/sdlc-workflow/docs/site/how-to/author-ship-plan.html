Author a ship plan

Before /wf ship works, the repo needs a .ai/ship-plan.md. You author it once per project.

Pre-conditionsPlugin v9.5.0+. A repo open. No existing .ai/ship-plan.md — if one exists, see amend ship-plan instead.
GoalWrite .ai/ship-plan.md with all seven blocks (A–G) populated.
Time~15–30 minutes the first time. Subsequent runs reuse the plan.

Step 1 — Pick a template

Six templates ship with the plugin. Pick the one that best matches how you release:

TemplateWhen
kotlin-maven-centralJVM library published to Maven Central via Sonatype + GPG signing.
npm-publicnpm package published to the public registry, often with provenance.
pypiPython package via Trusted Publisher (OIDC) or API token.
container-imageContainer image to GHCR/ECR/Docker Hub (multi-arch).
server-deployKubernetes / ECS / fly.io rolling deploy.
library-internalInternal/private registry; merge-on-main = released.

Step 2 — Run init

/wf-meta init-ship-plan --from-template <kind>

Or omit the flag to be asked. Claude reads the template, then asks 13 grouped questions covering blocks A–G.

Step 3 — Answer the block questions

The plan has seven blocks:

A — Ship meaning + environments + cadence
What does "ship" mean here? Which environments? How often?
B — Versioning contract
Semver/calver/sequential. Which files hold the version literal. Bump rule (git-cliff, conventional commits, manual). Prerelease/post-release suffixes.
C — CI/CD contract
Release trigger (tag-on-main, merge-to-main, manual-dispatch). Release workflow file. Required secrets. Publish dry-run + publish commands.
D — Post-publish verification
Which checks confirm publish succeeded (registry-api, fresh-resolve, github-release, smoke-test, k8s-rollout-status). Propagation window. Poll interval.
E — Rollout strategy + rollback
Default rollout (immediate/staged/canary/flag). Rollback mechanism. DB migration reversibility.
F — Recovery playbooks
Templates pre-seed two or three (signing failures, token 401s, version-already-uploaded). Add more over time as runs hit new failure modes.
G — Announcements
Channels (#releases, mailing list). Optional template path.

Step 4 — Confirm + write

Claude shows a summary table and asks for confirmation. Confirm writes .ai/ship-plan.md at repo root. Adjust goes back to one block. Cancel discards.

Step 5 — Commit the plan

Commit .ai/ship-plan.md. It's a project-level contract; every workflow on this repo will ship through it.

git add .ai/ship-plan.md
git commit -m "chore: author ship plan (template: npm-public)"

Amending the plan

When CI/CD changes, secrets rotate, or you accumulate enough run experience to add recovery playbooks:

/wf-meta amend ship-plan

Pick which block (A–G) to edit. The amend flow re-runs only that block's questions. plan-version bumps by 1.
