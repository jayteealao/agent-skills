how-to

Close a workflow

You are done with a workflow — either because it shipped, because you decided not to do it, or because you are putting it on hold indefinitely. Use /wf close to mark it finished and record why.

Pre-conditionsA workflow exists. You know its slug — the short identifier the plugin assigned when the workflow was created (for example, add-dark-mode). You can list all open slugs with /wf status.
GoalMark the workflow closed with a recorded reason so the audit trail is complete.

Every close call writes a 99-close.md artifact — a workflow note that records the reason, a short rationale, the last stage reached, and any work that was deliberately left undone. This artifact is what makes a closure auditable rather than just silent.

I shipped it already (close cleanly)

If you ran /wf ship followed by /wf retro, the workflow index is marked status: complete — the normal finish state. /wf close is a separate operation that marks status: closed and writes 99-close.md for auditing an early or out-of-band termination.

If you shipped by some other means — merged the PR manually, deployed directly — the workflow is still technically open. Tell the plugin it is done:

/wf close add-dark-mode completed-externally

The plugin writes 99-close.md with close-reason: completed-externally and records that the workflow ended here. Nothing else changes. This is the most common close path for out-of-band completions.

I am abandoning it

Sometimes the right call is to stop. The idea was deprioritised, the approach turned out to be wrong, or a different workflow made this one unnecessary.

Two reasons cover this:

/wf close add-dark-mode cancelled
Use this when you decided not to do the work at all. The plugin will ask for a one-paragraph reason before writing 99-close.md.
/wf close add-dark-mode superseded
Use this when another workflow absorbed the intent of this one. The plugin will prompt you for the slug of the replacement workflow, which it stores as a pointer in 99-close.md.

Example — cancelling a workflow whose approach was rejected in review:

/wf close add-dark-mode cancelled

Output (truncated):

Closing add-dark-mode (cancelled).
Brief reason? > Design review decided to ship a theme toggle in Q3 instead.
Wrote: .ai/workflows/add-dark-mode/99-close.md
Workflow is now closed.

I am pausing it indefinitely

Closing a workflow is final. If you expect to return — even months from now — use deferred to record that the work is valid but not now.

On hold (external blocker):

/wf close add-dark-mode deferred

Use deferred only when the blocker is genuinely outside your control: a vendor API is unavailable, compliance review has not come back, a dependency has not published a release. If the blocker is just low priority or unclear requirements, the workflow is better left open or cancelled. Most deferred situations are better handled by extending the workflow with a new slice — /wf intake <slug> <scope>.

When the blocker clears, use /wf recap <slug> to reopen the workflow. See Resume paused work for the full steps.

Technical details — what 99-close.md contains

99-close.md is written with the following fields:

  • close-reason — one of cancelled, superseded, deferred, completed-externally, merged-into
  • One-paragraph rationale (from the prompt you provide)
  • The last numbered stage artifact that was present when close ran
  • Any deferred items noted during the workflow
  • For superseded or merged-into: a pointer to the replacement workflow slug

Close vs. skip — they are different things

Before closing a whole workflow, make sure you do not mean to skip a single stage. Skip says "this one step does not apply"; close says "the entire workflow is over".

ActionCommandEffect
Skip one stage /wf close <slug> <slice> Writes a stub artifact so later stages' prerequisites are satisfied. The workflow continues.
Close the workflow /wf close <slug> <reason> Writes 99-close.md. The workflow stops. No further stages run.
