Run a release

Your work is in handoff-ready state. Now you want to actually ship it. /wf ship runs a 13-step sequence against your ship plan — and if any step fails, you can resume from that point rather than start over.

Pre-conditions.ai/ship-plan.md exists (author it). Your handoff artifact (08-handoff.md) carries readiness-verdict: ready — meaning the automated readiness check (a gate that confirms all required evidence fields are filled before shipping is allowed) passed. You are on the workflow branch.
GoalA completed ship run captured in 09-ship-run-<run-id>.md.
If something failsThe run pauses with status: awaiting-input. Re-running /wf ship <slug> offers to resume from the failed step.

Invoke the command

/wf ship <slug>

Where <slug> is the short identifier for your workflow — for example auth-refresh. To override the plan's default environment:

/wf ship auth-refresh staging

Before the run begins, you are asked three questions once:

  • Rollout strategy — confirm the plan default or override it.
  • Release window — any blackout periods or on-call coverage to note.
  • Stakeholder overrides — anyone who needs explicit sign-off for this run.

These questions are never re-asked if you resume.

The 13 steps — reassurance, not a wall

Most steps are fully automated. The list below shows what the command is doing on your behalf.

  1. Orient — reads the ship plan and confirms a readiness check passed. Refuses if it did not.
  2. Pre-flight — updates version literals and commits them. Usually takes a few seconds.
  3. Dry run — runs your publish command without actually publishing. Takes 30 seconds to 5 minutes depending on build size.
  4. Rollout setup — configures the environment per the rollout strategy you confirmed.
  5. Freshness — checks that CI on the branch is green and no new commits arrived during dry-run.
  6. Go / no-go — summarises findings and asks for your final decision. The run stops here on "no-go."
  7. Merge — merges the branch to the target. Instant if CI is green.
  8. Tag — creates the version tag and GitHub release. Instant.
  9. Release workflow watch — monitors the CI release workflow to completion. Can take 5–30 minutes while the actual publish runs.
  10. Recovery (conditional) — if the release workflow fails and the failure matches a playbook in your plan, Claude runs the playbook steps as confirmed actions. Each step is confirmed before it runs.
  11. Post-publish poll — checks that the package or artifact is reachable at its published URL. Bounded by plan.propagation-window-max-minutes.
  12. Post-bump — bumps the version to the next development pre-release.
  13. Write artifact — writes 09-ship-run-<run-id>.md with all captured evidence.

Every step detects its already-done state before acting. If you resume, the command picks up at the first step that has an empty evidence field.

If a step fails

The run sets status: awaiting-input and records which step stopped. Re-run the same command:

/wf ship auth-refresh

You are offered three choices:

ChoiceEffect
Resume <run-id>Continue from the failed step. Same run-id; the existing artifact gets new evidence appended.
Start freshNew run-id; the prior run stays at status: awaiting-input.
Mark prior as failed, start freshPrior run gets status: failed; a new run-id begins.

If you need to roll back

Roll back manually using the method in plan.rollback-mechanism — for example, a git revert, a feature-flag toggle, or a redeploy of the prior version. Then record what happened in the run artifact:

# edit 09-ship-run-<run-id>.md and set:
rolled-back: true
rollback-sha: <sha>
rollback-reason: "<one sentence>"

To capture the failure as a recovery playbook so it is handled automatically next time:

/wf-meta amend ship-plan
