Use the design pipeline

You are building a change that touches a visual or UX surface and you want design work to be part of it. The /wf-design command — the design pipeline — gives you 22 sub-commands for that work. This page helps you pick the right one for your situation.

Pre-conditionsFor modes A and B: a workflow exists and its 02-shape.md file is present. For mode C (freestanding): nothing required.
GoalChoose the invocation mode and sub-command that fits the design task you are doing right now.
One-time setupRun /wf-design setup once per project to create PRODUCT.md and DESIGN.md. Most sub-commands read these files for context.

The three ways to invoke /wf-design

Before picking a sub-command, decide which mode fits your situation.

Mode A — design as a stage inside a full workflow

You have already run /wf shape <slug> for a UI feature. You want to continue the workflow by authoring the design brief. Run:

/wf-design <slug>

This writes 02b-design.md (the UX brief) and then automatically queues the next stage. Use this when you are following the full workflow pipeline from intake to ship.

Mode B — design sub-command layered on an existing workflow

A workflow already exists. You want to run one specific design operation against it — for example, an accessibility audit or a typography pass — without advancing the pipeline manually. Run:

/wf-design <slug> <sub-command>

Example: /wf-design my-feature harden applies an accessibility transformation to the UI code in the my-feature workflow. The result is recorded as an add-on (a registered add-on in 00-index.md) so handoff and ship can surface it.

Mode C — freestanding, no workflow attached

You want design help that is not tied to any workflow at all — for instance, refining copy across the product or running a one-off accessibility check on an existing screen. Run:

/wf-design <sub-command>

The command reads your project's PRODUCT.md and DESIGN.md for context but does not touch any workflow notes.

Picking the right sub-command

The 22 sub-commands fall into four groups.

1. Project context (run once, any time)

/wf-design setup
Interview-driven setup. Writes PRODUCT.md (audience, voice, register) and DESIGN.md (tokens, components, patterns). Run once per repo. Almost every other sub-command reads these files.
/wf-design teach
Re-runs the relevant interview round and updates PRODUCT.md or DESIGN.md in place. Use after the project's brand or design system has evolved.

2. Planning sub-commands (before code exists)

These require 02-shape.md to exist (i.e., shape has already run for the workflow).

/wf-design shape <slug> (also the default for Mode A)
Authors 02b-design.md: the UX brief covering register, anti-goals, layout approach, key states, and interaction model.
/wf-design <slug> craft
Authors 02c-craft.md — the visual contract reviewers will check the implementation against. Requires 02b-design.md to exist first.

3. Review and extraction sub-commands (once code exists)

These require the workflow to be at stage implement or later. You cannot review code that does not exist yet.

/wf-design <slug> audit
Writes 07-design-audit.md with severity-graded findings across accessibility, performance, theming, and responsive layout.
/wf-design <slug> critique
Writes 07-design-critique.md — an expert UX critique covering visual hierarchy, information architecture, cognitive load, and persona-based testing.
/wf-design <slug> extract
Reads the workflow's UI code and pulls reusable tokens and components into a design-notes file, then updates DESIGN.md. Available at any stage.

4. Transformation sub-commands (focused improvements)

Each transformation sub-command modifies existing UI code and records itself as an add-on in 00-index.md. Handoff and ship then translate each add-on into reviewer mentions and changelog lines automatically. Available from implement through handoff.

Sub-commandWhat it does
animateAdd purposeful motion, micro-interactions, transitions.
bolderAmplify a too-safe or bland design with more visual character.
clarifyImprove unclear UX copy, error messages, microcopy, and labels.
colorizeAdd strategic color to a monochromatic surface.
delightAdd moments of joy, personality, and unexpected touches.
distillStrip to essence; remove unnecessary complexity.
hardenAccessibility — keyboard navigation, screen-reader support, contrast, focus.
layoutImprove composition, spacing, and visual rhythm.
onboardFirst-run UX — intro screens, empty states, tooltips, progressive disclosure.
optimizeUI performance — loading, rendering, animation, bundle size.
overdrivePush past convention with shaders, spring physics, scroll-driven reveals.

A worked example

You are building a new settings screen. The workflow slug is settings-v2.

  1. Shape is already done: .ai/wf/settings-v2/02-shape.md exists.
  2. Author the design brief: /wf-design settings-v2 — this is Mode A. It writes 02b-design.md.
  3. Pin the visual contract: /wf-design settings-v2 craft — writes 02c-craft.md.
  4. After implement, the screen renders but the contrast is low. Run: /wf-design settings-v2 harden. The accessibility fixes are applied to the code and registered as an add-on.
  5. At handoff, the plugin reads the add-on and adds "Accessibility improvements applied — N components updated, axe-core scan clean" to 08-handoff.md automatically.
